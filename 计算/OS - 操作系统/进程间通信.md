## 互斥与同步问题

在多道程序设计环境下，并发执行的多个进程相互隔离、独立异步地向前推进。虽然进程隔离保证了稳定性和安全性，但不同进程需要协同工作时便面临挑战。**进程间通信 (inter-process communication, IPC)** 主要解决两个问题：
+ **信息共享与数据交换**——高级进程间通信
+ **共享资源的有序访问**——互斥与同步

如果没有有序访问机制，IPC带来的不是协作而是混乱，典型的后果是**竞争条件 (race condition)**：多个进程访问共享资源时，最终结果取决于进程运行的精确时序。

> [!definition] 互斥、同步
> 进程的**互斥 (mutual exclusion)** 是指若干个进程同时竞争一个需要互斥使用的资源时，任何时刻最多允许一个进程去使用，其他要使用该资源的进程必须等待，直到该资源被释放。互斥产生于**资源共享**，针对「同一时刻访问」问题，确保共享资源在某一时刻只能被一个进程拥有，属于无意识的安排。
> 
> 进程的**同步 (synchronization)** 是指协作进程之间相互等待对方消息或信号的协调关系。同步产生于**进程合作**，针对「访问先后顺序」问题，是有意识的安排。
> 
> 逐次使用互斥资源也是对进程使用资源次序上的一种协调，因此**互斥是一种特殊的同步**。

互斥与同步问题对于线程同样适用。

### 临界资源和临界区

系统中某些共享资源一次只允许一个进程使用，这类资源称为**临界资源 (critical resource)**。硬件临界资源如打印机、光盘刻录机；软件临界资源如只能互斥访问的变量、表格、队列。并非所有共享资源都是临界资源。

**临界区 (critical section)** 是进程中访问临界资源的代码片段，其附近的代码结构分为：
+ **进入区 (entry section)**：检查可否进入临界区；如果可以进入，设置相应「正在访问临界区」标志。
+ **临界区**：访问临界资源，**必须互斥地执行**。
+ **退出区 (exit section)**：重置「正在访问临界区」标志，允许其他进程进入。
+ 剩余区 (remainder section)：不访问临界资源的代码。

访问临界区时，应遵循以下四个原则，以保证使用临界资源的进程能够正确和高效地进行协作，避免竞争条件：
1. **空闲让进**。当无进程在临界区时，任何有权进程可以进入。
2. **忙则等待**。不允许两个以上进程同时进入临界区。
3. **有限等待**。任何进入临界区的要求应在有限时间内得到满足。
4. **让权等待**。不能进入临界区的进程应放弃占用 CPU，即应避免连续测试一个变量直到出现某个值的**忙等待 (busy waiting)** 浪费 CPU 时间。用于忙等待的锁称为**自旋锁 (spin lock)**。

### 经典软件互斥算法

#### 锁变量法

使用共享变量 `lock` 标记临界区是否被占用：

```c
while (lock == 1);   // 忙等待
lock = 1;
// 临界区
lock = 0;
```

这一方法非常简单，但存在以下问题：
+ 违反「忙则等待」原则：检查和设置操作不原子，`while (lock == 1)` 和 `lock = 1` 之间可能发生进程切换，导致多个进程同时进入临界区；
+ 违反「让权等待」原则：存在忙等待问题，浪费 CPU 时间。

#### 轮转法

设置变量 `turn` 记录轮到哪个进程进入临界区，进程严格轮流进入。

```tx
| 进程 0 | 进程 1 |
| :- | :- |
| ```c | ```c |\
| while (turn != 0); | while (turn != 1); |\
| // 临界区 | // 临界区 |\
| turn = 1; | turn = 0; |\
| ``` | ``` |
```

这一方法解决了锁变量法的互斥问题，但仍有忙等待，此外因**程序代码不对称**还存在以下缺点：
+ 违反「空闲让进」原则：当一个进程完成临界区后，必须等待另一个进程进入并离开临界区才能再次进入，即使另一个进程不需要进入临界区；
+ 违反「有限等待」原则：当一个进程完成临界区后，另一个进程可能永远不进入临界区，导致前者无法再次进入。

#### Peterson算法

设置整型变量 `turn` 和布尔数组 `interested[N]`：

```c
void enter_region(int process) {
    int other = 1 - process;      // 另一个进程的编号
    interested[process] = TRUE;   // process 想进入临界区
    turn = other;                 // 让另一个进程先行
    while (interested[other] == TRUE && turn == other);  // 等待
}
```

只有当对方不感兴趣（`interested[other]` 为 `FALSE`）或轮到本进程（`turn` 为 `process`）时，进程才能进入临界区，**正确解决了两个进程的互斥问题**，克服了轮转法的缺点。但该算法**仅适用于两个进程**，且仍存在忙等待问题。

> [!note] 标志牌算法的死锁与Peterson算法的破解
> 若只有 `interested[]` 标志牌而没有 `turn`，两进程同时举牌时将互相看到对方举牌，从而在 while 上死等。
>
> 为此，Peterson 加入**共享变量 `turn`** 作为冲突裁决者，两进程都执行 `turn = other` 把优先权让给对方，后写的将覆盖前写的，等同于后写 `turn` 的进程把通道让给了对方，自己被while条件挡住。只有一方想进入临界区时，`interested[other]` 为 `FALSE`，结果与 `turn` 无关，不影响直接通过。
> 
> 但是，`turn` 的裁决依赖于其只有两个可能值，`other = 1 - process`，因此Peterson算法无法扩展到三个或更多进程。在其多进程推广Filter算法中，`turn` 变为一个数组，进程需要逐级竞争进入临界区，虽然能正确解决多进程互斥问题，但实现复杂，忙等待开销大。

### 硬件互斥方法

#### 禁止中断方法

因为只有在发生时钟中断或其他中断时才会进行进程切换，可以考虑**进入临界区前关中断**，离开后开中断。

这一方法简单但不实用：
+ 把特权指令交给用户进程导致系统可靠性较差；
+ 不适用于多处理器。

#### 专用硬件指令方法

利用处理机提供的专门的硬件指令，**将读和写合并在一条指令中完成**——CPU 以指令为最小执行单位，指令执行期间不响应中断、不发生进程切换，因此外界只能看到执行前或执行后的结果，不存在「读完了但还没写」的时间窗口。

+ **测试并设置 (test-and-set) 指令**原子地将目标位旧值存入 `CF` 标志位，同时将该位置为置1或0，如 x86 的 `BTS`/`BTR`：
    ```assembly
    enter_region:
        LOCK BTS lock, 0     // CF ← lock 第 0 位旧值，同时将该位置 1
        JC enter_region      // CF = 1 说明原来就是 1（已被占），继续忙等
        RET                  // CF = 0 说明原来空闲，刚抢到了锁
    
    leave_region:
        LOCK BTR lock, 0     // 原子地将 lock 第 0 位清 0：CF ← 旧值，同时置 0
        RET
    ```
    其中指令前缀 `LOCK` 用于锁内存总线，以防其他处理器访问内存；**`BTS` 本身只做「读旧值并置 1」而不做判断**，判断由下一条 `JC` 根据已保存的 `CF` 完成。原子性保护的是「读—写」这一对操作，「读—判」并不需要原子，因为旧值已安全存入 `CF`。
+ **对换 (SWAP) 指令**原子地交换两个操作数，如 x86 的 `XCHG`：
    ```assembly
    enter_region:
        MOV reg, 1           // reg <- 1 将要写入 lock，表示想进入临界区
        LOCK XCHG lock, reg  // lock <--> reg
        CMP reg, 0           // lock 原来为 1 则忙等
        JNZ enter_region
        RET
    
    leave_region:
        LOCK MOV lock, 0
        RET
    ```

硬件指令方法的优点在于，其简单且易验证，可支持进程中多个临界区，**适用于任意数目进程**；缺点在于，仍存在忙等待问题，且需要处理器提供专门的指令支持。


> [!danger] 优先级反转问题
> 在存在优先级调度的系统中，忙等待可能导致**优先级反转 (priority inversion)** 问题。
> 
> 例如，假设高优先级进程H就绪后抢占正在临界区中的低优先级进程L，但因L被抢占**无法离开临界区**，H在忙等循环中无限等待。

## 信号量

软件和硬件互斥方法均为平等进程间的协商机制，且都有忙等待的缺点。1965年，Dijkstra提出**信号量 (semaphore)** 概念，由 OS 作为地位高于进程的管理者来解决公有资源的使用问题。

### 信号量的数据结构

每个信号量 `s` 包含一个整型计数值 `s.count` 和一个等待队列 `s.queue`，队列中是阻塞在该信号量上的各个进程的标识。信号量只能通过初始化以及 P、V 两个原语来访问，作为 OS 核心代码执行，不受进程调度的打断。

一般地，信号量计数值表示资源的可用情况，称为**资源信号量**：
+ `s.count` > 0 表示有 count 个资源可用；
+ `s.count` = 0 表示无资源可用；
+ `s.count` < 0 时，|`s.count`| 表示等待队列中的进程个数。

若不需要信号量的计数能力，只需要对临界资源的互斥访问，则通常置初值为1，称为**互斥信号量**。

### P、V原语

**P (_proberen_) 操作**表示申请资源，也称为 wait/down 操作；**V (_verhogen_) 操作**表示释放资源，也称为 signal/up 操作。P、V 原语的定义可写为：
```tx
| P原语 | V原语 |
| :- | :- | 
| ```c | ```c \
| P(s) {                         | V(s) {                        \
|     --s.count;                 |     ++s.count;                   \
|     if (s.count < 0) {         |     if (s.count <= 0) {               \
|         调用进程进入 s.queue;  |          从 s.queue 取出一个进程 P;   \
|         阻塞调用进程;          |          P 进入就绪队列;         \
|     }                          |    }             \
| }                              | }                              \
| ``` | ``` |
```

P、V操作都是**原语**，由操作系统内核提供支持，执行过程中不可被打断，因此具有原子性。

#### 使用P、V实现互斥

为临界资设置互斥信号量 `mutex`，初值为1，**将临界区代码包裹在 `P(mutex)` 和 `V(mutex)` 之间**，即可实现互斥访问。

```c
P(mutex);
// 临界区
V(mutex);
```

#### 使用P、V实现同步

为每个前趋关系设置信号量，初值为0，**在前趋进程中执行V操作，在后续进程中执行P操作**，即可实现同步。

例如，若要求代码段 C1 在 C2 开始前完成，可设信号量 `s12`，C1 中执行 `V(s12)`，C2 中执行 `P(s12)`：
```tx
| 进程 1 | 进程 2 |
| :- | :- |
| ```c | ```c \
| // C1 | P(s12);   // 取用 s12 资源 \
| V(s12);   // s12 资源自增 | // C2 \
| ``` | ``` |
```

> [!tip] P、V使用要点
> + 必须**成对使用**，不能重复或遗漏。遗漏P不能保证互斥，遗漏V不能释放资源。
> + 互斥操作时，P和V在同一进程中；同步操作时，P和V不在同一进程中。
> + 有多个P操作时，**次序至关重要**。同步P必须在互斥P之前，否则可能死锁。两个V操作的顺序则无关紧要。

### 使用信号量解决经典 IPC 问题

经典 IPC 问题是描述、分析和测试进程同步与互斥机制正确性的模型。

#### 生产者—消费者问题

**生产者—消费者问题 (the producer-consumer problem)** 也称**有限缓冲区问题 (bounded-buffer problem)**，考虑控制若干进程通过有限的共享缓冲区交换数据。假定有 $N$ 个数据单元的共享缓冲区，生产者进程不断写入，而消费者进程不断读出，任何时刻只能有一个进程操作缓冲区。

为解决该问题，需要三个信号量：
+ 资源信号量 `empty`，初值为 $N$，表示缓冲区剩余空位数；
+ 资源信号量 `full`，初值为 0，表示缓冲区中已占用的单元数；
+ 互斥信号量 `mutex`，初值为 1，保护对缓冲区的互斥访问。

```tx
| 生产者 | 消费者 |
| :- | :- |
| ```c | ```c |\
| P(empty); | P(full); |\
| P(mutex); | P(mutex); |\
| // 将一个数据单元放入缓冲区 | // 从缓冲区取出一个数据单元 |\
| V(mutex); | V(mutex); |\
| V(full); | V(empty); |\
| ``` | ``` |
```

> [!danger] P 操作次序
> 每个进程中各个P操作的次序是重要的。在生产者中必须先 `P(empty)` 再 `P(mutex)`，在消费者中必须先 `P(full)` 再 `P(mutex)`）。若先 `P(mutex)` 再 `P(empty)`，当缓冲区满时，生产者持有互斥锁但等待 `empty`，消费者无法获得互斥锁来释放资源，形成死锁。

#### 读者—写者问题

**读者—写者问题 (the readers-writers problem)** 考虑读者、写者两组并发进程，共享一组数据区。数据区允许多个读者同时读，但不允许多个写者同时写，也不允许读者、写者同时操作。

为解决该问题，需要两个互斥信号量：
+ 互斥信号量 `mutex`，初值为 1，保护对数据区的互斥访问；
+ 互斥信号量 `rc_mutex`，初值为 1，保护读者计数器 `rc` 的互斥访问。

```tx
| 读者 | 写者 |
| :- | :- |
| ```c                      | ```c              \
| while (true) {            | while (true) {    \
|     P(rc_mutex);              |            \
|     rc++;                     |              \
|     if (rc == 1) P(mutex);  // 第一个读者负责关闭写入口    |     P(mutex);           \
|     V(rc_mutex);              |                   \
|     // 读操作                 |     // 写操作               \
|     P(rc_mutex);              |                   \
|     rc--;                     |                     \
|     if (rc == 0) V(mutex);  // 最后一个读者负责打开写入口  |     V(mutex);                \
|     V(rc_mutex);              |                   \
| }                         | }                 \
| ```                       | ```               |
```


#### 哲学家进餐问题

**哲学家进餐问题 (the dining philosophers problem)** 是多线程的互斥问题。考虑5个哲学家围圆桌而坐，桌子上放着5把叉子，每两个哲学家之间放1把；哲学家的动作包括思考和进餐，进餐时需要同时拿起他左边和右边的两把叉子，思考时则同时将两把叉子放回原处。

可以针对叉子设置互斥信号量数组 `fork[5]`，初值为1，表示叉子可用。哲学家进餐时需要同时申请两把叉子，吃完后释放两把叉子。但这种方法可能导致死锁：如果每个哲学家都先拿起左边的叉子，那么所有哲学家都在等待右边的叉子，形成循环等待。为避免死锁，可以引入限制，例如最多允许 4 个哲学家同时尝试进餐，或要求奇数号哲学家先拿左叉、偶数号先拿右叉。
```c
#define N 5

void take_forks(int i) {
    if (i % 2 == 0) {
        P(fork[i]);             // 先拿左边的叉子
        P(fork[(i + 1) % N]);   // 再拿右边的叉子
    } else {
        P(fork[(i + 1) % N]);   // 先拿右边的叉子
        P(fork[i]);             // 再拿左边的叉子
    }
}
```

另一种思路是为哲学家设置信号量数组 `s[5]`，初值为0，表示没有进餐。哲学家拿叉子前先检查左右邻居是否正在进餐，只有邻居都不在吃时才允许拿起叉子。这种方案天然避免了死锁：哲学家不会持有一把叉子又等待另一把，因为要么一次性获得两把、要么根本就不拿。
```c
#define N 5

void test(int i) {
    if (state[i] == HUNGRY && state[LEFT] != EATING && state[RIGHT] != EATING) {
        state[i] = EATING;
        V(s[i]);      // 唤醒哲学家 i，使其通过 take_forks 中的 P(s[i])
    }
}

void take_forks(int i) {
    P(mutex);
    state[i] = HUNGRY;
    test(i);          // 尝试拿叉子
    V(mutex);
    P(s[i]);          // 若 test 未令其进餐，则在此阻塞
}

void put_forks(int i) {
    P(mutex);
    state[i] = THINKING;
    test(LEFT);       // 检查左邻居是否可以进餐
    test(RIGHT);      // 检查右邻居是否可以进餐
    V(mutex);
}
```

#### 睡眠理发师问题

理发店有一位理发师、一把理发椅和 $N$ 把等候椅。无顾客时理发师在理发椅上睡觉；顾客到来须唤醒理发师，理发师正忙时若有空椅则坐下等，否则离开。

为解决该问题，需要三个信号量：
+ 资源信号量 `customers`，初值为0，表示等待理发的顾客数；
+ 资源信号量 `barber`，初值为0，表示理发师是否空闲；
+ 互斥信号量 `mutex`，初值为1，保护对等候椅的互斥访问。

```tx
| 顾客 | 理发师 |
| :- | :- |
| ```c | ```c \
| P(mutex);                      | while (true) {                \
| if (waiting < N) {             |     P(customers);             \
|     waiting++;                 |     P(mutex);                 \
|     V(customers);              |     waiting--;                \
|     V(mutex);                  |     V(barber);  // 响应需求而非释放资源 \
|     P(barber);                 |     V(mutex);                  \
|     // 得以理发                |     // 理发操作                \
| } else {                       | }                            \
|     V(mutex);                  |                              \
|     // 离开理发店              |                              \
| }                              |                              \
| ``` | ``` |
```

这里 `barber` 更接近「信号」而非「资源」，顾客通过 `P(barber)` 等待理发师，理发师通过 `V(barber)` 响应顾客的到来，然后**同步**进入理发环节。`customers` 则是一个真正的资源信号量，表示等待理发的顾客数。

## 管程

### 引入动机

信号量的同步控制操作分散在各个进程中，导致正确性分析困难，易读性差，不利于修改和维护。因此，1971年Dijkstra提出「秘书」进程思想，1973年Hansen和Hoare推广为**管程**。

> [!definition] 管程
> **管程 (monitor)** 是关于共享资源的数据结构及一组针对该资源的操作过程所构成的软件模块，其基本思想是把信号量及其操作原语封装在一个对象内部，将共享变量及对其能够进行的所有操作集中在一个模块中。

管程的主要特性是：
+ **模块化**，一个管程是一个基本程序单位，可以单独编译；
+ **抽象数据类型**，不仅有数据，还有对数据操作的代码；
+ **信息封装**，共享变量在管程外部不可见，外部只能通过调用管程中的过程来间接访问。

引入管程可提高代码的可读性，便于修改和维护，正确性易于保证。

### 管程的组成

管程内部包含三个部分：**共享变量、条件变量、管程过程**。当一个进程调用管程过程时，若管程中已有其他活跃进程，调用进程被阻塞在**入口等待队列**，直到另一个进程离开管程而将其唤醒。

**条件变量 (condition variables)** 是管程内的一种数据结构，只能通过 `wait` 和 `signal` 两个原语操作：
+ 调用 `wait(c)`，调用进程阻塞并移入与条件变量 `c` 相关的等待队列，释放管程。
+ 调用 `signal(c)`，若 `c` 链为空则相当于空操作，否则唤醒 `c` 链中第一个等待者。当执行唤醒操作时，管程中存在两个同时活动的进程，
    + 在Hansen语义下，唤醒是管程中最后一个可执行的操作，执行完 `signal` 后唤醒者直接离开管程；
    + 在Hoare语义下，被唤醒者继续运行，而唤醒者等待，通常使用**紧急等待队列**管理这种情形。

> [!note] 管程的 `signal` 与信号量的 V
> 管程中的 `signal` 操作与信号量中的 V 操作不同。若无等待进程，管程的 signal 被忽略，不会记录历史；而信号量的 V 操作会使信号量值增加，影响后续的 P 操作。

管程是**编程语言概念**，编译器必须识别管程并对其互斥做出安排。Java、Pascal 等支持管程，但 C、C++ 及多数语言不直接支持管程。

## 高级进程间通信

信号量、互斥锁和条件变量等同步与互斥机制的核心作用是**建立进程（或线程）之间的时序约束**，主要用于协调执行顺序和传递状态信息，不适合复杂或大规模数据传输。在需要结构化或大量数据交换时，需引入更高级的通信机制。

### 共享内存

在物理内存中划出一块**共享存储区**，通信进程将其映射到各自的进程地址空间，通过直接读写共享区数据实现通信。这一方式效率很高，进程直接读写内存，无需数据拷贝。

操作系统一般只提供共享的内存空间，进程间的读写互斥由程序开发人员负责解决，通过进程的互斥与同步来保证数据一致性。

> [!example]- Linux/Windows 共享内存机制概览
> Linux 支持 `mmap()` 系统调用、POSIX 共享内存和 System V 共享内存三种方式。Windows 采用文件映射机制(`CreateFileMapping` / `MapViewOfFile`)，将文件映射为进程虚拟地址空间的一部分。

使用全局变量在同一个进程内部的线程之间实现通信不称为共享内存。

### 消息传递

消息传递通信是进程间以内存缓冲区为基础、以**消息 (message)** 为单位进行数据交换，分为两种方式：
+ **直接通信**：发送和接收进程显式指定对方标识，实现点对点通信。原语形式为 `send(receiver, message)` / `receive(sender, message)`。主要用于早期微内核系统，**通信关系绑定在进程身份上**，无法处理服务重启、迁移和复杂通信模式，现代系统中逐渐被淘汰。
+ **间接通信**：进程通过中介实体（邮箱、消息队列）进行通信，发送者和接收者完全解耦，支持异步通信。原语形式为 `send(mailbox/queue, message)` / `receive(mailbox/queue, message)`。

### 管道

**管道 (pipe)** 是 UNIX 首创的**以文件系统为基础**的通信方式，以字节流方式传送信息。利用管道时，一个进程的输出可成为另外一个进程的输入，典型应用是 shell 中的管道命令 `|`，如：
```bash
ls | grep "txt" | wc -l
```

管道分为两类：
+ **无名管道 (anonymous pipe)** 是临时对象，通过 `pipe()` 系统调用创建，获得读、写两个文件描述符。只适用于父子进程或父进程安排的子进程之间的通信。
+ **命名管道 (named pipe)** 也称为FIFO，是文件系统的真正实体，可在无亲缘关系的进程间通信。

> [!example]- POSIX/Linux 同步机制
> POSIX 提供三种同步 API：**互斥锁**(mutex, `pthread_mutex_lock/unlock`)、**条件变量**(cond, `pthread_cond_wait/signal`)、**读写锁**(rwlock, `pthread_rwlock_rdlock/wrlock`)。POSIX 信号量（SEM 扩展）分为有名信号量和无名信号量，通过 `sem_wait/sem_post` 实现 P/V 操作。

> [!example]- Windows 同步对象
> Windows 内核支持三种同步对象：**互斥对象**（Mutex, 等价互斥信号量）、**信号量对象**（Semaphore, 等价资源信号量）、**事件对象**（Event, 相当于触发器）。此外还有**临界区对象 (critical section)**，只能在同一进程内使用（非内核对象）。等待函数 `WaitForSingleObject` / `WaitForMultipleObjects` 用于阻塞等待对象变为有信号状态。
