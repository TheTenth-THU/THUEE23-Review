## 互斥与同步

### 问题引入

在多道程序设计环境下，并发执行的多个进程相互隔离、独立异步地向前推进。虽然进程隔离保证了稳定性和安全性，但不同进程需要协同工作时便面临挑战。**进程间通信 (IPC, Inter-Process communication)**主要解决两个问题：

+ **信息共享与数据交换**——高级进程间通信
+ **共享资源的有序访问**——互斥与同步

如果没有有序访问机制，IPC 带来的不是协作而是混乱，典型的后果是**竞争条件 (race condition)**——多个进程访问共享资源时，最终结果取决于进程运行的精确时序。

> [!definition] 互斥与同步
> **互斥 (mutual exclusion)**：确保共享资源在某一时刻只能被一个进程拥有，解决"同一时刻访问"问题。
>
> **同步 (synchronization)**：协作进程之间相互等待对方消息或信号的协调关系，解决"先后顺序"问题。互斥是一种特殊的同步。

互斥产生于**资源共享**（无意识的安排），同步产生于**进程合作**（有意识的安排）。互斥与同步问题对于线程同样适用。

### 临界资源和临界区

系统中某些共享资源一次只允许一个进程使用，这类资源称为**临界资源 (critical resource)**。硬件临界资源如打印机、光盘刻录机；软件临界资源如只能互斥访问的变量、表格、队列。并非所有共享资源都是临界资源（如只读数据）。

**临界区 (critical section)**是进程中访问临界资源的代码片段，其结构如下：

```
进入区 (entry section)    // 检查可否进入，设置"正在访问"标志
临界区 (critical section)  // 访问临界资源
退出区 (exit section)      // 清除"正在访问"标志
剩余区 (remainder section) // 代码其余部分
```

> [!theorem] 访问临界区的四个原则
> 1. **空闲让进**——当无进程在临界区时，任何有权进程可以进入
> 2. **忙则等待**——不允许两个以上进程同时进入临界区
> 3. **有限等待**——任何进入临界区的要求应在有限时间内得到满足
> 4. **让权等待**——不能进入临界区的进程应放弃占用 CPU

## 软件互斥算法

### 锁变量法

使用共享变量 `lock` 标记临界区是否被占用：

```
while (lock == 1);   // 忙等待
lock = 1;
// 临界区
lock = 0;
```

**问题**：`while(lock==1)` 和 `lock=1` 之间可能发生进程切换（检查和设置操作不原子），导致违反"忙则等待"原则；同时存在**忙等待 (busy waiting)**——连续测试一个变量直到出现某个值，浪费 CPU 时间。用于忙等待的锁称为**自旋锁 (spin lock)**。

### 轮转法

设置变量 `turn` 记录轮到哪个进程进入临界区，进程严格轮流进入。**问题**：违反"空闲让进"（一个进程执行完临界区后短时间内不再需要进入时, 另一个进程无法连续进入）。

### Peterson 算法

设置整型变量 `turn` 和布尔数组 `interested[N]`：

```
// 进程 i 的 enter_region:
interested[i] = TRUE;
turn = j;                          // j 为另一个进程
while (interested[j] && turn == j);  // 忙等待

// 进程 i 的 leave_region:
interested[i] = FALSE;
```

只有当对方不感兴趣(`interested[other] == FALSE`)或轮到本进程(`turn == i`)时，进程才能进入临界区。

+ **优点**：正确解决了两个进程的互斥问题，克服了轮转法的缺点
+ **缺点**：忙等待

> [!danger] 标志牌算法的死锁
> 进程 0 和进程 1 各自拥有标志牌 `flag[0]`、`flag[1]`，进入临界区前先举牌再观察对方——若两个进程几乎同时举牌，则互相看到对方举牌，都在 while 循环上忙等，形成**死锁**。

## 硬件互斥方法

### 禁止中断

进入临界区前关中断，离开后开中断。**简单但不实用**——把特权指令交给用户进程不可靠，且不适用于多处理器。

### 专用硬件指令

利用处理器提供的原子指令（读-修改-写由一条指令完成, 不被打断）。

**Test-and-Set 指令**（如 x86 的 BTS/BTR）：

```
enter_region:
    LOCK BTS lock, 0     // 测试 lock 第 0 位，并将其置 1
    JC enter_region      // 原来为 1 则忙等
    RET

leave_region:
    LOCK BTR lock, 0     // 将 lock 第 0 位清 0
    RET
```

**SWAP 指令**（如 x86 的 XCHG）：

```
enter_region:
    MOV reg, 1
    LOCK XCHG lock, reg  // 交换 lock 和 reg
    CMP reg, 0           // reg 原来为 1 则忙等
    JNZ enter_region
    RET

leave_region:
    LOCK MOV lock, 0
    RET
```

+ **优点**：适用于任意数目进程；简单且易验证正确性；可支持进程中多个临界区
+ **缺点**：忙等待

> [!danger] 优先级反转问题
> 忙等待可能导致**优先级反转 (priority inversion)**：高优先级进程 H 就绪后抢占正在临界区中的低优先级进程 L，但因 L 被抢占无法离开临界区，H 在忙等循环中无限等待。H 等待 L，但 H 优先级高于 L 使 L 无法运行——这就是优先级反转。

## 信号量

软件和硬件互斥方法均为平等进程间的协商机制，且都有忙等待的缺点。1965 年 Dijkstra 提出**信号量 (semaphore)**概念——由 OS 作为地位高于进程的管理者来解决公有资源的使用问题。

### 信号量的数据结构

每个信号量 $s$ 包含：
+ `s.count`：整型计数值
+ `s.queue`：进程等待队列

信号量只能通过初始化以及 P、V 两个原语来访问——作为 OS 核心代码执行，不受进程调度的打断。

信号量计数值的含义（资源信号量）：
+ `s.count > 0`：有 count 个资源可用
+ `s.count = 0`：无资源可用
+ `s.count < 0`：|count| 表示等待队列中的进程个数

若只需要对临界资源的互斥访问，可使用**互斥信号量**（通常初值为 1）。

### P、V 原语

```
P(s):                         V(s):
    --s.count;                    ++s.count;
    if (s.count < 0) {            if (s.count <= 0) {
        调用进程进入 s.queue;          从 s.queue 取出一个进程 P;
        阻塞调用进程;                  P 进入就绪队列;
    }                            }
```

P(wait/down)表示申请资源，V(signal/up)表示释放资源。

### 应用模式

**实现互斥**：设置互斥信号量 `mutex`，初值为 1。

```
P(mutex);
// 临界区
V(mutex);
```

**实现同步（前趋关系）**：为每个前趋关系设置信号量，初值为 0。若要求 C1 在 C2 开始前完成：

```
P1:  C1;  V(S12);          P2:  P(S12);  C2;
```

> [!danger] P、V 使用要点
> + 必须**成对使用**，不能重复或遗漏（遗漏 P 不能保证互斥, 遗漏 V 不能释放资源）
> + 互斥操作时，P 和 V 在同一进程中；同步操作时，P 和 V 不在同一进程中
> + **P 操作次序至关重要**：同步 P 必须在互斥 P 之前，否则可能死锁；两个 V 操作顺序无关紧要

## 经典 IPC 问题

经典 IPC 问题是描述、分析和测试进程同步与互斥机制正确性的模型。

### 生产者-消费者问题

也称为**有限缓冲区问题 (bounded-buffer problem)**。N 个数据单元的共享缓冲区，生产者写入，消费者读出，任何时刻只能有一个进程操作缓冲区。

```
信号量：full = 0, empty = N, mutex = 1

// 生产者                          // 消费者
P(empty);                          P(full);
P(mutex);                          P(mutex);
// 将一个数据单元放入缓冲区          // 从缓冲区取出一个数据单元
V(mutex);                          V(mutex);
V(full);                           V(empty);
```

> [!danger] P 操作次序
> 必须先 `P(empty)` 再 `P(mutex)`（消费者先 `P(full)` 再 `P(mutex)`）——若先 P(mutex) 再 P(empty)，当缓冲区满时生产者持有互斥锁但等待 empty，消费者无法获得互斥锁来释放资源，形成死锁。两个同步 P 之间以及两个同步 V 之间次序无要求。

### 读者-写者问题

两组并发进程（读者、写者）共享一组数据区。要求：允许多个读者同时读；不允许读者/写者同时操作；不允许多个写者同时操作。

```
信号量：mutex = 1（写互斥）, rc_mutex = 1（保护读者计数）

// 读者                            // 写者
P(rc_mutex);                       P(mutex);
rc++;                              // 写操作
if (rc == 1) P(mutex);             V(mutex);
V(rc_mutex);
// 读操作
P(rc_mutex);
rc--;
if (rc == 0) V(mutex);
V(rc_mutex);
```

第一个读者进入时锁住写者，最后一个读者离开时释放写者。

### 哲学家进餐问题

Dijkstra 提出。5 个哲学家围圆桌而坐，5 把叉子（每两个哲学家之间一支）。进餐需同时拿起左右两把叉子。

**错误算法**（死锁）：每个哲学家同时拿起左边叉子 → 所有人都在等右边叉子。

**正确算法**：使用互斥信号量数组 `fork[5]`，每个初值为 1；并引入限制——最多允许 4 个哲学家同时进餐，或要求奇数号哲学家先拿左叉、偶数号先拿右叉。

### 睡眠理发师问题

理发店有一位理发师、一把理发椅和 N 把等候椅。无顾客时理发师在理发椅上睡觉；顾客到来须唤醒理发师；理发师正忙时若有空椅则坐下等，否则离开。

## 管程

### 引入动机

信号量的同步操作分散在各个进程中，导致：正确性分析困难、易读性差、不利于修改和维护、大系统中正确性难以保证。

1971 年 Dijkstra 提出"秘书"进程思想，1973 年 Hansen 和 Hoare 推广为**管程**。

> [!definition] 管程
> **管程 (monitor)**是关于共享资源的数据结构及一组针对该资源的操作过程所构成的软件模块。基本思想是把信号量及其操作原语封装在一个对象内部——将共享变量及对其能够进行的所有操作集中在一个模块中。

### 管程的主要特性

+ **模块化**：一个管程是一个基本程序单位，可以单独编译
+ **抽象数据类型**：不仅有数据，还有对数据操作的代码
+ **信息封装**：共享变量在管程外部不可见，外部只能通过调用管程中的过程来间接访问

### 管程的组成

管程内部包含：共享变量、**条件变量 (condition variables)**、管程过程。

**互斥进入**：当一个进程调用管程过程时，若管程中已有其他活跃进程，调用进程被阻塞在**入口等待队列**。

**条件变量**是管程内的一种数据结构，只能通过 `wait` 和 `signal` 两个原语操作：
+ `wait(c)`：调用进程阻塞并移入与条件变量 c 相关的等待队列，释放管程
+ `signal(c)`：若 c 链为空则空操作；否则唤醒 c 链中第一个等待者（Hoare 语义下, 唤醒者等待, 被唤醒者继续）

当执行唤醒操作时，管程中存在两个同时活动的进程，通常使用**紧急等待队列**管理这种情形。

> [!note] 管程 vs 信号量中的 signal/V
> 管程中的 `signal` 操作与信号量中的 V 操作不同：若无等待进程，管程的 signal 被忽略（不会记录历史），而信号量的 V 操作会使信号量值增加，影响后续的 P 操作。

管程是**编程语言概念**，编译器必须识别管程并对其互斥做出安排。Java、Pascal 等支持管程，但 C、C++ 及多数语言不直接支持管程。

## 高级进程间通信

信号量和互斥锁等同步机制主要用于协调执行顺序和传递状态信息，不适合复杂或大规模数据传输。在需要结构化或大量数据交换时，需引入更高级的通信机制。

### 共享内存

在物理内存中划出一块**共享存储区**，通信进程将其映射到各自的进程地址空间，通过直接读写共享区数据实现通信。

+ **优点**：效率高，进程直接读写内存无需数据拷贝
+ **问题**：操作系统只提供共享内存空间，进程间的读写互斥由程序开发人员负责

> [!example]- Linux/Windows 共享内存机制概览
> Linux 支持 `mmap()` 系统调用、POSIX 共享内存和 System V 共享内存三种方式。Windows 采用文件映射机制(`CreateFileMapping` / `MapViewOfFile`)，将文件映射为进程虚拟地址空间的一部分。

### 消息传递

进程间以**消息 (message)**为单位进行数据交换。分为两种方式：

**直接通信**：发送和接收进程显式指定对方标识，实现点对点通信。原语形式为 `send(receiver, message)` / `receive(sender, message)`。主要用于早期微内核系统，现代系统中逐渐被淘汰——通信关系绑定在进程身份上，无法处理服务重启、迁移和复杂通信模式。

**间接通信**：进程通过中介实体（邮箱、消息队列）进行通信，发送者和接收者完全解耦，支持异步通信。原语形式为 `send(mailbox/queue, message)` / `receive(mailbox/queue, message)`。

### 管道

**管道 (pipe)**是 UNIX 首创的以文件系统为基础的通信方式，以字节流方式传送信息。

+ **无名管道(pipe)**：临时对象，通过 `pipe()` 系统调用创建，获得读写两个文件描述符。只适用于父子进程或父进程安排的子进程之间
+ **命名管道(FIFO)**：文件系统的真正实体，可在无亲缘关系的进程间通信

> [!example]- POSIX/Linux 同步机制
> POSIX 提供三种同步 API：**互斥锁**(mutex, `pthread_mutex_lock/unlock`)、**条件变量**(cond, `pthread_cond_wait/signal`)、**读写锁**(rwlock, `pthread_rwlock_rdlock/wrlock`)。POSIX 信号量（SEM 扩展）分为有名信号量和无名信号量，通过 `sem_wait/sem_post` 实现 P/V 操作。

> [!example]- Windows 同步对象
> Windows 内核支持三种同步对象：**互斥对象**（Mutex, 等价互斥信号量）、**信号量对象**（Semaphore, 等价资源信号量）、**事件对象**（Event, 相当于触发器）。此外还有**临界区对象 (critical section)**，只能在同一进程内使用（非内核对象）。等待函数 `WaitForSingleObject` / `WaitForMultipleObjects` 用于阻塞等待对象变为有信号状态。
