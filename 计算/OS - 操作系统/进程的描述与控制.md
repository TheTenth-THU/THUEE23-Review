## 程序的执行方式

### 顺序执行

**程序的顺序执行**是指计算机系统中只有一个程序在运行，该程序独占系统中的所有资源，一个程序执行完成后另一个程序才能开始执行。这是单道批处理系统的执行方式。

顺序执行具有三个特征：

- **顺序性**：一个程序必须等到前一个程序执行完成才能开始执行
- **封闭性**：程序独占全部资源，计算机的状态只由该程序的控制逻辑决定
- **可再现性**：初始条件相同则结果相同

### 并发执行

**程序的并发执行**是指一定时间内，系统中有两个或两个以上的程序同处于开始运行但尚未结束的状态，并且次序不是事先确定的。引入并发执行的目的是提高资源利用率。

并发执行的特征与顺序执行截然相反：

- **间断性（异步性）**："走走停停"，一个程序可能走到中途停下来，失去原有的时序关系
- **失去封闭性**：共享资源，受其他程序的控制逻辑影响
- **失去可再现性**：失去封闭性 → 失去可再现性

> [!danger] 并发执行的关键问题
> 程序推进的时序不可再现是允许的（并发本身的特征），但**执行的结果不可再现是不可接受的**。因此操作系统必须提供机制来协调并发程序的执行。

> [!example]+ 并发执行结果不可再现的示例
> 订票系统，两个终端运行 T1、T2 程序：
>
> ```
> T1:                    T2:
> ① read(x)             ① read(x)
> ② if x >= 1 then      ② if x >= 1 then
> ③   x := x - 1        ③   x := x - 1
> ④ write(x)            ④ write(x)
> ```
>
> 假设执行顺序为 T1①②③ → T2①②③④ → T1④，卖出 2 张票，写回数据库的票款余额只减了 1——这就是并发执行结果不可再现的典型案例。

程序并发执行引发的问题：程序间同步与互斥、作业调度、进程调度、内存管理——这些正是操作系统需要解决的核心问题。

## 进程的概念

### 引入动机

当允许程序并发执行时，用"程序"概念已不能方便地描述程序的并发执行——并发执行的程序可能是同一个程序在不同数据集合上的执行，也可能是不同的程序在不同数据集合上的执行，它们共享系统资源。为了描述程序在并发执行时对系统资源的共享，需要一个描述程序执行时**动态特征**的概念，因此引入了**进程**。

> [!definition] 进程的定义
> **进程**（Process）是具有独立功能的程序在某个数据集合上的一次运行活动，是系统进行**资源分配和调度**的独立单位。进程有时也称为**任务**（Task）。

### 进程的特征

- **动态性**：进程是程序的执行，动态产生、动态消亡，在生命周期内在多种状态间转换
- **独立性**：进程是资源分配的一个独立单位，各进程的地址空间相互独立
- **并发性**：任何进程都可以同其他进程一起向前推进
- **异步性**：每个进程都以其相对独立的、不可预知的速度向前推进
- **交互性**：进程在执行过程中可能与其他进程产生直接或间接的关系
- **结构化**：进程的组成 = 程序 + 数据 + PCB

### 进程与程序的区别

| 方面 | 进程 | 程序 |
|------|------|------|
| 性质 | 动态的（程序的执行） | 静态的（有序代码的集合） |
| 生命周期 | 暂时的（状态变化的过程） | 长久的（可长久保存） |
| 组成 | 程序 + 数据 + PCB | 仅代码 |
| 对应关系 | 一个程序可对应多个进程；一个进程可包括多个程序 | — |

## 进程控制块

**进程控制块**（PCB，Process Control Block）是操作系统为了管理进程而设置的一个专门的数据结构，用来记录进程的外部特征，描述进程的运动变化过程。系统利用 PCB 来控制和管理进程，所以 **PCB 是系统感知进程存在的唯一标志**。进程与 PCB 是一一对应的。

### PCB 的内容

PCB 通常包含以下信息：

- **进程描述信息**：进程标识符（PID，唯一整数）、进程名、用户标识符（UID）、进程组
- **进程控制信息**：当前状态、优先级、代码执行入口地址、程序外存地址、运行统计信息（执行时间、页面调度）、进程间同步和通信信息、阻塞原因
- **资源占用信息**：虚拟地址空间现状、打开文件列表
- **CPU 现场信息**：寄存器值（通用寄存器、程序计数器 PC、状态寄存器 PSW、栈指针）、指向段/页表的指针

### PCB 表的组织方式

系统把所有 PCB 组织在一起，放在内存的固定区域，构成 **PCB 表**（进程表）。PCB 表的大小决定了系统中最多可同时存在的进程个数，称为系统的**并发度**。

两种主要的组织方式：

- **链接结构**：同一状态的进程 PCB 组成链表（就绪链表、阻塞链表等），多个状态对应多个不同链表
- **索引结构**：同一状态的进程归入一个索引表（由索引指向 PCB），多个状态对应多个不同的索引表

## 进程映像与上下文

### 进程映像

进程在内存中的完整映像（Process Image）包含：

- **用户地址空间**：程序代码（Text）、用户数据（Data）、堆（Heap，动态内存分配）、栈（Stack，过程调用和参数传递）
- **内核空间**：PCB，用户进程不能直接访问和修改自己的 PCB

### 进程上下文

**进程上下文**（Context）是对进程执行过程的描述，由三部分组成：

- **用户级上下文**：进程的用户地址空间（用户正文段、用户数据段、用户栈）
- **寄存器级上下文**：程序计数器、处理机状态寄存器、栈指针、通用寄存器的值
- **系统级上下文**：PCB 和资源表格、核心栈（进程调用系统核心过程时需要）

## 进程状态模型

### 三种基本状态

进程在其生命周期中会在以下三种基本状态之间转换：

```
          ┌──────────┐
     ┌───→│ 运  行  │───┐
     │    │ Running │   │
     │    └──────────┘   │
     │ ③  ↑       ↓ ①   │
     │    │       ↓      │
┌───────┐│    ┌────────┐ │
│ 就  绪 ││←───│ 阻  塞 │←┘
│ Ready ││ ④  │Blocked │
└───────┘└───→└────────┘
     ②
```

状态转换条件：
1. **运行 → 阻塞**：进程等待某事件（资源不可访问、I/O 初始化等待结果、等待 IPC）
2. **运行 → 就绪**：时间片用完，或被高优先级进程抢占
3. **就绪 → 运行**：调度程序选择该进程运行
4. **阻塞 → 就绪**：所等待的事件发生

> [!danger] 常见误区
> **阻塞 → 运行** 和 **就绪 → 阻塞** 是不可能直接发生的状态转换。阻塞进程必须先经过就绪态才能被调度运行，就绪进程必须经过运行态才能因等待事件而阻塞。

### 五状态进程模型

在三态基础上增加**新建态**（New）和**终止态**（Terminated）：

- **新建态**：OS 已完成为创建进程所必要的工作（分配 PID、创建管理表格），但因资源有限尚未许可执行
- **终止态**：进程被撤销后进入，不再有执行资格，但表格和其他信息可能暂时保留

### 七状态进程模型

当系统内存不足时，可能将进程暂时换出到外存，引入**挂起状态**（Suspended）：

- **就绪挂起**（Suspended Ready）：进程在外存，但只要进入内存即可运行
- **阻塞挂起**（Suspended Blocked）：进程在外存并等待某事件的出现

**挂起**（Suspend）操作将进程从内存转到外存：
- 阻塞 → 阻塞挂起（释放内存以提交新进程）
- 就绪 → 就绪挂起（挂起低优先级就绪进程）
- 运行 → 就绪挂起（高优先级阻塞挂起进程进入就绪挂起时，抢占当前进程）

**激活**（Activate）操作将进程从外存调回内存：
- 就绪挂起 → 就绪（没有就绪进程或挂起就绪进程优先级更高）
- 阻塞挂起 → 阻塞（释放足够内存，激活高优先级阻塞挂起进程）

## 进程控制

**进程控制**是操作系统使用具有特定功能的程序段来创建与撤销进程以及完成进程状态转换，以达到多进程高效率并发执行和协调、实现资源共享的目的。

进程控制由**原语**（Primitive）完成。原语是由若干条指令构成的**原子操作**过程——作为一个整体而不可分割（要么全都完成，要么全都不做）。

> [!tip]
> 并非所有系统调用都是原语。例如进程 A 调用 `read()` 因无数据而阻塞，在 `read()` 里未返回；然后进程 B 调用 `read()`，此时 `read()` 被重入。系统调用不一定一次执行完并返回该进程，有可能在特定点暂停而转入其他进程。

### 核心原语

**创建原语**：为新进程分配 PCB 空间 → 填入进程信息 → 将 PCB 送入就绪队列 → 转处理机调度

**撤销原语**：搜索 PCB 链表找到目标 PCB → 释放该进程占有的所有资源 → 释放 PCB 结构本身 → 转处理机调度。导致撤销的原因：正常退出（自愿）、出错退出（自愿）、严重错误（非自愿）、被其他进程杀死（非自愿）。

**阻塞原语**：保存进程上下文 → 设置进程状态为阻塞 → 将 PCB 置入阻塞队列 → 转处理机调度

**唤醒原语**：从阻塞队列删除 PCB → 设置进程状态为就绪 → 将 PCB 置入就绪队列 → 转处理机调度

**挂起原语**：将进程从内存调出到外存 → 根据当前状态（就绪/阻塞）从对应队列删除 → 设置为相应的挂起状态 → 置入对应挂起队列

**激活原语**：将进程从外存调入内存 → 根据当前状态（就绪挂起/阻塞挂起）从对应队列删除 → 设置为相应状态 → 置入对应队列

> [!example]- Linux 进程描述符
> Linux 的 PCB 是内核数据结构 `task_struct`（定义在 `include/linux/sched.h`），约 1000 字节，包含状态、调度、内存管理、身份标识、家族关系、时间统计、文件系统和 IPC 等信息。所有进程通过双向循环链表组织为进程链表，表头为 `init_task` 的 PCB。PID 类型为 `pid_t`，取值范围 0~32767。
>
> Linux 进程状态：`TASK_RUNNING`（可运行/正在运行）、`TASK_INTERRUPTIBLE`（浅度睡眠，可被信号唤醒）、`TASK_UNINTERRUPTIBLE`（深度睡眠，不可被信号唤醒）、`TASK_STOPPED`（暂停，用于调试）、`TASK_ZOMBIE`（僵死，已结束运行但 PCB 未释放）。

> [!example]- Windows 进程控制块
> Windows 的进程由执行体进程对象 `EPROCESS` 描述，PCB 也称为内核进程块 `KPROCESS`。创建进程使用 `CreateProcess()`，终止使用 `ExitProcess()`（完整终止）或 `TerminateProcess()`（异常情况下使用，不完整终止）。
