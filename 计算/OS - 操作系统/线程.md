## 线程的引入

### 引入线程的动机

未引入线程时，[[进程]]具有两个基本属性：
+ 进程是**资源分配单位**，拥有虚拟地址空间，保存进程映像，控制一些存储和设备资源，有状态、优先级；
+ 进程是 **CPU调度单位**，是程序的执行轨迹。

这两个属性构成进程并发执行的基础，但也带来了问题：
1. **降低并发开销的需求**：由于进程是资源分配单位，在创建、终止、切换的过程中系统必须付出**较大的时空开销**，因此系统中并发进程数目不宜过多，进程切换频率也不宜过高，这限制了系统的并发程度。
2. **利用多处理器/多核架构的需求**：在多处理器系统和多核处理器中，传统单线程进程在一个时刻只能运行在一个处理器上，不能充分利用硬件并行能力。
3. **应用内在的并发需求**：许多应用（如字处理程序、Web 服务器）天然存在可并发的任务，需要操作系统提供轻量级的并发支持。

> [!note] 多处理器系统架构
> 
> 按照处理器之间的关系，多处理器系统分为：
> + **对称多处理 (symmetric multiprocessing, SMP)**，所有CPU地位对等，共享同一个操作系统实例，任意CPU均可执行内核代码和用户程序。SMP是现代通用CPU的主流模型。
> + **非对称多处理 (asymmetric multiprocessing, AMP)**，CPU之间存在主从关系，常见于嵌入式系统。
> 
> 按照内存组织方式，多处理器系统分为：
> + 共享内存架构，又分为——
>     + **统一内存访问 (uniform memory access, UMA)**，所有CPU共享一块物理内存，访问延迟相同。UMA结构简单，但扩展性差。
>     + **非统一内存访问 (non-uniform memory access, NUMA)**，每个节点有独立的局部内存，访问本地内存快、远程内存慢。NUMA扩展性好，适合大规模服务器。
> + **非共享内存架构**，典型的是**集群系统 (cluster)**，每个处理部件都是一个独立的计算机，通过高速网络互连，构成廉价的高性能计算机系统。

### 线程的基本概念

解决以上问题的思路是，**将进程的两个基本属性分开**，引入线程的概念，作为进程的组成部分。在支持线程的操作系统中，进程只作为**资源分配单位**，线程则作为 **CPU调度单位**。

> [!definition] 线程
> **线程 (thread)** 是进程中能够独立执行的实体（控制流），是**处理器调度的基本单位**。
> 每个进程内允许包含多个并发执行的线程，这就是**多线程 (multithreading)**。同一进程中的所有线程共享进程获得的主存空间和资源。

在多线程环境下，**进程是操作系统中除处理器外进行资源分配和保护的基本单位**。进程拥有独立的虚拟地址空间，容纳进程映像，并以进程为单位对各种资源实施保护，例如受保护地访问文件、外部设备及其他进程（即进程间通信）。

由于处理机调度的基本单位现在是线程而不是进程，因此在创建一个进程时，同时要为该进程创建一个**主线程**，否则该进程无法被调度执行。

### 线程的属性

线程概念试图实现的是**共享一组资源**的**多个线程的执行能力**，以便这些线程可以为完成某一任务而共同工作。从资源归属角度，线程的属性可划分为**线程级资源**（各线程独享）和**进程级资源**（同进程内所有线程共享）。

#### 线程级资源

每个线程拥有自己**独立的执行上下文**，这些资源是**线程私有**的，创建线程时分配，切换线程时保存和恢复。

```tx
| 资源 | 说明 |
| :- | :- |
| 线程标识 (TID) | 线程的唯一标识，记录在 TCB 中 |
| 基本状态 | 运行、就绪、阻塞；各线程状态独立，转换类似于[[进程]] |
| CPU 现场（上下文） | 通用寄存器、程序计数器 (PC)、状态寄存器 (PSW) |
| 执行栈 | 用户栈（函数调用参数、返回地址、局部变量）和内核栈（陷入内核时使用） |
| 线程控制块 (TCB) | 包含线程标识、线程状态、调度参数、CPU 现场信息等 |
```

> [!note] 线程栈的双重角色
> 线程的栈包含**用户栈**和**内核栈**两部分。用户栈用于用户态函数调用；当线程通过系统调用陷入内核时，使用内核栈保存内核态的执行上下文。内核级线程的 TCB 中同时记录用户栈指针和系统栈指针。

#### 进程级资源

同一进程内的所有线程**共享**以下资源，这些资源由进程统一获取和管理，线程间可直接访问而无需通过内核：

```tx
| 共享资源 | 说明 |
| :- | :- |
| 虚拟地址空间 | 所有线程运行在同一个进程地址空间内，进程映像为所有线程所见 |
| 程序代码段（正文段） | 同一份代码可被多个线程并发执行（只读共享，天然线程安全） |
| 数据段 | 全局变量、静态变量对所有线程可见 |
| 堆 (heap) | `malloc`/`new` 分配的动态内存，所有线程均可读写 |
| 打开文件列表 | 文件描述符进程内共享——一个线程打开的文件可被同进程其他线程直接读写 |
| 信号处理设置 | 信号处理函数对进程内所有线程生效 |
| PCB 中的资源信息 | PID、UID、内存管理信息、资源占用统计等由 PCB 统一记录 |
```

> [!tip] 共享带来的同步需求
> 线程间共享数据段和堆内存使得通信非常高效（直接读写全局变量即可），但也带来了**竞态条件 (race condition)** 的风险——多个线程同时读写同一数据可能导致结果不确定。因此线程间通信需要**同步机制**（互斥锁、信号量等）来协调对共享数据的访问。

> [!danger] 局部变量是否线程安全？
> 线程栈上的**局部变量**是线程私有的——它们分配在各线程自己的栈帧中，其他线程无法直接访问。因此局部变量天然线程安全，无需额外的同步保护。但若局部变量是**指向堆内存的指针**，则指针本身安全而所指向的堆数据仍需同步保护。

> [!tip] 线程控制
> 线程之间还可以互相进行控制操作，例如一个线程可以创建、撤销同进程内的另一个线程。线程创建的开销远小于进程创建，因为无需重新分配地址空间和资源表。

> [!note] 挂起状态是进程级状态
> 挂起状态对线程没有意义——如果进程挂起后被换出主存，则它的所有线程（共享进程的地址空间）也必须全部换出。类似地，进程的终止会导致所有线程的终止。

### 线程控制块

**线程控制块 (TCB, thread control block)** 是标志线程存在的数据结构，包含线程标识、线程状态、调度参数和 CPU 现场信息（通用寄存器、指令计数器 PC、用户栈指针；内核支持的线程还包含系统栈指针）。TCB 中的内容一般较少，因为资源分配等信息已记录于所属进程的 PCB 中。

### 进程与线程的比较

| 方面 | 进程 | 线程 |
|------|------|------|
| 地址空间与资源 | 相互独立 | 同一进程内共享，其他进程不可见 |
| 通信 | 需要 IPC 机制 | 可直接读写进程数据段（需同步辅助） |
| 调度与切换 | 上下文切换开销大 | 上下文切换比进程快得多 |
| 创建与结束 | 花费时间长 | 创建和结束开销小 |
| 崩溃影响 | 不影响其他进程 | 可能影响整个进程的稳定性 |

## 线程的实现机制

### 用户级线程

**用户级线程 (ULT, user level thread)**的线程管理工作完全在用户空间内完成，操作系统内核**不知道线程的存在**。

线程管理通过用户空间的**线程库**（运行时系统）实现，它是管理 ULT 的例程包。应用程序需调用线程库中的例程进行编程，与线程库链接后形成可执行程序。

线程库的功能：创建与撤销线程、线程间传递消息和数据、调度线程执行、保护和恢复线程上下文。

每个进程需要有专用的**线程表 (thread table)**，由运行时系统管理。线程表中的每个表项即 TCB。

> [!note] ULT 与内核的关系
> 操作系统内核不知道线程的活动，仍然以**进程**为单位进行处理器调度。当线程调用阻塞型系统调用时，**整个进程阻塞**。线程状态与进程状态是独立的——可能线程处于运行态而进程处于就绪态。

+ **优点**：线程切换不调用内核（性能好）；调度是应用特定的，可针对应用优化；无论内核是否支持线程，只需要线程库即可运行
+ **缺点**：大多数系统调用是阻塞的，内核阻塞进程则进程中所有线程被阻塞；同一进程的两个线程不能同时运行于两个处理器上

### 内核级线程

**内核级线程 (KLT, kernel level thread)**的线程管理工作由**内核**完成。内核中需要有记录系统中所有线程的**线程表**。不需要线程库，每个进程中不需要线程表。

+ **优点**：在多处理器上，内核能同时调度同一进程中的多个线程并行执行；阻塞在**线程一级**完成（一个线程阻塞不影响同进程其他线程）；内核自身也可用多线程技术实现
+ **缺点**：线程在用户态运行而调度管理在内核实现，需要频繁进行模式切换，系统开销较大

### 混合实现

混合实现将 ULT 映射到 KLT 上：内核只认识 KLT 并对其进行调度，而某些 KLT 的顶部有若干可以轮流使用的 ULT 集合。线程创建、调度、同步在用户级完成，程序员可以调节 KLT 数量以求最佳效果。

> [!example]- Solaris 的三层线程模型
> Solaris 实现了三个层次：**ULT** → **LWP**（Light Weight Process, 轻量级进程）→ **KLT**。LWP 是基于内核线程的高级抽象，由内核独立调度，能在多处理器中并发执行。LWP 作为 ULT 和 KLT 之间的桥梁，实现了多对多映射，灵活高效。但从 Solaris 10 之后因其复杂度过高而被放弃。

> [!example]- POSIX 线程(pthread)
> `pthread` 是广泛采用的 POSIX 多线程编程接口。线程库提供 `pthread_create()`、`pthread_exit()`、`pthread_join()` 等例程。程序员通过链接 pthread 库即可实现多线程，无需关心底层是 ULT、KLT 还是混合实现。

## 引入线程的权衡

**优点**：
+ 减小并发执行的时间和空间开销（创建、退出、调度），允许建立更多线程来提高并发度
+ 线程间通信无需调用内核（同进程内线程共享内存和文件）
+ 适合多处理机系统

**缺点**：
+ 程序设计模型引入额外的复杂性
+ 一个线程的崩溃可能影响整个进程的稳定性
+ 线程过多后调度变复杂，需要消耗较多 CPU 资源
