## 线程的引入

### 引入线程的动机

未引入线程时，[[进程]]具有两个基本属性：
+ 进程是**资源分配单位**，拥有虚拟地址空间，保存进程映像，控制一些存储和设备资源，有状态、优先级；
+ 进程是 **CPU调度单位**，是程序的执行轨迹。

这两个属性构成进程并发执行的基础，但也带来了问题：
1. **降低并发开销的需求**：由于进程是资源分配单位，在创建、终止、切换的过程中系统必须付出**较大的时空开销**，因此系统中并发进程数目不宜过多，进程切换频率也不宜过高，这限制了系统的并发程度。
2. **利用多处理器/多核架构的需求**：在多处理器系统和多核处理器中，传统单线程进程在一个时刻只能运行在一个处理器上，不能充分利用硬件并行能力。
3. **应用内在的并发需求**：许多应用（如字处理程序、Web 服务器）天然存在可并发的任务，需要操作系统提供轻量级的并发支持。

> [!note] 多处理器系统架构
> 
> 按照处理器之间的关系，多处理器系统分为：
> + **对称多处理 (symmetric multiprocessing, SMP)**，所有CPU地位对等，共享同一个操作系统实例，任意CPU均可执行内核代码和用户程序。SMP是现代通用CPU的主流模型。
> + **非对称多处理 (asymmetric multiprocessing, AMP)**，CPU之间存在主从关系，常见于嵌入式系统。
> 
> 按照内存组织方式，多处理器系统分为：
> + 共享内存架构，又分为——
>     + **统一内存访问 (uniform memory access, UMA)**，所有CPU共享一块物理内存，访问延迟相同。UMA结构简单，但扩展性差。
>     + **非统一内存访问 (non-uniform memory access, NUMA)**，每个节点有独立的局部内存，访问本地内存快、远程内存慢。NUMA扩展性好，适合大规模服务器。
> + **非共享内存架构**，典型的是**集群系统 (cluster)**，每个处理部件都是一个独立的计算机，通过高速网络互连，构成廉价的高性能计算机系统。

### 线程的基本概念

解决以上问题的思路是，**将进程的两个基本属性分开**，引入线程的概念，作为进程的组成部分。在支持线程的操作系统中，进程只作为**资源分配单位**，线程则作为 **CPU调度单位**。

> [!definition] 线程
> **线程 (thread)** 是进程中能够独立执行的实体（控制流），是**处理器调度的基本单位**。
> 每个进程内允许包含多个并发执行的线程，这就是**多线程 (multithreading)**。同一进程中的所有线程共享进程获得的主存空间和资源。

在多线程环境下，**进程是操作系统中除处理器外进行资源分配和保护的基本单位**。进程拥有独立的虚拟地址空间，容纳进程映像，并以进程为单位对各种资源实施保护，例如受保护地访问文件、外部设备及其他进程（即进程间通信）。

由于处理机调度的基本单位现在是线程而不是进程，因此在创建一个进程时，同时要为该进程创建一个**主线程**，否则该进程无法被调度执行。

### 线程的属性

线程概念试图实现的是**共享一组资源**的**多个线程的执行能力**，以便这些线程可以为完成某一任务而共同工作。从资源归属角度，线程的属性可划分为**线程级资源**（各线程独享）和**进程级资源**（同进程内所有线程共享）。

#### 线程级资源

每个线程拥有自己**独立的执行上下文**，这些资源是**线程私有**的，创建线程时分配，切换线程时保存和恢复。

```tx
| 线程的私有资源 | 说明 |
| :- | :- |
| 线程标识 (TID) | 线程的唯一标识，记录在 TCB 中 |
| 执行的基本状态 | 运行、就绪、阻塞；各线程状态独立，转换类似于[[进程]] |
| 上下文 | CPU 现场，包含通用寄存器、程序计数器 (PC)、状态寄存器 (PSW) |
| 执行栈 | **用户栈**（函数调用参数、返回地址、局部变量）、**内核栈** |\
|        | ——用户栈用于用户态函数调用；当线程通过系统调用陷入内核时，使用内核栈保存内核态的执行上下文。 |
| [[#线程控制块]] (TCB) | 包含线程标识、线程状态、调度参数、CPU 现场信息等 |\
|                  | ——内核级线程的 TCB 中同时记录用户栈指针和系统栈指针。 |
```

#### 进程级资源

同一进程内的所有线程**共享**以下资源，这些资源由进程统一获取和管理，线程间可直接访问而无需通过内核。

```tx
| 线程间共享资源 | 说明 |
| :- | :- |
| 地址空间 | 所有线程运行在同一个进程的虚拟地址空间内，进程映像为所有线程所见 |
| 全局变量、堆内存 | 全局变量、静态变量、堆上分配的内存空间对所有线程可见 |
| 打开文件 | 文件描述符进程内共享，一个线程打开的文件可被同进程其他线程直接读写 |
| 子进程 | 进程的父子关系为进程级属性 |\
|      | ——通过 `fork()` 创建的子进程属于该进程，所有线程共享这一关系。多线程进程中某线程调用 `fork()` 时，子进程通常只复制调用线程。 |
| 待处理警告 (alarm) | `alarm()` 和间隔定时器是进程级资源，一个进程只能设置一个实时闹钟 |
| 信号及其句柄 | 信号处理函数（句柄）在进程级安装，所有线程共享相同的信号处置 |\
|             | ——但每个线程可拥有独立的**信号掩码**，信号可面向进程投递（由任一线程处理）或面向特定线程投递。 |
| 记账信息 (accounting information) | 累计 CPU 时间、页面调度次数、I/O 统计等 |
| [[进程#进程控制块\|进程控制块]] (PCB) | PID、UID、内存管理信息、资源占用统计等由 PCB 统一记录 |
```

> [!note] 各类变量的线程/进程归属与安全
> 进程地址空间中不同区域的变量，在**可见性**和**线程安全性**上有本质区别。凡是在线程栈帧上分配的内容，均为线程私有的；凡是在进程全局空间中分配的内容，均为线程共享的，共享即意味着需要同步。
>
> ```tx
> | 变量类型 | 存储区域 | 可见性 | 线程安全性 | 说明 |
> | :- | :- | :- | :- | :- |
> | 局部变量 | 线程栈 | 线程私有 | 天然安全 | 分配在各线程自己的栈帧中，其他线程无法访问 |
> | 全局变量 | 数据段 | 所有线程共享 | 需同步保护 | 同进程内任一线程均可读写 |
> | 静态变量 (`static`) | 数据段 | 所有线程共享 | 需同步保护 | 和全局变量一样，编译期分配在数据段 |
> | 堆内存 (`malloc`/`new`) | 堆 | 所有线程共享 | 需同步保护 | 堆是进程级资源，任何线程拿到地址后均可访问 |
> ```
>
> 特别地，指针变量本身若为局部变量，则存储在栈上，属线程私有；但若它指向的是堆内存，则安全的是指针本身，所指向的数据仍为所有线程共享，必须同步保护。
>

线程之间还可以互相进行控制操作，例如一个线程可以创建、撤销同进程内的另一个线程。**线程创建的开销远小于进程创建**，因为无需重新分配地址空间和资源表。

> [!note] 挂起状态是进程级状态
> 挂起状态对线程没有意义——如果进程挂起后被换出主存，则它的所有线程（共享进程的地址空间）也必须全部换出。类似地，进程的终止会导致所有线程的终止。

### 线程控制块

**线程控制块 (thread control block, TCB)** 是标志线程存在的数据结构，包含线程标识、线程状态、调度参数和CPU现场信息。其中，CPU现场信息主要指通用寄存器、指令计数器PC、用户栈指针；内核支持的线程的TCB还包含系统栈指针。

TCB中的内容一般较少，因为资源分配等信息已记录于所属进程的PCB中。不同于PCB严格属于内核，TCB可能属于操作系统空间，也可能属于用户进程空间，这取决于线程的实现方式。

### 进程与线程的比较

相较于进程，线程具有以下特点：
+ **地址空间和其他资源**在进程间相互独立，而在同一进程的**各线程间共享**，某进程内的线程在其他进程不可见；
+ 进程间通信采用IPC机制，线程间可以直接**读写进程数据段**通信，只是仍需要进程同步和互斥手段的辅助以保证数据的一致性；
+ 线程**上下文切换**比进程上下文切换要快得多。

因此，引入线程的**优点**主要在于：
+ 线程的创建、退出和调度都更轻量，减小了并发执行的时间和空间开销，因此容许在系统中建立更多的线程来提高并发程度。
+ 同一进程内的线程共享内存和文件，因此它们之间**相互通信无须调用内核**。
+ 适合多处理机系统。

其**缺点**主要是引入了更多复杂性，在影响稳定性的同时，线程本身的调度也需要消耗较多的CPU资源。


## 线程的实现

### 用户级线程

**用户级线程 (user level thread, ULT)** 的线程管理工作完全在用户空间内完成，操作系统内核**不知道线程的存在**，故线程切换不需要核心态特权。

> [!note] ULT与内核的关系
> 操作系统内核不知道线程的活动，仍然以**进程**为单位进行处理器调度。当线程调用阻塞型系统调用时，**整个进程阻塞**。线程状态与进程状态是独立的，可能线程处于运行态而进程处于就绪态。
> 
> 例如，假设正在执行进程P中的线程T1，可能出现下列情况：
> + 线程T1发出一个阻塞式系统调用（例如I/O操作），导致控制转到内核，内核启动I/O操作，将进程P置为**阻塞态**，并切换到另一个进程。按照由线程库所维护的数据结构，线程T1仍处在**运行态**。
> + 进程P时间片用完，内核从进程P剥夺处理器，把它放入就绪队列，切换到另一个就绪进程。此时，由于进程P中线程T1正在运行，按由线程库维护的数据结构，线程T1仍处于**运行态**，但是进程P处于**就绪态**。
> + 线程T1执行到某处后需要线程T2的某些操作，于是让线程T1变成**阻塞态**，而线程T2从就绪态转为**运行态**，在这一过程中进程P始终处于**运行态**。

线程管理通过用户空间的**线程库**（运行时系统）实现。线程库是管理ULT的例行程序包，包含执行以下功能的例程：
+ 创建与撤销线程；
+ 线程间传递消息和数据；
+ 调度线程执行；
+ 保护和恢复线程上下文。

应用程序需调用线程库中的例程进行编程，与线程库**链接**后形成可执行程序。对于POSIX，`pthread` 是广泛采用的线程库接口。

每个进程需要有专用的**线程表 (thread table)**，由运行时系统管理。线程表中的每个表项即一个TCB，对应一个线程，记录线程标识、线程状态、调度参数和CPU现场信息等。线程表**由线程库管理**，当线程由运行态切换到就绪态或阻塞态时，线程库负责保存重新运行该线程所需的信息到TCB中，并在切换回运行态时恢复上下文。

使用ULT的**优点**主要是：
+ 线程切换不调用内核，性能好；
+ 调度是应用特定的，可针对应用优化，线程库的线程调度算法与操作系统的低级调度算法无关；
+ 无论内核是否支持线程，只需要线程库即可运行。

使用ULT的**缺点**主要是：
+ 调度通常采用非抢先式和更简单的规则；
+ 大多数系统调用是阻塞的，内核阻塞进程则进程中所有线程被阻塞；
+ **同一进程的两个线程不能同时运行于两个处理器上**。

### 内核级线程

**内核级线程 (kernel level thread, KLT)** 的线程管理工作由**内核**完成。内核中需要有记录系统中所有线程的**线程表**，包含每个线程的TCB；相应地，不需要线程库，每个进程中不需要线程表。

对应于ULT的缺点，使用KLT的**优点**主要是：
+ 在多处理器上，内核能同时调度同一进程中的多个线程并行执行；
+ 阻塞在**线程一级**完成，一个线程阻塞不影响同进程其他线程；
+ 内核自身也可用多线程技术实现，从而提高内核的并发能力。

使用KLT的**缺点**主要是，线程在用户态运行，而调度管理在内核实现，需要频繁进行模式切换，系统开销较大。

### 混合实现

混合实现将 ULT 映射到 KLT 上：内核只认识 KLT 并对其进行调度，而某些 KLT 的顶部有若干可以轮流使用的 ULT 集合。线程创建、调度、同步在用户级完成，程序员可以调节 KLT 数量以求最佳效果。

> [!example]- Solaris 的三层线程模型
> Solaris 实现了三个层次：**ULT** → **LWP**（Light Weight Process, 轻量级进程）→ **KLT**。LWP 是基于内核线程的高级抽象，由内核独立调度，能在多处理器中并发执行。LWP 作为 ULT 和 KLT 之间的桥梁，实现了多对多映射，灵活高效。但从 Solaris 10 之后因其复杂度过高而被放弃。
