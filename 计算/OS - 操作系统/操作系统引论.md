## 计算机系统与操作系统

**计算机系统**由硬件和软件两部分组成。计算机硬件是根据电、磁、光、机械等原理构成的各种物理设备的有机组合，是系统赖以工作的实体；软件是计算机系统中程序和有关文件的集合。

现代计算机硬件十分复杂，程序员直接面对硬件编程是不现实的。因此，必须在硬件裸机上加载一层软件来管理整个系统，同时给用户提供一个更容易理解和编程的接口——这层软件就是**操作系统 (operating system, OS)**。

> [!definition] 操作系统的定义
> 操作系统是计算机系统中的一个**系统软件**，它是一系列程序模块的集合：这些模块能有效地组织和管理计算机的软硬件资源，合理地组织计算机的工作流程，控制程序的执行并向用户提供各种服务功能，使得用户能够方便地使用计算机，使整个计算机系统能高效地运行。
>
> 关键要点：系统软件，程序模块的集合，资源管理和用户接口功能。

操作系统的三大作用可以从不同视角理解：

### 视角一：作为扩展机器的操作系统

操作系统为用户提供一台等价的**扩展机器**或**虚拟机 (virtual machine)**，它比底层硬件更容易编程和使用。在裸机基础上，OS 添加了：

+ 设备管理
+ 文件管理
+ 存储管理（内存）
+ 处理机管理(CPU)
+ 作业管理（合理组织工作流程）

### 视角二：作为资源管理者的操作系统

操作系统管理的资源包括 CPU、存储器、外部设备、信息（数据和软件）。管理内容包括：资源的当前状态（数量和使用情况）、资源的分配和回收、访问操作、相应管理策略（包括用户权限）。

从资源管理的观点看，操作系统的首要任务是**跟踪资源使用状况、满足资源请求、提高资源利用率**，以及**协调各程序和用户对资源的使用冲突**。

### 视角三：作为用户接口的操作系统

操作系统向用户提供两种接口：
+ **命令接口**：命令行、菜单式、命令脚本、图形用户接口(GUI)
+ **程序接口**：系统调用(system call)，形式上类似于过程调用，供应用程序使用

计算机系统中，**用户态 (user mode)**与**核心态 (kernel mode)**是两个关键概念：操作系统专指在核心态下运行的软件，而应用软件和操作系统以外的系统软件运行在用户态下。

> [!note] "有效、合理、方便"的含义
> + **有效**：系统效率（如 CPU 利用率）、资源利用率（内存、外设是否忙碌）
> + **合理**：公平性，不公平则可能产生"死锁"或"饥饿"
> + **方便**：提供良好、一致的用户接口，弥补硬件系统的类型和数量差别

## 操作系统的发展历史

推动操作系统发展的主要动力包括：提高资源利用率和系统性能、方便用户、以及计算机硬件技术的进步。操作系统历史可按硬件技术发展划分为四个时代。

### 第 1 代：电子管时代(1946–1955)

以 ENIAC 为代表，运算速度约 1000 次/秒，数万个真空管，占地 100 平方米。没有程序设计语言，更谈不上操作系统——所谓「编程」就是在插线板上插接导线。50 年代早期出现穿孔卡片，但仍为**手工操作**方式，效率极低。

### 第 2 代：晶体管时代(1955–1965)

晶体管发明使计算机可靠性提高，可成批生产。FORTRAN、ALGOL、COBOL 等程序设计语言相继诞生，程序员、操作员和维护人员之间有了明确分工。

**批处理操作系统**——现代操作系统的雏形——应运而生。其思想是：用一台相对便宜的计算机（如 IBM 1401）将作业读到磁带上，再用昂贵的计算机（如 IBM 7094）完成真正的计算，以改进主存和 I/O 设备之间的吞吐量。

> [!note] 批处理的主要特征
> + 用户**脱机**使用计算机：作业提交后直到获得结果之前，用户无法与作业交互
> + 作业**成批处理**

典型系统：FMS(FORTRAN Monitor System)、IBMSYS（IBM 7094 机配备的操作系统），由监控程序、特权指令、存储保护和简单的批处理构成。

### 第 3 代：集成电路时代(1965–1980)

这一时代操作系统发生了质的飞跃，出现了多项关键技术。

**多道程序设计 (multiprogramming)**：将内存分为几个部分，每部分放不同的作业。当某个作业因等待 I/O 而暂停时，另一个作业可以使用 CPU。

> [!note] 多道程序设计的特点（单 CPU）
> + **多道**：内存中同时存放多道相互独立的程序
> + **宏观上并行**：多道程序都在运行，在一定时间间隔内都取得进展
> + **微观上串行**：任意时刻，只有一道程序占有 CPU

多道批处理系统引入的关键技术：作业调度（上下文切换）、资源共享（互斥和同步机制）、内存使用（覆盖、交换、虚拟存储）、内存保护、文件非顺序存放和随机存取。

**分时系统 (timesharing system)**：实际上是多道程序的一个变种。将 CPU 时间分割成**时间片 (time slice)**，每个用户依次轮流使用时间片。每个用户有一个联机终端，能够与作业进行交互。

分时系统的特征：**多路性**（多个用户同时工作）、**独立性**（各用户互不干扰）、**交互性**（系统及时响应用户操作, 显著提高调试和修改程序的效率）。

**UNIX 的崛起**：1969 年，贝尔实验室的 Ken Thompson 在一台闲置的 PDP-7 上用汇编语言编写了操作系统内核、文件系统和 shell，取名 UNICS。后与 Dennis Ritchie 合作，用 C 语言重写了 UNIX，使其具有良好的可移植性。UNIX 成为现代操作系统的代表。

### 第 4 代：大规模集成电路时代(1980–)

个人计算机时代到来，各种类型的个人计算机和操作系统层出不穷。典型系统包括：

+ 单用户单任务：MS-DOS
+ 单用户多任务：Windows Desktop、Mac OS X
+ 多用户多任务：UNIX、Linux、Windows Server

当代操作系统向大型和微型两个方向发展：**分布式操作系统**（大型, 集群形态）和**嵌入式操作系统**（微型, 运行在嵌入式智能芯片环境中, 在性能和实时性方面有严格限制）。

> [!note] 分布式操作系统 vs 网络操作系统
> + **分布式操作系统**：在用户看来就像一个普通的单处理机操作系统，用户不感知程序在哪台机器上运行、文件存放在哪里
> + **网络操作系统**：用户知道多台计算机的存在，每台计算机运行自己本地的操作系统，有自己的本地用户

## 操作系统的基本特征

现代操作系统具有四个基本特征：

**并发 (concurrency)**：一定时间内，系统中有两个或两个以上的程序同处于开始运行但尚未结束的状态。并发是操作系统最重要的特征，是其他特征的基础。

**共享 (sharing)**：系统中的资源可供多个并发执行的进程共同使用。共享分为**互斥共享**（一段时间内只允许一个进程访问）和**同时共享**（一段时间内允许多个进程"同时"访问）。

**虚拟 (virtualization)**：通过某种技术将一个物理实体映射为若干个逻辑上的对应物。例如虚拟存储、虚拟设备。

**异步性 (asynchronism)**：进程以不可预知的速度向前推进，即"走走停停"。异步性是并发执行的必然结果。

## 操作系统的主要功能

### 处理机管理

完成处理机资源的分配调度。处理机调度的单位可为进程或线程：
+ **进程控制**：创建、撤销、挂起、改变运行优先级
+ **进程同步与互斥**：协调并发进程之间的推进步骤，以协调资源共享
+ **进程间通信**：进程之间传送数据，以协调进程间的协作
+ **进程调度**：进程的运行切换，以充分利用 CPU 资源和提高系统性能

### 存储管理

提高内存利用率，方便用户使用，提供足够的存储空间，方便进程并发运行：
+ **存储分配与回收**
+ **存储保护**：保证进程间互不干扰、相互保密
+ **地址映射**：进程逻辑地址到内存物理地址的映射
+ **内存扩充**：覆盖、交换和虚拟存储——逻辑上扩充内存

### 设备管理

方便设备使用，提高 CPU 与 I/O 设备利用率：
+ **设备操作**：利用设备驱动程序完成对设备的操作
+ **设备分配与回收**：在多用户间共享 I/O 设备资源
+ **虚拟设备**：设备由多个进程共享，每个进程如同独占该设备
+ **缓冲区管理**：匹配 CPU 和外设的速度

### 文件管理

解决信息资源的存储、共享、保密和保护，操作系统中负责这一功能的部分称为**文件系统**：
+ **文件存储空间管理**：解决如何存放信息，提高空间利用率和读写性能
+ **目录管理**：解决信息检索问题
+ **文件的读写管理和存取控制**：解决信息安全问题

### 用户接口

操作系统向用户提供的与计算机打交道的外部机制：
+ **命令接口**：命令行、菜单式、GUI、命令脚本——供用户组织和控制自己作业的运行
+ **程序接口**：系统调用——供用户程序和系统程序调用操作系统功能

## 操作系统的用户接口

### 命令接口

**操作命令接口**是操作系统为用户操作控制计算机工作和提供服务的手段的集合。实现手段包括：

+ **作业控制语言 (JCL)**：专为批处理作业用户提供，是一种**脱机用户接口**。用户使用 JCL 语句把运行意图写在作业说明书中，连同作业一起提交给系统
+ **操作控制命令**：联机用户接口，用户键入命令后系统立即解释执行

操作控制命令的两种界面形式：

+ **字符显示用户界面**：包括**命令行方式**（以命令为基本单位, 由命令动词和参数构成）和**脚本方式**（预先将一系列命令组织在文件中, 一次建立多次执行）
+ **图形用户接口 (GUI)**：使用 WIMP 界面——窗口(Window)、图标(Icon)、菜单(Menu)和指点设备(Pointing Device)

命令解释程序负责读入控制卡或命令行，并作出相应解释和执行。命令分为**内部命令**（命令解释程序的一部分, 随系统加载驻留内存）和**外部命令**（需要时才调入内存, 实体不包含在命令解释程序中）。

### 程序接口——系统调用

**系统调用 (system call)**是操作系统和用户程序之间的接口，也称为**系统服务 (system service)**。系统调用由**陷入指令 (trap)**实现，该指令使 CPU 主动进入核心态。

系统调用的处理过程：
1. 用户程序执行陷入指令，传递系统调用功能号和参数
2. CPU 转入核心态，保护处理机现场
3. 获取系统调用功能号，在系统调用表中查找对应的内核函数入口地址
4. 执行相应的服务例程
5. 恢复处理机现场，返回用户态

每个系统调用都对应一个事先给定的功能号，陷入指令中包含该功能号。内核维护一张系统调用表，将功能号映射到对应的服务例程入口地址。

> [!note] 系统调用封装
> 陷入指令依赖处理器体系结构(x86_64: `syscall`, ARM64: `svc`, RISC-V: `ecall`)，直接使用将导致程序难以移植。在标准 C 库（如 glibc）中，为每个系统调用提供了封装例程（库函数），用户程序通过调用这些封装例程执行系统调用。

> [!note] POSIX 标准
> **POSIX**（Portable Operating System Interface, 可移植操作系统接口）由 IEEE 开发，ANSI 和 ISO 标准化。POSIX 定义了构造系统所必须提供的一套过程，但并不规定它们是系统调用、库函数调用还是其他形式。POSIX 过程调用与系统调用并非一一对应的关系。POSIX 并不局限于 UNIX，Windows 也支持 POSIX 标准。

> [!example]- Windows API
> Windows 以 DLL 的形式提供了 **Win32 API (application programming interface)**，用户通过调用 API 函数来使用 Windows 操作系统的系统调用。API 调用与系统调用不存在一一对应的关系——例如 `VirtualAlloc` 可能涉及多个系统调用（`NtAllocateMemory` 等），而 `DeleteFile` 通过 `NtOpenFile`、`NtSetInfoFile`、`NtCloseFile` 等多个系统调用实现。利用 API 间接调用系统调用的优点：使用方便，增加应用程序的可移植性。

## 操作系统的设计原则

### 结构性设计原则

+ **分层设计**：将系统划分为清晰层次，每层只依赖下一层
+ **模块化**：把内核划分为相对独立的模块
+ **微内核思想**：内核只保留最核心功能，其余移到用户态
+ **策略与机制分离**：内核提供机制，策略可替换

> [!definition] 机制与策略的分离
> **机制 (mechanism)**是实现某一功能的具体执行机构，**策略 (policy)**是如何使用这些功能。
>
> 例如：定时器是一种确保 CPU 保护的机制，对于特定用户将定时器设置成多长时间是策略；操作系统内核实现基于优先级的调度是机制，决定 I/O 密集型程序比 CPU 密集型程序有更高优先级还是相反是策略。
>
> 策略和机制的分离对于灵活性很重要——策略改变时不应对底层机制造成影响。

### 安全与保护原则

+ **最小特权原则**：每个进程、用户或服务只被授予完成当前任务所必需的最小权限
+ **默认拒绝**：权限系统采取默认拒绝
+ **完全中介 (complete mediation)**：每次资源访问都必须经过权限检查
+ **可信计算基最小化 (TCB minimization)**：能不在内核实现的功能不放进内核

### 性能与工程原则

+ **快路径优化**：优化最常见的路径
+ **延迟绑定**：尽可能推迟决策时间（如动态链接、写时复制）
+ **局部性原则**：时间局部性和空间局部性（虚拟内存设计的基础）

### 可演化与长期生存原则

+ **向后兼容**：决定系统存活时间
+ **简洁性**：简单系统更容易存活
+ **可移植性**：硬件会改变，系统需要通过可移植性来适应
+ **可观测性**：通过日志、跟踪、性能分析等手段监测系统运行状态

## 操作系统体系结构

### 单体系统

整个系统按功能进行设计和模块划分，是一个单一的、庞大的软件系统，各模块可以随意相互调用。

+ **优点**：灵活性好，运行效率高，接口简单直接
+ **缺点**：功能划分和模块接口难以保证正确和合理，模块间依赖关系复杂

> [!example]- 早期 UNIX 的单体结构
> 早期 UNIX 受硬件功能限制，由系统程序和内核两个独立部分组成。内核包括物理硬件之上和系统调用接口之下的所有部分，提供文件系统、CPU 调度、内存管理和其他操作系统功能，大多数功能都结合放在这一层。

### 分层系统

把所有功能模块按功能的调用次序排列成若干层，各层之间的模块只能是**单向依赖或单向调用**的关系——每层只能利用较低层的功能和服务。

> [!note] 分层原则
> + 被调用功能在低层（如文件系统管理 → 设备管理 → 设备驱动程序）
> + 活跃功能在低层，提高运行效率
> + 资源管理的公用模块放在低层
> + 最低层为硬件抽象层
> + 资源分配策略放在高层，便于修改或适应不同环境

第一个按分层模型构造的操作系统是 E.W. Dijkstra 和他的学生开发的 **THE 系统**（1968 年），共分 6 层(0–5)：处理机分配 → 内存和磁鼓管理 → 进程与控制台通信 → I/O 管理 → 用户程序 → 系统操作员。

+ **优点**：功能明确，调用关系清晰，便于修改扩充，高层错误不会影响低层
+ **缺点**：效率较低——每层为系统调用增加了额外开销

### 虚拟机结构

通过**虚拟机监控器 (virtual machine monitor, VMM)**使物理计算机作为共享资源创建虚拟机。每台虚拟机提供与裸机相同的接口，可以运行裸机所能运行的任何类型的操作系统。

+ **1 型超级监控程序 (type 1 hypervisor)**：如 IBM VM/370，VMM 直接运行在裸机上
+ **2 型超级监控程序 (type 2 hypervisor)**：如 VMWare，超级监控程序作为应用程序运行在主机操作系统上，通过二进制翻译技术将客户操作系统的特权指令替换为超级监控程序调用

+ **优点**：完全隔离，安全性好；无直接资源共享；是研究和开发操作系统的良好工具

### 微内核结构

从操作系统中去掉尽可能多的东西，只留一个最小的核心运行在核心态下，其他模块作为**服务进程**运行在用户态下。

+ **微内核**（核心态）：提供最基本的 OS 功能——中断处理、处理机调度、进程间通信
+ **服务进程**（用户态）：其他 OS 服务由运行在用户模式下的进程完成

+ **优点**：体现机制与策略分离；扩充性好（添加新服务不需要修改内核）；可靠性好（服务进程中错误不会使整个系统崩溃）
+ **缺点**：消息传递比直接调用效率要低

### 客户-服务器模式

微内核结构的变体，将进程分为两类：
+ **服务器**：提供某种服务
+ **客户**：使用这些服务

客户进程与服务器进程之间使用消息进行通信。服务器和客户可以运行在相同的机器上，也可以运行在不同的机器上（分布式环境）。

> [!example]- 主要操作系统体系结构实例
> + **Linux**：模块化的单体内核，支持动态加载内核模块
> + **Windows**：混合结构，结合了分层和微内核思想，核心组件包括 HAL、内核、执行体，Win32 子系统等运行在用户态
> + **Mac OS**：基于 Mach 微内核和 BSD UNIX 的混合内核(XNU)
> + **Android**：基于 Linux 内核，上层包含 Android Runtime(ART)、系统库和应用框架
