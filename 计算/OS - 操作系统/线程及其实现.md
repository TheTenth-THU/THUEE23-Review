## 引入线程的动机

进程具有两个基本属性：**资源分配单位**（拥有虚拟地址空间、控制资源）和 **CPU 调度单位**（程序的执行轨迹）。这两个属性构成进程并发执行的基础，但也带来了问题：

**动机 1 — 降低并发开销**：由于进程是资源分配单位，在创建、终止、切换的过程中系统必须付出较大的时空开销。因此系统中并发进程数目不宜过多，进程切换频率也不宜过高，这限制了系统的并发程度。

**动机 2 — 利用多处理器/多核架构**：在多处理器系统和多核处理器中，传统单线程进程在一个时刻只能运行在一个处理器上，不能充分利用硬件并行能力。

> [!note] 多处理器系统架构
> - **SMP**（Symmetric Multiprocessing，对称多处理）：所有 CPU 地位对等，共享同一个操作系统实例，任意 CPU 均可执行内核代码和用户程序。现代通用 CPU 的主流模型
> - **AMP**（Asymmetric Multiprocessing，非对称多处理）：CPU 之间存在主从关系，常见于嵌入式系统
> - **UMA**（Uniform Memory Access）：所有 CPU 共享一块物理内存，访问延迟相同。结构简单但扩展性差
> - **NUMA**（Non-Uniform Memory Access）：每个节点有独立的局部内存，访问本地内存快、远程内存慢。扩展性好，适合大规模服务器

**动机 3 — 应用内在的并发需求**：许多应用（如字处理程序、Web 服务器）天然存在可并发的任务，需要操作系统提供轻量级的并发支持。

**解决思路**：将进程的两个基本属性分开——进程只作为**资源分配单位**，线程则作为 **CPU 调度单位**。

## 线程的基本概念

> [!definition] 线程
> **线程**（Thread）是进程中能够独立执行的实体（控制流），是**处理器调度的基本单位**。每个进程内允许包含多个并发执行的线程，这就是**多线程**（multithreading）。同一进程中的所有线程共享进程获得的主存空间和资源。

### 线程的属性

- **执行状态**：运行、就绪、阻塞（状态转换类似于进程）
- **上下文**：拥有自己的 CPU 现场（通用寄存器、程序计数器、栈指针）
- **执行栈**：每个线程有自己的栈（函数调用、参数传递、局部变量）
- **资源共享**：可存取所在进程的内存和其他资源
- **线程控制**：可以创建、撤销另一个线程

> [!note] 挂起状态是进程级状态
> 挂起状态对线程没有意义——如果进程挂起后被换出主存，则它的所有线程（共享进程的地址空间）也必须全部换出。类似地，进程的终止会导致所有线程的终止。

### 线程控制块

**线程控制块**（TCB，Thread Control Block）是标志线程存在的数据结构，包含线程标识、线程状态、调度参数和 CPU 现场信息（通用寄存器、指令计数器 PC、用户栈指针；内核支持的线程还包含系统栈指针）。TCB 中的内容一般较少，因为资源分配等信息已记录于所属进程的 PCB 中。

### 进程与线程的比较

| 方面 | 进程 | 线程 |
|------|------|------|
| 地址空间与资源 | 相互独立 | 同一进程内共享，其他进程不可见 |
| 通信 | 需要 IPC 机制 | 可直接读写进程数据段（需同步辅助） |
| 调度与切换 | 上下文切换开销大 | 上下文切换比进程快得多 |
| 创建与结束 | 花费时间长 | 创建和结束开销小 |
| 崩溃影响 | 不影响其他进程 | 可能影响整个进程的稳定性 |

## 线程的实现机制

### 用户级线程

**用户级线程**（ULT，User Level Thread）的线程管理工作完全在用户空间内完成，操作系统内核**不知道线程的存在**。

线程管理通过用户空间的**线程库**（运行时系统）实现，它是管理 ULT 的例程包。应用程序需调用线程库中的例程进行编程，与线程库链接后形成可执行程序。

线程库的功能：创建与撤销线程、线程间传递消息和数据、调度线程执行、保护和恢复线程上下文。

每个进程需要有专用的**线程表**（thread table），由运行时系统管理。线程表中的每个表项即 TCB。

> [!note] ULT 与内核的关系
> 操作系统内核不知道线程的活动，仍然以**进程**为单位进行处理器调度。当线程调用阻塞型系统调用时，**整个进程阻塞**。线程状态与进程状态是独立的——可能线程处于运行态而进程处于就绪态。

- **优点**：线程切换不调用内核（性能好）；调度是应用特定的，可针对应用优化；无论内核是否支持线程，只需要线程库即可运行
- **缺点**：大多数系统调用是阻塞的，内核阻塞进程则进程中所有线程被阻塞；同一进程的两个线程不能同时运行于两个处理器上

### 内核级线程

**内核级线程**（KLT，Kernel Level Thread）的线程管理工作由**内核**完成。内核中需要有记录系统中所有线程的**线程表**。不需要线程库，每个进程中不需要线程表。

- **优点**：在多处理器上，内核能同时调度同一进程中的多个线程并行执行；阻塞在**线程一级**完成（一个线程阻塞不影响同进程其他线程）；内核自身也可用多线程技术实现
- **缺点**：线程在用户态运行而调度管理在内核实现，需要频繁进行模式切换，系统开销较大

### 混合实现

混合实现将 ULT 映射到 KLT 上：内核只认识 KLT 并对其进行调度，而某些 KLT 的顶部有若干可以轮流使用的 ULT 集合。线程创建、调度、同步在用户级完成，程序员可以调节 KLT 数量以求最佳效果。

> [!example]- Solaris 的三层线程模型
> Solaris 实现了三个层次：**ULT** → **LWP**（Light Weight Process，轻量级进程）→ **KLT**。LWP 是基于内核线程的高级抽象，由内核独立调度，能在多处理器中并发执行。LWP 作为 ULT 和 KLT 之间的桥梁，实现了多对多映射，灵活高效。但从 Solaris 10 之后因其复杂度过高而被放弃。

> [!example]- POSIX 线程（pthread）
> `pthread` 是广泛采用的 POSIX 多线程编程接口。线程库提供 `pthread_create()`、`pthread_exit()`、`pthread_join()` 等例程。程序员通过链接 pthread 库即可实现多线程，无需关心底层是 ULT、KLT 还是混合实现。

## 引入线程的权衡

**优点**：
- 减小并发执行的时间和空间开销（创建、退出、调度），允许建立更多线程来提高并发度
- 线程间通信无需调用内核（同进程内线程共享内存和文件）
- 适合多处理机系统

**缺点**：
- 程序设计模型引入额外的复杂性
- 一个线程的崩溃可能影响整个进程的稳定性
- 线程过多后调度变复杂，需要消耗较多 CPU 资源
