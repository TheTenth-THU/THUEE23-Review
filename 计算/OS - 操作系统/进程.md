## 进程的引入

### 程序的执行方式

#### 顺序执行

**程序的顺序执行**是指计算机系统中只有一个程序在运行，该程序独占系统中的所有资源，一个程序执行完成后另一个程序才能开始执行。这是单道批处理系统的执行方式。

顺序执行具有三个特征：
+ **顺序性**：一个程序必须等到前一个程序执行完成才能开始执行
+ **封闭性**：程序独占全部资源，计算机的状态只由该程序的控制逻辑决定
+ **可再现性**：初始条件相同则结果相同

#### 并发执行

**程序的并发执行**是指一定时间内，系统中有两个或两个以上的程序同处于开始运行但尚未结束的状态，并且次序不是事先确定的。引入并发执行的目的是提高资源利用率。

并发执行的特征与顺序执行截然相反：
+ **间断性（异步性）**：「走走停停」，一个程序可能走到中途停下来，失去原有的时序关系
+ **失去封闭性**：共享资源，受其他程序的控制逻辑影响
+ **失去可再现性**：失去封闭性导致失去可再现性，外界环境在程序的两次执行期间发生变化，失去原有的可重复特征

程序推进的时序不可再现是允许的，这是并发本身的特征；但**执行的结果不可再现是不可接受的**。因此操作系统必须提供机制来协调并发程序的执行。

> [!example]- 并发执行结果不可再现的示例
> 考虑如下的订票系统，两个终端同时运行 T1、T2 程序：
>
> ```
> T1:                    T2:
> ① read(x)             ① read(x)
> ② if x >= 1 then      ② if x >= 1 then
> ③   x := x - 1        ③   x := x - 1
> ④ write(x)            ④ write(x)
> ```
>
> 假设执行顺序为 T1 ①②③ → T2 ①②③④ → T1 ④，两个终端分别卖出2张票，但写回数据库的余额只减了1。

程序并发执行需求带来如下一些问题：
+ 如何协调各程序的执行顺序？——同步
+ 多个并发执行的程序如何共享系统资源？——互斥
+ 选择哪些，选择多少个程序进入内存执行？——作业调度
+ 内存中并发执行的程序哪个先执行，哪个后执行？——进程调度
+ 内存如何有效分配？——内存管理

### 进程的概念

当允许程序并发执行时，用「程序」概念已不能方便地描述程序的并发执行——并发执行的程序可能是同一个程序在不同数据集合上的执行，也可能是不同的程序在不同数据集合上的执行，它们共享系统资源。为了描述程序在并发执行时对系统资源的共享，需要一个描述程序执行时**动态特征**的概念。

> [!definition] 进程
> **进程 (process)** 是具有**独立功能**的程序在**某个数据集合**上的一次运行活动，是系统进行**资源分配和调度的独立单位**。进程有时也称为**任务 (task)**。

进程具有如下特征：
+ **动态性**：进程是程序的执行，动态产生、动态消亡，在生命周期内在多种状态间转换，具有动态的地址空间
+ **独立性**：进程是资源分配的一个独立单位，各进程的地址空间相互独立
+ **并发性**：任何进程都可以同其他进程一起向前推进
+ **异步性**：每个进程都以其相对独立的、不可预知的速度向前推进
+ **交互性**：进程在执行过程中可能与其他进程产生直接或间接的关系
+ **结构化**：进程的组成 = 程序 + 数据 + PCB

```tx
| 方面 | 进程 | 程序 |
|:-----|:-----|:-----|
| 性质 | **动态的** | **静态的** |\
|      | 是程序的执行，通常不可在计算机之间迁移 | 是有序代码的集合，通常对应着文件，可以复制 |
| 生命 | **暂时的** | **长久的** |\
|          | 是状态变化的过程 | 可长久保存 |
| 组成 | 进程的组成包括程序、数据、PCB ||
| 对应 | 通过多次执行，一个程序可对应多个进程 ||\
|  | 通过调用关系，一个进程可包括多个程序 ||
```

## 进程的描述

### 进程控制块

**进程控制块 (PCB, process control block)** 是操作系统为了管理进程而设置的一个专门的数据结构，用来记录进程的外部特征，描述进程的运动变化过程。

系统利用 PCB 来控制和管理进程，所以 **PCB 是系统感知进程存在的唯一标志**。进程与 PCB 是**一一对应**的。

#### PCB的内容

PCB 通常包含以下信息：

```tx
| 信息类别 | 具体内容 | 说明 |
|:---------|:---------|:-----|
| **进程描述信息** | 进程标识符 (process ID, PID) | 唯一整数 |
| ^^ | 进程名 | 通常基于可执行文件名 |
| ^^ | 用户标识符 (user ID, UID) | |
| ^^ | 进程组 | |
| **进程控制信息** | 当前状态 | |
| ^^ | 优先级 | |
| ^^ | 代码执行入口地址 | |
| ^^ | 程序外存地址 | |
| ^^ | 运行统计信息 | 执行时间、页面调度等 |
| ^^ | 进程间同步和通信信息 | |
| ^^ | 阻塞原因 | |
| **资源占用信息** | 虚拟地址空间现状 | |
| ^^ | 打开文件列表 | |
| **CPU 现场信息** | 寄存器值 | 通用寄存器、PC、PSW、栈指针等 |
| ^^ | 段/页表指针 | 指向赋予该进程的段/页表 |
```

> [!note] 内核对进程信息的访问与修改
> PCB 位于内核空间，用户进程**不能直接访问或修改自己的 PCB**，所有 PCB 的读写操作必须通过**系统调用**进入内核态完成。
>
> 内核在以下典型场景中修改 PCB 信息：
> + **状态转换**：通过阻塞原语、唤醒原语等将进程状态字段从「运行」改为「阻塞」「就绪」等；
> + **现场保存**：进程切换时，将寄存器级上下文写入 PCB 的 CPU 现场信息区；
> + **资源记账**：更新进程的运行时间、页面调度统计等；
> + **调度决策**：调整优先级、记录等待事件或阻塞原因。
>
> 简言之，PCB 是**内核与进程之间的信息枢纽**：内核通过 PCB 感知和控制进程，进程只能通过系统调用间接请求内核修改自己的 PCB。

#### PCB表

系统把所有PCB组织在一起，放在内存的固定区域，构成 **PCB 表**，即进程表 (process table)。PCB 表的大小决定了系统中最多可同时存在的进程个数，称为系统的**并发度**。

PCB表有两种主要的组织方式：
+ **链接结构**：同一状态的进程 PCB 组成链表，形成就绪链表、阻塞链表等，多个状态对应多个不同链表
+ **索引结构**：同一状态的进程归入一个索引表，由索引指向 PCB，多个状态对应多个不同的索引表

> [!example]- Linux 进程描述符
> Linux 的 PCB 是内核数据结构 `task_struct`（定义在 `include/linux/sched.h`），约 1000 字节，包含状态、调度、内存管理、身份标识、家族关系、时间统计、文件系统和 IPC 等信息。所有进程通过双向循环链表组织为进程链表，表头为 `init_task` 的 PCB。PID 类型为 `pid_t`，取值范围 0~32767。
>
> Linux 进程状态：`TASK_RUNNING`（可运行/正在运行）、`TASK_INTERRUPTIBLE`（浅度睡眠, 可被信号唤醒）、`TASK_UNINTERRUPTIBLE`（深度睡眠, 不可被信号唤醒）、`TASK_STOPPED`（暂停, 用于调试）、`TASK_ZOMBIE`（僵死, 已结束运行但 PCB 未释放）。

> [!example]- Windows 进程控制块
> Windows 的进程由执行体进程对象 `EPROCESS` 描述，PCB 也称为内核进程块 `KPROCESS`。创建进程使用 `CreateProcess()`，终止使用 `ExitProcess()`（完整终止）或 `TerminateProcess()`（异常情况下使用, 不完整终止）。

### 进程映像与上下文

进程在内存中的完整**映像 (process image)** 包含：

```tx
| 权限层级 | 内容 | 说明 |
| :- | :- | :- |
| 内核 | PCB | 用户进程不能直接访问、修改自己的PCB |
| 用户 | 栈 (stack) | 用于过程调用和参数传递 |
| ^^   | 堆 (heap)  | 用于动态内存分配 |
| ^^   | 用户数据 | |
| ^^   | 用户程序 | |
```

进程**上下文 (context)** 是对进程执行过程的描述，由进程的用户地址空间内容、硬件寄存器内容及与该进程相关的核心数据结构组成：
+ **用户级上下文**：进程的用户地址空间，包含用户正文段、用户数据段、用户栈；
+ **寄存器级上下文**：程序计数器、处理机状态寄存器、栈指针、通用寄存器的值；
+ **系统级上下文**：PCB 和资源表格、核心栈（进程调用系统核心过程时需要）。

> [!note] 进程映像与进程上下文的关系
> 进程**映像**是进程的**静态存储结构**，回答「进程在内存中有什么」；进程**上下文**是进程的**动态执行状态**，回答「进程执行到哪里了」。
>
> 三部分上下文的来源：
> + **用户级上下文**来源于映像的用户空间（用户程序、数据、栈、堆）；
> + **系统级上下文**来源于映像的内核空间（PCB）加上内核运行时栈；
> + **寄存器级上下文**来源于 CPU 瞬态寄存器值，**不在映像中**。
>
> 进程切换保存和恢复的是**上下文**（尤其寄存器级上下文），映像始终驻留在内存中（或被换出到磁盘），不需要在切换时整体搬移。

> [!note] 操作系统内核与进程的关系
> **OS 内核本身不是一个进程**，内核代码的执行不被调度，而是在特定触发条件下获得 CPU 控制权（中断、陷阱、系统调用），以内核态运行。
>
> OS 与进程地址空间的关系存在三种模式：
> + **传统方法**：OS 不作为进程地址空间的一部分。OS 拥有自己独立的内存区域，进程地址空间与 OS 内核空间完全分离。进程通过陷入内核的方式请求 OS 服务。
> + **早期 UNIX 方法**：OS 整体作为每个进程地址空间的一部分。每个进程的虚拟地址空间中，高端地址区域映射到同一份 OS 内核代码和数据，进程切换到内核态时无需切换地址空间，系统调用开销低。
> + **Windows / Linux 方法**：OS 功能分置——仅 OS **核心**（微内核或内核主体）作为进程地址空间的一部分映射在每个进程的高端地址，而 OS 的**系统服务**（如文件系统服务、网络服务）以独立**服务进程**的形式运行，受调度器管理。
>
> 三种模式的共同前提：OS 内核在**内核态**执行，用户进程在**用户态**执行。用户进程只能通过系统调用陷入内核，由内核代为完成特权操作。

## 进程状态模型

### 三状态进程模型

进程在其生命周期中，会在以下三种**基本状态**之间转换：
+ **运行态 (running)**：进程占有CPU，并在CPU上运行。
+ **就绪态 (ready)**：进程已经具备运行条件，但由于无CPU暂时不能运行，当调度给其CPU时立即可以运行。
+ **阻塞态 (blocked)**：进程因等待某种事件的发生而暂时不能运行，即使CPU空闲该进程也不可运行。

在进程运行过程中，由于进程自身进展情况及外界环境的变化，三种**基本状态**之间可以实现以下的转换：
+ **运行 → 阻塞**：进程等待某事件（资源不可访问、I/O 初始化等待结果、等待 IPC）。
+ **运行 → 就绪**：时间片用完，或被高优先级进程抢占。
+ **就绪 → 运行**：调度程序选择该进程运行。
+ **阻塞 → 就绪**：所等待的事件发生。

![[三状态进程模型.png]]

> [!note] 不可能直接发生的状态转换
> 「**阻塞 → 运行**」和「**就绪 → 阻塞**」是不可能直接发生的状态转换。阻塞进程必须先经过就绪态才能被调度运行，就绪进程必须经过运行态才能因等待事件而阻塞。

### 五状态进程模型

在[[#三状态进程模型]]基础上，增加**新建态**和**终止态**：
+ **新建态 (new)**：OS 已完成为创建进程所必要的工作（分配 PID、创建管理表格），但因资源有限尚未**许可 (admit)** 执行。
+ **终止态 (terminated)**：进程被**撤销 (destroy)** 后进入，不再有执行资格，但表格和其他信息可能暂时保留。

![[五状态进程模型.png]]

### 七状态进程模型

当系统内存不足时，可能将进程暂时换出到外存，于是引入**挂起状态 (suspended)**：
+ **就绪挂起 (suspended ready)**：进程在外存，但只要进入内存即可运行。
+ **阻塞挂起 (suspended blocked)**：进程在外存并等待某事件的出现。

**挂起 (suspend)** 操作将进程从内存转到外存：
+ 阻塞 → 阻塞挂起：释放内存以提交新进程
+ 就绪 → 就绪挂起：挂起低优先级就绪进程
+ 运行 → 就绪挂起：高优先级阻塞挂起进程进入就绪挂起时，抢占当前进程

**激活 (activate)** 操作将进程从外存调回内存：
+ 就绪挂起 → 就绪：没有就绪进程或挂起就绪进程优先级更高
+ 阻塞挂起 → 阻塞：释放足够内存，激活高优先级阻塞挂起进程

## 进程控制

**进程控制**是操作系统使用具有特定功能的程序段来创建与撤销进程以及完成进程状态转换，以达到多进程高效率并发执行和协调、实现资源共享的目的。进程控制由**原语**完成。

> [!definition] 原语
> **原语 (primitive)** 是由若干条指令构成的**原子操作**过程，作为一个整体而不可分割。

> [!note] 不是原语的系统调用
> 并非所有系统调用都是原语。例如进程 A 调用 `read()` 因无数据而阻塞，在 `read()` 里未返回；然后进程 B 调用 `read()`，此时 `read()` 被重入。系统调用不一定一次执行完并返回该进程，有可能在特定点暂停而转入其他进程。

进程控制主要体现在三个方面：创建与撤销、阻塞与唤醒、挂起与激活。每类操作由对应的**原语**完成，原语执行完毕后统一**转处理机调度**，使调度器有机会重新选择进程运行。

### 创建与撤销

#### 创建原语

进程的创建由**创建原语**完成。进程以 PCB 为存在标志，因此创建新进程的主要任务就是为新进程建立 PCB。

导致进程创建的原因：
+ 系统初始化
+ 执行一个创建进程的系统调用（如 `fork()`）
+ 用户请求创建一个新进程
+ 批处理作业的初始化

#### 撤销原语

进程的终止由**撤销原语**完成。进程终止之后，系统应及时回收它占有的所有资源并释放 PCB 结构本身。

导致进程被撤销的原因：
```tx
| 类型 | 方式 | 示例 |
| :- | :- | :- |
| 正常退出 | 自愿的 | `exit()` 调用 |
| 出错退出 | 自愿的 | 程序自行检测到不可恢复错误后退出 |
| 严重错误 | 非自愿的 | 非法指令、访问越界等，由 OS 强制终止 |
| 被杀死 | 非自愿的 | 被其他进程通过 `kill` 等信号杀死 |
```

### 阻塞与唤醒

#### 阻塞原语

**阻塞原语**实现进程从**运行状态到阻塞状态**的转换。当进程在执行中需要等待某一事件发生（如等待 I/O 完成、等待资源可用）时，调用阻塞原语将自己阻塞。

> [!note] 阻塞是主动行为
> 阻塞是由**本进程自己调用**的原语——只有运行中的进程才知道何时需要等待某事件。没有其他进程可以「强行阻塞」一个正在运行的进程。

#### 唤醒原语

**唤醒原语**实现进程从**阻塞状态到就绪状态**的转换。当阻塞进程所等待的事件发生时，由发现者（通常是中断处理程序或其他进程）调用唤醒原语将阻塞进程唤醒。

![[阻塞与唤醒.png]]

> [!note] 阻塞与唤醒的配对
> 阻塞和唤醒是一对**成对使用**的原语：阻塞由进程自己调用，唤醒由事件发现者调用。两者协同实现进程间的**同步**。

### 挂起与激活

当系统内存不足时，可能需要将部分进程暂时换出到外存，以释放内存给更需要的进程。挂起与激活操作成对出现，负责内存和外存之间的进程迁移。

#### 挂起原语

**挂起原语**将进程从内存调出到外存，根据进程当前状态分别处理：就绪进程挂起到就绪挂起队列，阻塞进程挂起到阻塞挂起队列。

#### 激活原语

**激活原语**将进程从外存调入内存，是挂起的逆操作。根据进程挂起前的状态，分别恢复到就绪队列或阻塞队列。

![[挂起与激活.png]]
