# _1_  操作系统引论

## 计算机系统与操作系统

**计算机系统**由硬件和软件两部分组成。计算机硬件是根据电、磁、光、机械等原理构成的各种物理设备的有机组合，是系统赖以工作的实体；软件是计算机系统中程序和有关文件的集合。

现代计算机硬件十分复杂，程序员直接面对硬件编程是不现实的。因此，必须在硬件裸机上加载一层软件来管理整个系统，同时给用户提供一个更容易理解和编程的接口——这层软件就是**操作系统 (operating system, OS)**。

> [!definition] 操作系统的定义
> 操作系统是计算机系统中的一个**系统软件**，它是一系列程序模块的集合：这些模块能有效地组织和管理计算机的软硬件资源，合理地组织计算机的工作流程，控制程序的执行并向用户提供各种服务功能，使得用户能够方便地使用计算机，使整个计算机系统能高效地运行。
>
> 关键要点：系统软件，程序模块的集合，资源管理和用户接口功能。

操作系统的三大作用可以从不同视角理解：

### 视角一：作为扩展机器的操作系统

操作系统为用户提供一台等价的**扩展机器**或**虚拟机 (virtual machine)**，它比底层硬件更容易编程和使用。在裸机基础上，OS 添加了：

+ 设备管理
+ 文件管理
+ 存储管理（内存）
+ 处理机管理(CPU)
+ 作业管理（合理组织工作流程）

### 视角二：作为资源管理者的操作系统

操作系统管理的资源包括 CPU、存储器、外部设备、信息（数据和软件）。管理内容包括：资源的当前状态（数量和使用情况）、资源的分配和回收、访问操作、相应管理策略（包括用户权限）。

从资源管理的观点看，操作系统的首要任务是**跟踪资源使用状况、满足资源请求、提高资源利用率**，以及**协调各程序和用户对资源的使用冲突**。

### 视角三：作为用户接口的操作系统

操作系统向用户提供两种接口：
+ **命令接口**：命令行、菜单式、命令脚本、图形用户接口(GUI)
+ **程序接口**：系统调用(system call)，形式上类似于过程调用，供应用程序使用

计算机系统中，**用户态 (user mode)**与**核心态 (kernel mode)**是两个关键概念：操作系统专指在核心态下运行的软件，而应用软件和操作系统以外的系统软件运行在用户态下。

> [!note] "有效、合理、方便"的含义
> + **有效**：系统效率（如 CPU 利用率）、资源利用率（内存、外设是否忙碌）
> + **合理**：公平性，不公平则可能产生"死锁"或"饥饿"
> + **方便**：提供良好、一致的用户接口，弥补硬件系统的类型和数量差别

## 操作系统的发展历史

推动操作系统发展的主要动力包括：提高资源利用率和系统性能、方便用户、以及计算机硬件技术的进步。操作系统历史可按硬件技术发展划分为四个时代。

### 第 1 代：电子管时代(1946–1955)

以 ENIAC 为代表，运算速度约 1000 次/秒，数万个真空管，占地 100 平方米。没有程序设计语言，更谈不上操作系统——所谓「编程」就是在插线板上插接导线。50 年代早期出现穿孔卡片，但仍为**手工操作**方式，效率极低。

### 第 2 代：晶体管时代(1955–1965)

晶体管发明使计算机可靠性提高，可成批生产。FORTRAN、ALGOL、COBOL 等程序设计语言相继诞生，程序员、操作员和维护人员之间有了明确分工。

**批处理操作系统**——现代操作系统的雏形——应运而生。其思想是：用一台相对便宜的计算机（如 IBM 1401）将作业读到磁带上，再用昂贵的计算机（如 IBM 7094）完成真正的计算，以改进主存和 I/O 设备之间的吞吐量。

> [!note] 批处理的主要特征
> + 用户**脱机**使用计算机：作业提交后直到获得结果之前，用户无法与作业交互
> + 作业**成批处理**

典型系统：FMS(FORTRAN Monitor System)、IBMSYS（IBM 7094 机配备的操作系统），由监控程序、特权指令、存储保护和简单的批处理构成。

### 第 3 代：集成电路时代(1965–1980)

这一时代操作系统发生了质的飞跃，出现了多项关键技术。

**多道程序设计 (multiprogramming)**：将内存分为几个部分，每部分放不同的作业。当某个作业因等待 I/O 而暂停时，另一个作业可以使用 CPU。

> [!note] 多道程序设计的特点（单 CPU）
> + **多道**：内存中同时存放多道相互独立的程序
> + **宏观上并行**：多道程序都在运行，在一定时间间隔内都取得进展
> + **微观上串行**：任意时刻，只有一道程序占有 CPU

多道批处理系统引入的关键技术：作业调度（上下文切换）、资源共享（互斥和同步机制）、内存使用（覆盖、交换、虚拟存储）、内存保护、文件非顺序存放和随机存取。

**分时系统 (timesharing system)**：实际上是多道程序的一个变种。将 CPU 时间分割成**时间片 (time slice)**，每个用户依次轮流使用时间片。每个用户有一个联机终端，能够与作业进行交互。

分时系统的特征：**多路性**（多个用户同时工作）、**独立性**（各用户互不干扰）、**交互性**（系统及时响应用户操作, 显著提高调试和修改程序的效率）。

**UNIX 的崛起**：1969 年，贝尔实验室的 Ken Thompson 在一台闲置的 PDP-7 上用汇编语言编写了操作系统内核、文件系统和 shell，取名 UNICS。后与 Dennis Ritchie 合作，用 C 语言重写了 UNIX，使其具有良好的可移植性。UNIX 成为现代操作系统的代表。

### 第 4 代：大规模集成电路时代(1980–)

个人计算机时代到来，各种类型的个人计算机和操作系统层出不穷。典型系统包括：

+ 单用户单任务：MS-DOS
+ 单用户多任务：Windows Desktop、Mac OS X
+ 多用户多任务：UNIX、Linux、Windows Server

当代操作系统向大型和微型两个方向发展：**分布式操作系统**（大型, 集群形态）和**嵌入式操作系统**（微型, 运行在嵌入式智能芯片环境中, 在性能和实时性方面有严格限制）。

> [!note] 分布式操作系统 vs 网络操作系统
> + **分布式操作系统**：在用户看来就像一个普通的单处理机操作系统，用户不感知程序在哪台机器上运行、文件存放在哪里
> + **网络操作系统**：用户知道多台计算机的存在，每台计算机运行自己本地的操作系统，有自己的本地用户

## 操作系统的基本特征

现代操作系统具有四个基本特征：

**并发 (concurrency)**：一定时间内，系统中有两个或两个以上的程序同处于开始运行但尚未结束的状态。并发是操作系统最重要的特征，是其他特征的基础。

**共享 (sharing)**：系统中的资源可供多个并发执行的进程共同使用。共享分为**互斥共享**（一段时间内只允许一个进程访问）和**同时共享**（一段时间内允许多个进程"同时"访问）。

**虚拟 (virtualization)**：通过某种技术将一个物理实体映射为若干个逻辑上的对应物。例如虚拟存储、虚拟设备。

**异步性 (asynchronism)**：进程以不可预知的速度向前推进，即"走走停停"。异步性是并发执行的必然结果。

## 操作系统的主要功能

### 处理机管理

完成处理机资源的分配调度。处理机调度的单位可为进程或线程：
+ **进程控制**：创建、撤销、挂起、改变运行优先级
+ **进程同步与互斥**：协调并发进程之间的推进步骤，以协调资源共享
+ **进程间通信**：进程之间传送数据，以协调进程间的协作
+ **进程调度**：进程的运行切换，以充分利用 CPU 资源和提高系统性能

### 存储管理

提高内存利用率，方便用户使用，提供足够的存储空间，方便进程并发运行：
+ **存储分配与回收**
+ **存储保护**：保证进程间互不干扰、相互保密
+ **地址映射**：进程逻辑地址到内存物理地址的映射
+ **内存扩充**：覆盖、交换和虚拟存储——逻辑上扩充内存

### 设备管理

方便设备使用，提高 CPU 与 I/O 设备利用率：
+ **设备操作**：利用设备驱动程序完成对设备的操作
+ **设备分配与回收**：在多用户间共享 I/O 设备资源
+ **虚拟设备**：设备由多个进程共享，每个进程如同独占该设备
+ **缓冲区管理**：匹配 CPU 和外设的速度

### 文件管理

解决信息资源的存储、共享、保密和保护，操作系统中负责这一功能的部分称为**文件系统**：
+ **文件存储空间管理**：解决如何存放信息，提高空间利用率和读写性能
+ **目录管理**：解决信息检索问题
+ **文件的读写管理和存取控制**：解决信息安全问题

### 用户接口

操作系统向用户提供的与计算机打交道的外部机制：
+ **命令接口**：命令行、菜单式、GUI、命令脚本——供用户组织和控制自己作业的运行
+ **程序接口**：系统调用——供用户程序和系统程序调用操作系统功能

## 操作系统的用户接口

### 命令接口

**操作命令接口**是操作系统为用户操作控制计算机工作和提供服务的手段的集合。实现手段包括：

+ **作业控制语言 (JCL)**：专为批处理作业用户提供，是一种**脱机用户接口**。用户使用 JCL 语句把运行意图写在作业说明书中，连同作业一起提交给系统
+ **操作控制命令**：联机用户接口，用户键入命令后系统立即解释执行

操作控制命令的两种界面形式：

+ **字符显示用户界面**：包括**命令行方式**（以命令为基本单位, 由命令动词和参数构成）和**脚本方式**（预先将一系列命令组织在文件中, 一次建立多次执行）
+ **图形用户接口 (GUI)**：使用 WIMP 界面——窗口(Window)、图标(Icon)、菜单(Menu)和指点设备(Pointing Device)

命令解释程序负责读入控制卡或命令行，并作出相应解释和执行。命令分为**内部命令**（命令解释程序的一部分, 随系统加载驻留内存）和**外部命令**（需要时才调入内存, 实体不包含在命令解释程序中）。

### 程序接口——系统调用

**系统调用 (system call)**是操作系统和用户程序之间的接口，也称为**系统服务 (system service)**。系统调用由**陷入指令 (trap)**实现，该指令使 CPU 主动进入核心态。

系统调用的处理过程：
1. 用户程序执行陷入指令，传递系统调用功能号和参数
2. CPU 转入核心态，保护处理机现场
3. 获取系统调用功能号，在系统调用表中查找对应的内核函数入口地址
4. 执行相应的服务例程
5. 恢复处理机现场，返回用户态

每个系统调用都对应一个事先给定的功能号，陷入指令中包含该功能号。内核维护一张系统调用表，将功能号映射到对应的服务例程入口地址。

> [!note] 系统调用封装
> 陷入指令依赖处理器体系结构(x86_64: `syscall`, ARM64: `svc`, RISC-V: `ecall`)，直接使用将导致程序难以移植。在标准 C 库（如 glibc）中，为每个系统调用提供了封装例程（库函数），用户程序通过调用这些封装例程执行系统调用。

> [!note] POSIX 标准
> **POSIX**（Portable Operating System Interface, 可移植操作系统接口）由 IEEE 开发，ANSI 和 ISO 标准化。POSIX 定义了构造系统所必须提供的一套过程，但并不规定它们是系统调用、库函数调用还是其他形式。POSIX 过程调用与系统调用并非一一对应的关系。POSIX 并不局限于 UNIX，Windows 也支持 POSIX 标准。

> [!example]- Windows API
> Windows 以 DLL 的形式提供了 **Win32 API (application programming interface)**，用户通过调用 API 函数来使用 Windows 操作系统的系统调用。API 调用与系统调用不存在一一对应的关系——例如 `VirtualAlloc` 可能涉及多个系统调用（`NtAllocateMemory` 等），而 `DeleteFile` 通过 `NtOpenFile`、`NtSetInfoFile`、`NtCloseFile` 等多个系统调用实现。利用 API 间接调用系统调用的优点：使用方便，增加应用程序的可移植性。

## 操作系统的设计原则

### 结构性设计原则

+ **分层设计**：将系统划分为清晰层次，每层只依赖下一层
+ **模块化**：把内核划分为相对独立的模块
+ **微内核思想**：内核只保留最核心功能，其余移到用户态
+ **策略与机制分离**：内核提供机制，策略可替换

> [!definition] 机制与策略的分离
> **机制 (mechanism)**是实现某一功能的具体执行机构，**策略 (policy)**是如何使用这些功能。
>
> 例如：定时器是一种确保 CPU 保护的机制，对于特定用户将定时器设置成多长时间是策略；操作系统内核实现基于优先级的调度是机制，决定 I/O 密集型程序比 CPU 密集型程序有更高优先级还是相反是策略。
>
> 策略和机制的分离对于灵活性很重要——策略改变时不应对底层机制造成影响。

### 安全与保护原则

+ **最小特权原则**：每个进程、用户或服务只被授予完成当前任务所必需的最小权限
+ **默认拒绝**：权限系统采取默认拒绝
+ **完全中介 (complete mediation)**：每次资源访问都必须经过权限检查
+ **可信计算基最小化 (TCB minimization)**：能不在内核实现的功能不放进内核

### 性能与工程原则

+ **快路径优化**：优化最常见的路径
+ **延迟绑定**：尽可能推迟决策时间（如动态链接、写时复制）
+ **局部性原则**：时间局部性和空间局部性（虚拟内存设计的基础）

### 可演化与长期生存原则

+ **向后兼容**：决定系统存活时间
+ **简洁性**：简单系统更容易存活
+ **可移植性**：硬件会改变，系统需要通过可移植性来适应
+ **可观测性**：通过日志、跟踪、性能分析等手段监测系统运行状态

## 操作系统体系结构

### 单体系统

整个系统按功能进行设计和模块划分，是一个单一的、庞大的软件系统，各模块可以随意相互调用。

+ **优点**：灵活性好，运行效率高，接口简单直接
+ **缺点**：功能划分和模块接口难以保证正确和合理，模块间依赖关系复杂

> [!example]- 早期 UNIX 的单体结构
> 早期 UNIX 受硬件功能限制，由系统程序和内核两个独立部分组成。内核包括物理硬件之上和系统调用接口之下的所有部分，提供文件系统、CPU 调度、内存管理和其他操作系统功能，大多数功能都结合放在这一层。

### 分层系统

把所有功能模块按功能的调用次序排列成若干层，各层之间的模块只能是**单向依赖或单向调用**的关系——每层只能利用较低层的功能和服务。

> [!note] 分层原则
> + 被调用功能在低层（如文件系统管理 → 设备管理 → 设备驱动程序）
> + 活跃功能在低层，提高运行效率
> + 资源管理的公用模块放在低层
> + 最低层为硬件抽象层
> + 资源分配策略放在高层，便于修改或适应不同环境

第一个按分层模型构造的操作系统是 E.W. Dijkstra 和他的学生开发的 **THE 系统**（1968 年），共分 6 层(0–5)：处理机分配 → 内存和磁鼓管理 → 进程与控制台通信 → I/O 管理 → 用户程序 → 系统操作员。

+ **优点**：功能明确，调用关系清晰，便于修改扩充，高层错误不会影响低层
+ **缺点**：效率较低——每层为系统调用增加了额外开销

### 虚拟机结构

通过**虚拟机监控器 (virtual machine monitor, VMM)**使物理计算机作为共享资源创建虚拟机。每台虚拟机提供与裸机相同的接口，可以运行裸机所能运行的任何类型的操作系统。

+ **1 型超级监控程序 (type 1 hypervisor)**：如 IBM VM/370，VMM 直接运行在裸机上
+ **2 型超级监控程序 (type 2 hypervisor)**：如 VMWare，超级监控程序作为应用程序运行在主机操作系统上，通过二进制翻译技术将客户操作系统的特权指令替换为超级监控程序调用

+ **优点**：完全隔离，安全性好；无直接资源共享；是研究和开发操作系统的良好工具

### 微内核结构

从操作系统中去掉尽可能多的东西，只留一个最小的核心运行在核心态下，其他模块作为**服务进程**运行在用户态下。

+ **微内核**（核心态）：提供最基本的 OS 功能——中断处理、处理机调度、进程间通信
+ **服务进程**（用户态）：其他 OS 服务由运行在用户模式下的进程完成

+ **优点**：体现机制与策略分离；扩充性好（添加新服务不需要修改内核）；可靠性好（服务进程中错误不会使整个系统崩溃）
+ **缺点**：消息传递比直接调用效率要低

### 客户-服务器模式

微内核结构的变体，将进程分为两类：
+ **服务器**：提供某种服务
+ **客户**：使用这些服务

客户进程与服务器进程之间使用消息进行通信。服务器和客户可以运行在相同的机器上，也可以运行在不同的机器上（分布式环境）。

> [!example]- 主要操作系统体系结构实例
> + **Linux**：模块化的单体内核，支持动态加载内核模块
> + **Windows**：混合结构，结合了分层和微内核思想，核心组件包括 HAL、内核、执行体，Win32 子系统等运行在用户态
> + **Mac OS**：基于 Mach 微内核和 BSD UNIX 的混合内核(XNU)
> + **Android**：基于 Linux 内核，上层包含 Android Runtime(ART)、系统库和应用框架
> + **HarmonyOS**：基于微内核的全场景分布式操作系统


# _2_  处理机管理

## 处理机管理概述

**处理机管理**是操作系统最核心的功能之一，负责协调多个程序对 CPU 的共享使用。多道程序环境下，系统中同时存在多个并发执行的进程，操作系统必须解决四个基本问题：

1. **如何描述进程**——将运行中的程序抽象为进程，通过 PCB 跟踪其状态和资源
2. **如何更轻量地并发**——引入线程，将资源分配与调度执行分离
3. **进程间如何协作**——提供互斥与同步机制，确保共享资源的有序访问
4. **如何分配 CPU 与避免死锁**——选择合适的调度算法，预防和处理死锁

以下依次概述这四个子主题的核心概念与它们之间的关联。

### 进程的描述与控制

**进程**是处理机管理的基本单位。操作系统中，程序的并发执行带来了间断性、失去封闭性和失去可再现性等问题——这正是引入进程概念的动机。

进程的抽象依赖于 **[[#进程#进程控制块|进程控制块(PCB)]]**，它是系统感知进程存在的唯一标志。PCB 记录了进程标识、状态、优先级、CPU 现场和资源占用信息。系统将所有 PCB 组织为 PCB 表，其大小决定了系统的**并发度**。

进程在其生命周期中经历多种状态。[[#进程#三种基本状态|三态模型]]（运行—就绪—阻塞）是基础，进一步扩展得到[[#进程#五状态进程模型|五态模型]]（增加新建态和终止态）和[[#进程#七状态进程模型|七态模型]]（增加挂起态）。状态之间的转换由**进程控制原语**（创建、撤销、阻塞、唤醒、挂起、激活）完成——原语是原子操作，不可被中断。

> [!definition] 进程
> **进程 (process)**是具有独立功能的程序在某个数据集合上的一次运行活动，是系统进行**资源分配和调度**的独立单位。进程的组成 = 程序 + 数据 + PCB。

### 线程及其实现

进程同时承担**资源分配单位**和 **CPU 调度单位**两个角色，导致创建、切换进程的开销较大，限制了系统并发度。**[[#线程#引入线程的动机|线程的引入]]**正是为了将这两个角色分离：进程仅作为资源分配单位，线程则成为 CPU 调度的基本单位。

> [!definition] 线程
> **线程 (thread)**是进程中能够独立执行的实体，是**处理器调度的基本单位**。同一进程中的所有线程共享进程的地址空间和资源，但各自拥有独立的 CPU 现场和执行栈。

线程的实现方式决定了系统的并发性能。[[#线程#用户级线程|用户级线程(ULT)]]的线程管理完全在用户空间完成，切换开销极小，但一个线程的阻塞系统调用会导致整个进程阻塞。[[#线程#内核级线程|内核级线程(KLT)]]由内核管理，阻塞在**线程一级**完成，且能在多处理器上并行执行，但模式切换开销较大。[[#线程#混合实现|混合实现]]将 ULT 映射到 KLT 上，兼顾灵活性和性能。

| 方面 | 进程 | 线程 |
|------|------|------|
| 地址空间 | 相互独立 | 同进程内共享 |
| 通信方式 | 需要 IPC 机制 | 可直接读写进程数据段 |
| 切换开销 | 大 | 小 |
| 崩溃影响 | 不影响其他进程 | 可能影响整个进程 |

### 进程间通信

并发进程之间需要两种基本的协调机制。**[[#进程间通信#互斥与同步|互斥]]**确保临界资源在某一时刻只能被一个进程访问——这源于无意识的资源共享。**[[#进程间通信#互斥与同步|同步]]**则确保协作进程按正确的先后顺序推进——这源于有意识的进程合作。

解决互斥问题的方法经历了从软件到硬件、从分散到集中的演进：

+ **[[#进程间通信#软件互斥算法|软件方法]]**（锁变量、Peterson 算法）存在忙等待问题
+ **[[#进程间通信#硬件互斥方法|硬件指令]] (Test-and-Set, SWAP)**利用原子操作简化实现，但仍有忙等待
+ **[[#进程间通信#信号量|信号量]] (semaphore)**由 OS 作为管理者，通过 P/V 原语实现无忙等待的互斥与同步
+ **[[#进程间通信#管程|管程]] (monitor)**将共享变量及操作封装为模块，由编译器保证互斥

> [!theorem] 临界区访问的四原则
> 1. **空闲让进** 2. **忙则等待** 3. **有限等待** 4. **让权等待**

[[#进程间通信#经典 IPC 问题|四个经典 IPC 问题]]（生产者-消费者、读者-写者、哲学家进餐、睡眠理发师）是检验同步机制正确性的标准测试用例。

当进程需要交换大量数据时，可采用[[#进程间通信#高级进程间通信|高级通信机制]]：**共享内存**效率最高但需自行处理互斥；**消息传递**通过邮箱/队列实现解耦通信；**管道**以字节流方式在父子进程或任意进程间传输数据。

### 处理机调度与死锁

**[[#处理机调度#处理机调度|处理机调度]]**决定哪个就绪进程获得 CPU。调度分三个层次：长程调度（作业调度, 选择作业进入内存）、中程调度（进程换入换出）、短程调度（进程调度, 选择就绪进程运行, 毫秒级）。

调度算法按应用场景划分：

+ **批处理系统**：[[#处理机调度#先来先服务(FCFS)|FCFS]]、[[#处理机调度#最短作业优先(SJF)|SJF]]、[[#处理机调度#最高响应比优先(HRRN)|HRRN]]——关注吞吐量和周转时间
+ **交互式系统**：[[#处理机调度#时间片轮转(RR)|时间片轮转(RR)]]、[[#处理机调度#优先级调度|优先级调度]]、[[#处理机调度#多级反馈队列(MLFQ)|多级反馈队列(MLFQ)]]——关注响应时间
+ **实时系统**：[[#处理机调度#速率单调调度(RMS)|RMS]]、[[#处理机调度#最早截止时限优先(EDF)|EDF]]——关注截止时间保证

**[[#处理机调度#死锁|死锁]]**是多个进程因竞争资源而无限期等待的僵局。死锁的发生需要四个必要条件同时成立——互斥、非抢占、请求和保持、环路等待。

> [!theorem] 死锁发生的四个必要条件
> 1. **互斥** 2. **非抢占** 3. **请求和保持** 4. **环路等待**

死锁的处理策略由弱到强：[[#处理机调度#死锁预防|死锁预防]]（破坏四个必要条件之一, 如预先静态分配法、有序资源使用法）→ [[#处理机调度#死锁避免|死锁避免]]（[[#处理机调度#银行家算法 (Dijkstra, 1965)|银行家算法]], 动态判断分配是否使系统进入不安全状态）→ [[#处理机调度#死锁检测与解除|死锁检测与解除]]（通过资源分配图化简检测死锁, 剥夺资源或撤销进程解除）。

> [!note] 鸵鸟算法
> 大多数实际系统(UNIX, Linux, Windows)采用"鸵鸟算法"——因死锁发生概率很小而处理代价很大，选择忽略。但这不意味着死锁理论不重要：在数据库、分布式系统等对一致性要求严格的场景中，死锁处理是核心问题。

### 小结

处理机管理的四个子主题构成一条完整的逻辑链：**进程**定义了并发执行的基本单位 → **线程**将资源分配与调度执行分离以提升并发效率 → **进程间通信**提供了并发体之间的协调机制 → **处理机调度与死锁**解决 CPU 的分配策略和资源竞争导致的僵局。这四个环节环环相扣，共同构成了操作系统对 CPU 这一核心资源的管理体系。


# _2.1_  进程

## 进程的引入

### 程序的执行方式

#### 顺序执行

**程序的顺序执行**是指计算机系统中只有一个程序在运行，该程序独占系统中的所有资源，一个程序执行完成后另一个程序才能开始执行。这是单道批处理系统的执行方式。

顺序执行具有三个特征：
+ **顺序性**：一个程序必须等到前一个程序执行完成才能开始执行
+ **封闭性**：程序独占全部资源，计算机的状态只由该程序的控制逻辑决定
+ **可再现性**：初始条件相同则结果相同

#### 并发执行

**程序的并发执行**是指一定时间内，系统中有两个或两个以上的程序同处于开始运行但尚未结束的状态，并且次序不是事先确定的。引入并发执行的目的是提高资源利用率。

并发执行的特征与顺序执行截然相反：
+ **间断性（异步性）**：「走走停停」，一个程序可能走到中途停下来，失去原有的时序关系
+ **失去封闭性**：共享资源，受其他程序的控制逻辑影响
+ **失去可再现性**：失去封闭性导致失去可再现性，外界环境在程序的两次执行期间发生变化，失去原有的可重复特征

程序推进的时序不可再现是允许的，这是并发本身的特征；但**执行的结果不可再现是不可接受的**。因此操作系统必须提供机制来协调并发程序的执行。

> [!example]- 并发执行结果不可再现的示例
> 考虑如下的订票系统，两个终端同时运行 T1、T2 程序：
>
> ```
> T1:                    T2:
> ① read(x)             ① read(x)
> ② if x >= 1 then      ② if x >= 1 then
> ③   x := x - 1        ③   x := x - 1
> ④ write(x)            ④ write(x)
> ```
>
> 假设执行顺序为 T1 ①②③ → T2 ①②③④ → T1 ④，两个终端分别卖出2张票，但写回数据库的余额只减了1。

程序并发执行需求带来如下一些问题：
+ 如何协调各程序的执行顺序？——同步
+ 多个并发执行的程序如何共享系统资源？——互斥
+ 选择哪些，选择多少个程序进入内存执行？——作业调度
+ 内存中并发执行的程序哪个先执行，哪个后执行？——进程调度
+ 内存如何有效分配？——内存管理

### 进程的概念

当允许程序并发执行时，用「程序」概念已不能方便地描述程序的并发执行——并发执行的程序可能是同一个程序在不同数据集合上的执行，也可能是不同的程序在不同数据集合上的执行，它们共享系统资源。为了描述程序在并发执行时对系统资源的共享，需要一个描述程序执行时**动态特征**的概念。

> [!definition] 进程
> **进程 (process)** 是具有**独立功能**的程序在**某个数据集合**上的一次运行活动，是系统进行**资源分配和调度的独立单位**。进程有时也称为**任务 (task)**。

进程具有如下特征：
+ **动态性**：进程是程序的执行，动态产生、动态消亡，在生命周期内在多种状态间转换，具有动态的地址空间
+ **独立性**：进程是资源分配的一个独立单位，各进程的地址空间相互独立
+ **并发性**：任何进程都可以同其他进程一起向前推进
+ **异步性**：每个进程都以其相对独立的、不可预知的速度向前推进
+ **交互性**：进程在执行过程中可能与其他进程产生直接或间接的关系
+ **结构化**：进程的组成 = 程序 + 数据 + PCB

```tx
| 方面 | 进程 | 程序 |
|:-----|:-----|:-----|
| 性质 | **动态的** | **静态的** |\
|      | 是程序的执行，通常不可在计算机之间迁移 | 是有序代码的集合，通常对应着文件，可以复制 |
| 生命 | **暂时的** | **长久的** |\
|          | 是状态变化的过程 | 可长久保存 |
| 组成 | 进程的组成包括程序、数据、PCB ||
| 对应 | 通过多次执行，一个程序可对应多个进程 ||\
|  | 通过调用关系，一个进程可包括多个程序 ||
```

## 进程的描述

### 进程控制块

**进程控制块 (PCB, process control block)** 是操作系统为了管理进程而设置的一个专门的数据结构，用来记录进程的外部特征，描述进程的运动变化过程。

系统利用 PCB 来控制和管理进程，所以 **PCB 是系统感知进程存在的唯一标志**。进程与 PCB 是**一一对应**的。

#### PCB的内容

PCB 通常包含以下信息：

```tx
| 信息类别 | 具体内容 | 说明 |
|:---------|:---------|:-----|
| **进程描述信息** | 进程标识符 (process ID, PID) | 唯一整数 |
| ^^ | 进程名 | 通常基于可执行文件名 |
| ^^ | 用户标识符 (user ID, UID) | |
| ^^ | 进程组 | |
| **进程控制信息** | 当前状态 | |
| ^^ | 优先级 | |
| ^^ | 代码执行入口地址 | |
| ^^ | 程序外存地址 | |
| ^^ | 运行统计信息 | 执行时间、页面调度等 |
| ^^ | 进程间同步和通信信息 | |
| ^^ | 阻塞原因 | |
| **资源占用信息** | 虚拟地址空间现状 | |
| ^^ | 打开文件列表 | |
| **CPU 现场信息** | 寄存器值 | 通用寄存器、PC、PSW、栈指针等 |
| ^^ | 段/页表指针 | 指向赋予该进程的段/页表 |
```

> [!note] 内核对进程信息的访问与修改
> PCB 位于内核空间，用户进程**不能直接访问或修改自己的 PCB**，所有 PCB 的读写操作必须通过**系统调用**进入内核态完成。
>
> 内核在以下典型场景中修改 PCB 信息：
> + **状态转换**：通过阻塞原语、唤醒原语等将进程状态字段从「运行」改为「阻塞」「就绪」等；
> + **现场保存**：进程切换时，将寄存器级上下文写入 PCB 的 CPU 现场信息区；
> + **资源记账**：更新进程的运行时间、页面调度统计等；
> + **调度决策**：调整优先级、记录等待事件或阻塞原因。
>
> 简言之，PCB 是**内核与进程之间的信息枢纽**：内核通过 PCB 感知和控制进程，进程只能通过系统调用间接请求内核修改自己的 PCB。

#### PCB表

系统把所有PCB组织在一起，放在内存的固定区域，构成 **PCB 表**，即进程表 (process table)。PCB 表的大小决定了系统中最多可同时存在的进程个数，称为系统的**并发度**。

PCB表有两种主要的组织方式：
+ **链接结构**：同一状态的进程 PCB 组成链表，形成就绪链表、阻塞链表等，多个状态对应多个不同链表
+ **索引结构**：同一状态的进程归入一个索引表，由索引指向 PCB，多个状态对应多个不同的索引表

> [!example]- Linux 进程描述符
> Linux 的 PCB 是内核数据结构 `task_struct`（定义在 `include/linux/sched.h`），约 1000 字节，包含状态、调度、内存管理、身份标识、家族关系、时间统计、文件系统和 IPC 等信息。所有进程通过双向循环链表组织为进程链表，表头为 `init_task` 的 PCB。PID 类型为 `pid_t`，取值范围 0~32767。
>
> Linux 进程状态：`TASK_RUNNING`（可运行/正在运行）、`TASK_INTERRUPTIBLE`（浅度睡眠, 可被信号唤醒）、`TASK_UNINTERRUPTIBLE`（深度睡眠, 不可被信号唤醒）、`TASK_STOPPED`（暂停, 用于调试）、`TASK_ZOMBIE`（僵死, 已结束运行但 PCB 未释放）。

> [!example]- Windows 进程控制块
> Windows 的进程由执行体进程对象 `EPROCESS` 描述，PCB 也称为内核进程块 `KPROCESS`。创建进程使用 `CreateProcess()`，终止使用 `ExitProcess()`（完整终止）或 `TerminateProcess()`（异常情况下使用, 不完整终止）。

### 进程映像与上下文

进程在内存中的完整**映像 (process image)** 包含：

```tx
| 权限层级 | 内容 | 说明 |
| :- | :- | :- |
| 内核 | PCB | 用户进程不能直接访问、修改自己的PCB |
| 用户 | 栈 (stack) | 用于过程调用和参数传递 |
| ^^   | 堆 (heap)  | 用于动态内存分配 |
| ^^   | 用户数据 | |
| ^^   | 用户程序 | |
```

进程**上下文 (context)** 是对进程执行过程的描述，由进程的用户地址空间内容、硬件寄存器内容及与该进程相关的核心数据结构组成：
+ **用户级上下文**：进程的用户地址空间，包含用户正文段、用户数据段、用户栈；
+ **寄存器级上下文**：程序计数器、处理机状态寄存器、栈指针、通用寄存器的值；
+ **系统级上下文**：PCB 和资源表格、核心栈（进程调用系统核心过程时需要）。

> [!note] 进程映像与进程上下文的关系
> 进程**映像**是进程的**静态存储结构**，回答「进程在内存中有什么」；进程**上下文**是进程的**动态执行状态**，回答「进程执行到哪里了」。
>
> 三部分上下文的来源：
> + **用户级上下文**来源于映像的用户空间（用户程序、数据、栈、堆）；
> + **系统级上下文**来源于映像的内核空间（PCB）加上内核运行时栈；
> + **寄存器级上下文**来源于 CPU 瞬态寄存器值，**不在映像中**。
>
> 进程切换保存和恢复的是**上下文**（尤其寄存器级上下文），映像始终驻留在内存中（或被换出到磁盘），不需要在切换时整体搬移。

> [!note] 操作系统内核与进程的关系
> **OS 内核本身不是一个进程**，内核代码的执行不被调度，而是在特定触发条件下获得 CPU 控制权（中断、陷阱、系统调用），以内核态运行。
>
> OS 与进程地址空间的关系存在三种模式：
> + **传统方法**：OS 不作为进程地址空间的一部分。OS 拥有自己独立的内存区域，进程地址空间与 OS 内核空间完全分离。进程通过陷入内核的方式请求 OS 服务。
> + **早期 UNIX 方法**：OS 整体作为每个进程地址空间的一部分。每个进程的虚拟地址空间中，高端地址区域映射到同一份 OS 内核代码和数据，进程切换到内核态时无需切换地址空间，系统调用开销低。
> + **Windows / Linux 方法**：OS 功能分置——仅 OS **核心**（微内核或内核主体）作为进程地址空间的一部分映射在每个进程的高端地址，而 OS 的**系统服务**（如文件系统服务、网络服务）以独立**服务进程**的形式运行，受调度器管理。
>
> 三种模式的共同前提：OS 内核在**内核态**执行，用户进程在**用户态**执行。用户进程只能通过系统调用陷入内核，由内核代为完成特权操作。

## 进程状态模型

### 三状态进程模型

进程在其生命周期中，会在以下三种**基本状态**之间转换：
+ **运行态 (running)**：进程占有CPU，并在CPU上运行。
+ **就绪态 (ready)**：进程已经具备运行条件，但由于无CPU暂时不能运行，当调度给其CPU时立即可以运行。
+ **阻塞态 (blocked)**：进程因等待某种事件的发生而暂时不能运行，即使CPU空闲该进程也不可运行。

在进程运行过程中，由于进程自身进展情况及外界环境的变化，三种**基本状态**之间可以实现以下的转换：
+ **运行 → 阻塞**：进程等待某事件（资源不可访问、I/O 初始化等待结果、等待 IPC）。
+ **运行 → 就绪**：时间片用完，或被高优先级进程抢占。
+ **就绪 → 运行**：调度程序选择该进程运行。
+ **阻塞 → 就绪**：所等待的事件发生。

![[三状态进程模型.png]]

> [!note] 不可能直接发生的状态转换
> 「**阻塞 → 运行**」和「**就绪 → 阻塞**」是不可能直接发生的状态转换。阻塞进程必须先经过就绪态才能被调度运行，就绪进程必须经过运行态才能因等待事件而阻塞。

### 五状态进程模型

在[[#三状态进程模型]]基础上，增加**新建态**和**终止态**：
+ **新建态 (new)**：OS 已完成为创建进程所必要的工作（分配 PID、创建管理表格），但因资源有限尚未**许可 (admit)** 执行。
+ **终止态 (terminated)**：进程被**撤销 (destroy)** 后进入，不再有执行资格，但表格和其他信息可能暂时保留。

![[五状态进程模型.png]]

### 七状态进程模型

当系统内存不足时，可能将进程暂时换出到外存，于是引入**挂起状态 (suspended)**：
+ **就绪挂起 (suspended ready)**：进程在外存，但只要进入内存即可运行。
+ **阻塞挂起 (suspended blocked)**：进程在外存并等待某事件的出现。

**挂起 (suspend)** 操作将进程从内存转到外存：
+ 阻塞 → 阻塞挂起：释放内存以提交新进程
+ 就绪 → 就绪挂起：挂起低优先级就绪进程
+ 运行 → 就绪挂起：高优先级阻塞挂起进程进入就绪挂起时，抢占当前进程

**激活 (activate)** 操作将进程从外存调回内存：
+ 就绪挂起 → 就绪：没有就绪进程或挂起就绪进程优先级更高
+ 阻塞挂起 → 阻塞：释放足够内存，激活高优先级阻塞挂起进程

![[七状态进程模型.png]]

## 进程控制

**进程控制**是操作系统使用具有特定功能的程序段来创建与撤销进程以及完成进程状态转换，以达到多进程高效率并发执行和协调、实现资源共享的目的。进程控制由**原语**完成。

> [!definition] 原语
> **原语 (primitive)** 是由若干条指令构成的**原子操作**过程，作为一个整体而不可分割。

> [!note] 不是原语的系统调用
> 并非所有系统调用都是原语。例如进程 A 调用 `read()` 因无数据而阻塞，在 `read()` 里未返回；然后进程 B 调用 `read()`，此时 `read()` 被重入。系统调用不一定一次执行完并返回该进程，有可能在特定点暂停而转入其他进程。

进程控制主要体现在三个方面：创建与撤销、阻塞与唤醒、挂起与激活。每类操作由对应的**原语**完成，原语执行完毕后统一**转处理机调度**，使调度器有机会重新选择进程运行。

### 创建与撤销

#### 创建原语

进程的创建由**创建原语**完成。进程以 PCB 为存在标志，因此创建新进程的主要任务就是为新进程建立 PCB。

导致进程创建的原因：
+ 系统初始化
+ 执行一个创建进程的系统调用（如 `fork()`）
+ 用户请求创建一个新进程
+ 批处理作业的初始化

#### 撤销原语

进程的终止由**撤销原语**完成。进程终止之后，系统应及时回收它占有的所有资源并释放 PCB 结构本身。

导致进程被撤销的原因：
```tx
| 类型 | 方式 | 示例 |
| :- | :- | :- |
| 正常退出 | 自愿的 | `exit()` 调用 |
| 出错退出 | 自愿的 | 程序自行检测到不可恢复错误后退出 |
| 严重错误 | 非自愿的 | 非法指令、访问越界等，由 OS 强制终止 |
| 被杀死 | 非自愿的 | 被其他进程通过 `kill` 等信号杀死 |
```

![[创建和撤销.png]]

### 阻塞与唤醒

#### 阻塞原语

**阻塞原语**实现进程从**运行状态到阻塞状态**的转换。当进程在执行中需要等待某一事件发生（如等待 I/O 完成、等待资源可用）时，调用阻塞原语将自己阻塞。

> [!note] 阻塞是主动行为
> 阻塞是由**本进程自己调用**的原语——只有运行中的进程才知道何时需要等待某事件。没有其他进程可以「强行阻塞」一个正在运行的进程。

#### 唤醒原语

**唤醒原语**实现进程从**阻塞状态到就绪状态**的转换。当阻塞进程所等待的事件发生时，由发现者（通常是中断处理程序或其他进程）调用唤醒原语将阻塞进程唤醒。

![[阻塞与唤醒.png]]

> [!note] 阻塞与唤醒的配对
> 阻塞和唤醒是一对**成对使用**的原语：阻塞由进程自己调用，唤醒由事件发现者调用。两者协同实现进程间的**同步**。

### 挂起与激活

当系统内存不足时，可能需要将部分进程暂时换出到外存，以释放内存给更需要的进程。挂起与激活操作成对出现，负责内存和外存之间的进程迁移。

#### 挂起原语

**挂起原语**将进程从内存调出到外存，根据进程当前状态分别处理：就绪进程挂起到就绪挂起队列，阻塞进程挂起到阻塞挂起队列。

#### 激活原语

**激活原语**将进程从外存调入内存，是挂起的逆操作。根据进程挂起前的状态，分别恢复到就绪队列或阻塞队列。

![[挂起与激活.png]]


# _2.2_  线程

## 线程的引入

### 引入线程的动机

未引入线程时，[[#进程]]具有两个基本属性：
+ 进程是**资源分配单位**，拥有虚拟地址空间，保存进程映像，控制一些存储和设备资源，有状态、优先级；
+ 进程是 **CPU调度单位**，是程序的执行轨迹。

这两个属性构成进程并发执行的基础，但也带来了问题：
1. **降低并发开销的需求**：由于进程是资源分配单位，在创建、终止、切换的过程中系统必须付出**较大的时空开销**，因此系统中并发进程数目不宜过多，进程切换频率也不宜过高，这限制了系统的并发程度。
2. **利用多处理器/多核架构的需求**：在多处理器系统和多核处理器中，传统单线程进程在一个时刻只能运行在一个处理器上，不能充分利用硬件并行能力。
3. **应用内在的并发需求**：许多应用（如字处理程序、Web 服务器）天然存在可并发的任务，需要操作系统提供轻量级的并发支持。

> [!note] 多处理器系统架构
> 
> 按照处理器之间的关系，多处理器系统分为：
> + **对称多处理 (symmetric multiprocessing, SMP)**，所有CPU地位对等，共享同一个操作系统实例，任意CPU均可执行内核代码和用户程序。SMP是现代通用CPU的主流模型。
> + **非对称多处理 (asymmetric multiprocessing, AMP)**，CPU之间存在主从关系，常见于嵌入式系统。
> 
> 按照内存组织方式，多处理器系统分为：
> + 共享内存架构，又分为——
>     + **统一内存访问 (uniform memory access, UMA)**，所有CPU共享一块物理内存，访问延迟相同。UMA结构简单，但扩展性差。
>     + **非统一内存访问 (non-uniform memory access, NUMA)**，每个节点有独立的局部内存，访问本地内存快、远程内存慢。NUMA扩展性好，适合大规模服务器。
> + **非共享内存架构**，典型的是**集群系统 (cluster)**，每个处理部件都是一个独立的计算机，通过高速网络互连，构成廉价的高性能计算机系统。

### 线程的基本概念

解决以上问题的思路是，**将进程的两个基本属性分开**，引入线程的概念，作为进程的组成部分。在支持线程的操作系统中，进程只作为**资源分配单位**，线程则作为 **CPU调度单位**。

> [!definition] 线程
> **线程 (thread)** 是进程中能够独立执行的实体（控制流），是**处理器调度的基本单位**。
> 每个进程内允许包含多个并发执行的线程，这就是**多线程 (multithreading)**。同一进程中的所有线程共享进程获得的主存空间和资源。

在多线程环境下，**进程是操作系统中除处理器外进行资源分配和保护的基本单位**。进程拥有独立的虚拟地址空间，容纳进程映像，并以进程为单位对各种资源实施保护，例如受保护地访问文件、外部设备及其他进程（即进程间通信）。

由于处理机调度的基本单位现在是线程而不是进程，因此在创建一个进程时，同时要为该进程创建一个**主线程**，否则该进程无法被调度执行。

### 线程的属性

线程概念试图实现的是**共享一组资源**的**多个线程的执行能力**，以便这些线程可以为完成某一任务而共同工作。从资源归属角度，线程的属性可划分为**线程级资源**（各线程独享）和**进程级资源**（同进程内所有线程共享）。

#### 线程级资源

每个线程拥有自己**独立的执行上下文**，这些资源是**线程私有**的，创建线程时分配，切换线程时保存和恢复。

```tx
| 线程的私有资源 | 说明 |
| :- | :- |
| 线程标识 (TID) | 线程的唯一标识，记录在 TCB 中 |
| 执行的基本状态 | 运行、就绪、阻塞；各线程状态独立，转换类似于[[#进程]] |
| 上下文 | CPU 现场，包含通用寄存器、程序计数器 (PC)、状态寄存器 (PSW) |
| 执行栈 | **用户栈**（函数调用参数、返回地址、局部变量）、**内核栈** |\
|        | ——用户栈用于用户态函数调用；当线程通过系统调用陷入内核时，使用内核栈保存内核态的执行上下文。 |
| [[#线程控制块]] (TCB) | 包含线程标识、线程状态、调度参数、CPU 现场信息等 |\
|                  | ——内核级线程的 TCB 中同时记录用户栈指针和系统栈指针。 |
```

#### 进程级资源

同一进程内的所有线程**共享**以下资源，这些资源由进程统一获取和管理，线程间可直接访问而无需通过内核。

```tx
| 线程间共享资源 | 说明 |
| :- | :- |
| 地址空间 | 所有线程运行在同一个进程的虚拟地址空间内，进程映像为所有线程所见 |
| 全局变量、堆内存 | 全局变量、静态变量、堆上分配的内存空间对所有线程可见 |
| 打开文件 | 文件描述符进程内共享，一个线程打开的文件可被同进程其他线程直接读写 |
| 子进程 | 进程的父子关系为进程级属性 |\
|      | ——通过 `fork()` 创建的子进程属于该进程，所有线程共享这一关系。多线程进程中某线程调用 `fork()` 时，子进程通常只复制调用线程。 |
| 待处理警告 (alarm) | `alarm()` 和间隔定时器是进程级资源，一个进程只能设置一个实时闹钟 |
| 信号及其句柄 | 信号处理函数（句柄）在进程级安装，所有线程共享相同的信号处置 |\
|             | ——但每个线程可拥有独立的**信号掩码**，信号可面向进程投递（由任一线程处理）或面向特定线程投递。 |
| 记账信息 (accounting information) | 累计 CPU 时间、页面调度次数、I/O 统计等 |
| [[#进程#进程控制块\|进程控制块]] (PCB) | PID、UID、内存管理信息、资源占用统计等由 PCB 统一记录 |
```

> [!note] 各类变量的线程/进程归属与安全
> 进程地址空间中不同区域的变量，在**可见性**和**线程安全性**上有本质区别。凡是在线程栈帧上分配的内容，均为线程私有的；凡是在进程全局空间中分配的内容，均为线程共享的，共享即意味着需要同步。
>
> ```tx
> | 变量类型 | 存储区域 | 可见性 | 线程安全性 | 说明 |
> | :- | :- | :- | :- | :- |
> | 局部变量 | 线程栈 | 线程私有 | 天然安全 | 分配在各线程自己的栈帧中，其他线程无法访问 |
> | 全局变量 | 数据段 | 所有线程共享 | 需同步保护 | 同进程内任一线程均可读写 |
> | 静态变量 (`static`) | 数据段 | 所有线程共享 | 需同步保护 | 和全局变量一样，编译期分配在数据段 |
> | 堆内存 (`malloc`/`new`) | 堆 | 所有线程共享 | 需同步保护 | 堆是进程级资源，任何线程拿到地址后均可访问 |
> ```
>
> 特别地，指针变量本身若为局部变量，则存储在栈上，属线程私有；但若它指向的是堆内存，则安全的是指针本身，所指向的数据仍为所有线程共享，必须同步保护。
>

线程之间还可以互相进行控制操作，例如一个线程可以创建、撤销同进程内的另一个线程。**线程创建的开销远小于进程创建**，因为无需重新分配地址空间和资源表。

> [!note] 挂起状态是进程级状态
> 挂起状态对线程没有意义——如果进程挂起后被换出主存，则它的所有线程（共享进程的地址空间）也必须全部换出。类似地，进程的终止会导致所有线程的终止。

### 线程控制块

**线程控制块 (thread control block, TCB)** 是标志线程存在的数据结构，包含线程标识、线程状态、调度参数和CPU现场信息。其中，CPU现场信息主要指通用寄存器、指令计数器PC、用户栈指针；内核支持的线程的TCB还包含系统栈指针。

TCB中的内容一般较少，因为资源分配等信息已记录于所属进程的PCB中。不同于PCB严格属于内核，TCB可能属于操作系统空间，也可能属于用户进程空间，这取决于线程的实现方式。

### 进程与线程的比较

相较于进程，线程具有以下特点：
+ **地址空间和其他资源**在进程间相互独立，而在同一进程的**各线程间共享**，某进程内的线程在其他进程不可见；
+ 进程间通信采用IPC机制，线程间可以直接**读写进程数据段**通信，只是仍需要进程同步和互斥手段的辅助以保证数据的一致性；
+ 线程**上下文切换**比进程上下文切换要快得多。

因此，引入线程的**优点**主要在于：
+ 线程的创建、退出和调度都更轻量，减小了并发执行的时间和空间开销，因此容许在系统中建立更多的线程来提高并发程度。
+ 同一进程内的线程共享内存和文件，因此它们之间**相互通信无须调用内核**。
+ 适合多处理机系统。

其**缺点**主要是引入了更多复杂性，在影响稳定性的同时，线程本身的调度也需要消耗较多的CPU资源。


## 线程的实现

### 用户级线程

**用户级线程 (user level thread, ULT)** 的线程管理工作完全在用户空间内完成，操作系统内核**不知道线程的存在**，故线程切换不需要核心态特权。

> [!note] ULT与内核的关系
> 操作系统内核不知道线程的活动，仍然以**进程**为单位进行处理器调度。当线程调用阻塞型系统调用时，**整个进程阻塞**。线程状态与进程状态是独立的，可能线程处于运行态而进程处于就绪态。
> 
> 例如，假设正在执行进程P中的线程T1，可能出现下列情况：
> + 线程T1发出一个阻塞式系统调用（例如I/O操作），导致控制转到内核，内核启动I/O操作，将进程P置为**阻塞态**，并切换到另一个进程。按照由线程库所维护的数据结构，线程T1仍处在**运行态**。
> + 进程P时间片用完，内核从进程P剥夺处理器，把它放入就绪队列，切换到另一个就绪进程。此时，由于进程P中线程T1正在运行，按由线程库维护的数据结构，线程T1仍处于**运行态**，但是进程P处于**就绪态**。
> + 线程T1执行到某处后需要线程T2的某些操作，于是让线程T1变成**阻塞态**，而线程T2从就绪态转为**运行态**，在这一过程中进程P始终处于**运行态**。

线程管理通过用户空间的**线程库**（运行时系统）实现。线程库是管理ULT的例行程序包，包含执行以下功能的例程：
+ 创建与撤销线程；
+ 线程间传递消息和数据；
+ 调度线程执行；
+ 保护和恢复线程上下文。

应用程序需调用线程库中的例程进行编程，与线程库**链接**后形成可执行程序。对于POSIX，`pthread` 是广泛采用的线程库接口。

每个进程需要有专用的**线程表 (thread table)**，由运行时系统管理。线程表中的每个表项即一个TCB，对应一个线程，记录线程标识、线程状态、调度参数和CPU现场信息等。线程表**由线程库管理**，当线程由运行态切换到就绪态或阻塞态时，线程库负责保存重新运行该线程所需的信息到TCB中，并在切换回运行态时恢复上下文。

使用ULT的**优点**主要是：
+ 线程切换不调用内核，性能好；
+ 调度是应用特定的，可针对应用优化，线程库的线程调度算法与操作系统的低级调度算法无关；
+ 无论内核是否支持线程，只需要线程库即可运行。

使用ULT的**缺点**主要是：
+ 调度通常采用非抢先式和更简单的规则；
+ 大多数系统调用是阻塞的，内核阻塞进程则进程中所有线程被阻塞；
+ **同一进程的两个线程不能同时运行于两个处理器上**。

### 内核级线程

**内核级线程 (kernel level thread, KLT)** 的线程管理工作由**内核**完成。内核中需要有记录系统中所有线程的**线程表**，包含每个线程的TCB；相应地，不需要线程库，每个进程中不需要线程表。

对应于ULT的缺点，使用KLT的**优点**主要是：
+ 在多处理器上，内核能同时调度同一进程中的多个线程并行执行；
+ 阻塞在**线程一级**完成，一个线程阻塞不影响同进程其他线程；
+ 内核自身也可用多线程技术实现，从而提高内核的并发能力。

使用KLT的**缺点**主要是，线程在用户态运行，而调度管理在内核实现，需要频繁进行模式切换，系统开销较大。

### 混合实现

混合实现将 ULT 映射到 KLT 上：内核只认识 KLT 并对其进行调度，而某些 KLT 的顶部有若干可以轮流使用的 ULT 集合。线程创建、调度、同步在用户级完成，程序员可以调节 KLT 数量以求最佳效果。

> [!example]- Solaris 的三层线程模型
> Solaris 实现了三个层次：**ULT** → **LWP**（Light Weight Process, 轻量级进程）→ **KLT**。LWP 是基于内核线程的高级抽象，由内核独立调度，能在多处理器中并发执行。LWP 作为 ULT 和 KLT 之间的桥梁，实现了多对多映射，灵活高效。但从 Solaris 10 之后因其复杂度过高而被放弃。


# _2.3_  进程间通信

## 互斥与同步问题

在多道程序设计环境下，并发执行的多个进程相互隔离、独立异步地向前推进。虽然进程隔离保证了稳定性和安全性，但不同进程需要协同工作时便面临挑战。**进程间通信 (inter-process communication, IPC)** 主要解决两个问题：
+ **信息共享与数据交换**——高级进程间通信
+ **共享资源的有序访问**——互斥与同步

如果没有有序访问机制，IPC带来的不是协作而是混乱，典型的后果是**竞争条件 (race condition)**：多个进程访问共享资源时，最终结果取决于进程运行的精确时序。

> [!definition] 互斥、同步
> 进程的**互斥 (mutual exclusion)** 是指若干个进程同时竞争一个需要互斥使用的资源时，任何时刻最多允许一个进程去使用，其他要使用该资源的进程必须等待，直到该资源被释放。互斥产生于**资源共享**，针对「同一时刻访问」问题，确保共享资源在某一时刻只能被一个进程拥有，属于无意识的安排。
> 
> 进程的**同步 (synchronization)** 是指协作进程之间相互等待对方消息或信号的协调关系。同步产生于**进程合作**，针对「访问先后顺序」问题，是有意识的安排。
> 
> 逐次使用互斥资源也是对进程使用资源次序上的一种协调，因此**互斥是一种特殊的同步**。

互斥与同步问题对于线程同样适用。

### 临界资源和临界区

系统中某些共享资源一次只允许一个进程使用，这类资源称为**临界资源 (critical resource)**。硬件临界资源如打印机、光盘刻录机；软件临界资源如只能互斥访问的变量、表格、队列。并非所有共享资源都是临界资源。

**临界区 (critical section)** 是进程中访问临界资源的代码片段，其附近的代码结构分为：
+ **进入区 (entry section)**：检查可否进入临界区；如果可以进入，设置相应「正在访问临界区」标志。
+ **临界区**：访问临界资源，**必须互斥地执行**。
+ **退出区 (exit section)**：重置「正在访问临界区」标志，允许其他进程进入。
+ 剩余区 (remainder section)：不访问临界资源的代码。

访问临界区时，应遵循以下四个原则，以保证使用临界资源的进程能够正确和高效地进行协作，避免竞争条件：
1. **空闲让进**。当无进程在临界区时，任何有权进程可以进入。
2. **忙则等待**。不允许两个以上进程同时进入临界区。
3. **有限等待**。任何进入临界区的要求应在有限时间内得到满足。
4. **让权等待**。不能进入临界区的进程应放弃占用 CPU，即应避免连续测试一个变量直到出现某个值的**忙等待 (busy waiting)** 浪费 CPU 时间。用于忙等待的锁称为**自旋锁 (spin lock)**。

### 经典软件互斥算法

#### 锁变量法

使用共享变量 `lock` 标记临界区是否被占用：

```c
while (lock == 1);   // 忙等待
lock = 1;
// 临界区
lock = 0;
```

这一方法非常简单，但存在以下问题：
+ 违反「忙则等待」原则：检查和设置操作不原子，`while (lock == 1)` 和 `lock = 1` 之间可能发生进程切换，导致多个进程同时进入临界区；
+ 违反「让权等待」原则：存在忙等待问题，浪费 CPU 时间。

#### 轮转法

设置变量 `turn` 记录轮到哪个进程进入临界区，进程严格轮流进入。

```tx
| 进程 0 | 进程 1 |
| :- | :- |
| ```c | ```c |\
| while (turn != 0); | while (turn != 1); |\
| // 临界区 | // 临界区 |\
| turn = 1; | turn = 0; |\
| ``` | ``` |
```

这一方法解决了锁变量法的互斥问题，但仍有忙等待，此外因**程序代码不对称**还存在以下缺点：
+ 违反「空闲让进」原则：当一个进程完成临界区后，必须等待另一个进程进入并离开临界区才能再次进入，即使另一个进程不需要进入临界区；
+ 违反「有限等待」原则：当一个进程完成临界区后，另一个进程可能永远不进入临界区，导致前者无法再次进入。

#### Peterson算法

设置整型变量 `turn` 和布尔数组 `interested[N]`：

```c
void enter_region(int process) {
    int other = 1 - process;      // 另一个进程的编号
    interested[process] = TRUE;   // process 想进入临界区
    turn = other;                 // 让另一个进程先行
    while (interested[other] == TRUE && turn == other);  // 等待
}
```

只有当对方不感兴趣（`interested[other]` 为 `FALSE`）或轮到本进程（`turn` 为 `process`）时，进程才能进入临界区，**正确解决了两个进程的互斥问题**，克服了轮转法的缺点。但该算法**仅适用于两个进程**，且仍存在忙等待问题。

> [!note] 标志牌算法的死锁与Peterson算法的破解
> 若只有 `interested[]` 标志牌而没有 `turn`，两进程同时举牌时将互相看到对方举牌，从而在 while 上死等。
>
> 为此，Peterson 加入**共享变量 `turn`** 作为冲突裁决者，两进程都执行 `turn = other` 把优先权让给对方，后写的将覆盖前写的，等同于后写 `turn` 的进程把通道让给了对方，自己被while条件挡住。只有一方想进入临界区时，`interested[other]` 为 `FALSE`，结果与 `turn` 无关，不影响直接通过。
> 
> 但是，`turn` 的裁决依赖于其只有两个可能值，`other = 1 - process`，因此Peterson算法无法扩展到三个或更多进程。在其多进程推广Filter算法中，`turn` 变为一个数组，进程需要逐级竞争进入临界区，虽然能正确解决多进程互斥问题，但实现复杂，忙等待开销大。

### 硬件互斥方法

#### 禁止中断方法

因为只有在发生时钟中断或其他中断时才会进行进程切换，可以考虑**进入临界区前关中断**，离开后开中断。

这一方法简单但不实用：
+ 把特权指令交给用户进程导致系统可靠性较差；
+ 不适用于多处理器。

#### 专用硬件指令方法

利用处理机提供的专门的硬件指令，**将读和写合并在一条指令中完成**——CPU 以指令为最小执行单位，指令执行期间不响应中断、不发生进程切换，因此外界只能看到执行前或执行后的结果，不存在「读完了但还没写」的时间窗口。

+ **测试并设置 (test-and-set) 指令**原子地将目标位旧值存入 `CF` 标志位，同时将该位置为置1或0，如 x86 的 `BTS`/`BTR`：
    ```assembly
    enter_region:
        LOCK BTS lock, 0     // CF ← lock 第 0 位旧值，同时将该位置 1
        JC enter_region      // CF = 1 说明原来就是 1（已被占），继续忙等
        RET                  // CF = 0 说明原来空闲，刚抢到了锁
    
    leave_region:
        LOCK BTR lock, 0     // 原子地将 lock 第 0 位清 0：CF ← 旧值，同时置 0
        RET
    ```
    其中指令前缀 `LOCK` 用于锁内存总线，以防其他处理器访问内存；**`BTS` 本身只做「读旧值并置 1」而不做判断**，判断由下一条 `JC` 根据已保存的 `CF` 完成。原子性保护的是「读—写」这一对操作，「读—判」并不需要原子，因为旧值已安全存入 `CF`。
+ **对换 (SWAP) 指令**原子地交换两个操作数，如 x86 的 `XCHG`：
    ```assembly
    enter_region:
        MOV reg, 1           // reg <- 1 将要写入 lock，表示想进入临界区
        LOCK XCHG lock, reg  // lock <--> reg
        CMP reg, 0           // lock 原来为 1 则忙等
        JNZ enter_region
        RET
    
    leave_region:
        LOCK MOV lock, 0
        RET
    ```

硬件指令方法的优点在于，其简单且易验证，可支持进程中多个临界区，**适用于任意数目进程**；缺点在于，仍存在忙等待问题，且需要处理器提供专门的指令支持。


> [!danger] 优先级反转问题
> 在存在优先级调度的系统中，忙等待可能导致**优先级反转 (priority inversion)** 问题。
> 
> 例如，假设高优先级进程H就绪后抢占正在临界区中的低优先级进程L，但因L被抢占**无法离开临界区**，H在忙等循环中无限等待。

## 信号量

软件和硬件互斥方法均为平等进程间的协商机制，且都有忙等待的缺点。1965年，Dijkstra提出**信号量 (semaphore)** 概念，由 OS 作为地位高于进程的管理者来解决公有资源的使用问题。

### 信号量的数据结构

每个信号量 `s` 包含一个整型计数值 `s.count` 和一个等待队列 `s.queue`，队列中是阻塞在该信号量上的各个进程的标识。信号量只能通过初始化以及 P、V 两个原语来访问，作为 OS 核心代码执行，不受进程调度的打断。

一般地，信号量计数值表示资源的可用情况，称为**资源信号量**：
+ `s.count` > 0 表示有 count 个资源可用；
+ `s.count` = 0 表示无资源可用；
+ `s.count` < 0 时，|`s.count`| 表示等待队列中的进程个数。

若不需要信号量的计数能力，只需要对临界资源的互斥访问，则通常置初值为1，称为**互斥信号量**。

### P、V原语

**P (_proberen_) 操作**表示申请资源，也称为 wait/down 操作；**V (_verhogen_) 操作**表示释放资源，也称为 signal/up 操作。P、V 原语的定义可写为：
```tx
| P原语 | V原语 |
| :- | :- | 
| ```c | ```c \
| P(s) {                         | V(s) {                        \
|     --s.count;                 |     ++s.count;                   \
|     if (s.count < 0) {         |     if (s.count <= 0) {               \
|         调用进程进入 s.queue;  |          从 s.queue 取出一个进程 P;   \
|         阻塞调用进程;          |          P 进入就绪队列;         \
|     }                          |    }             \
| }                              | }                              \
| ``` | ``` |
```

P、V操作都是**原语**，由操作系统内核提供支持，执行过程中不可被打断，因此具有原子性。

#### 使用P、V实现互斥

为临界资设置互斥信号量 `mutex`，初值为1，**将临界区代码包裹在 `P(mutex)` 和 `V(mutex)` 之间**，即可实现互斥访问。

```c
P(mutex);
// 临界区
V(mutex);
```

#### 使用P、V实现同步

为每个前趋关系设置信号量，初值为0，**在前趋进程中执行V操作，在后续进程中执行P操作**，即可实现同步。

例如，若要求代码段 C1 在 C2 开始前完成，可设信号量 `s12`，C1 中执行 `V(s12)`，C2 中执行 `P(s12)`：
```tx
| 进程 1 | 进程 2 |
| :- | :- |
| ```c | ```c \
| // C1 | P(s12);   // 取用 s12 资源 \
| V(s12);   // s12 资源自增 | // C2 \
| ``` | ``` |
```

> [!tip] P、V使用要点
> + 必须**成对使用**，不能重复或遗漏。遗漏P不能保证互斥，遗漏V不能释放资源。
> + 互斥操作时，P和V在同一进程中；同步操作时，P和V不在同一进程中。
> + 有多个P操作时，**次序至关重要**。同步P必须在互斥P之前，否则可能死锁。两个V操作的顺序则无关紧要。

### 使用信号量解决经典 IPC 问题

经典 IPC 问题是描述、分析和测试进程同步与互斥机制正确性的模型。

#### 生产者—消费者问题

**生产者—消费者问题 (the producer-consumer problem)** 也称**有限缓冲区问题 (bounded-buffer problem)**，考虑控制若干进程通过有限的共享缓冲区交换数据。假定有 $N$ 个数据单元的共享缓冲区，生产者进程不断写入，而消费者进程不断读出，任何时刻只能有一个进程操作缓冲区。

为解决该问题，需要三个信号量：
+ 资源信号量 `empty`，初值为 $N$，表示缓冲区剩余空位数；
+ 资源信号量 `full`，初值为 0，表示缓冲区中已占用的单元数；
+ 互斥信号量 `mutex`，初值为 1，保护对缓冲区的互斥访问。

```tx
| 生产者 | 消费者 |
| :- | :- |
| ```c | ```c |\
| P(empty); | P(full); |\
| P(mutex); | P(mutex); |\
| // 将一个数据单元放入缓冲区 | // 从缓冲区取出一个数据单元 |\
| V(mutex); | V(mutex); |\
| V(full); | V(empty); |\
| ``` | ``` |
```

> [!danger] P 操作次序
> 每个进程中各个P操作的次序是重要的。在生产者中必须先 `P(empty)` 再 `P(mutex)`，在消费者中必须先 `P(full)` 再 `P(mutex)`）。若先 `P(mutex)` 再 `P(empty)`，当缓冲区满时，生产者持有互斥锁但等待 `empty`，消费者无法获得互斥锁来释放资源，形成死锁。

#### 读者—写者问题

**读者—写者问题 (the readers-writers problem)** 考虑读者、写者两组并发进程，共享一组数据区。数据区允许多个读者同时读，但不允许多个写者同时写，也不允许读者、写者同时操作。

为解决该问题，需要两个互斥信号量：
+ 互斥信号量 `mutex`，初值为 1，保护对数据区的互斥访问；
+ 互斥信号量 `rc_mutex`，初值为 1，保护读者计数器 `rc` 的互斥访问。

```tx
| 读者 | 写者 |
| :- | :- |
| ```c                      | ```c              \
| while (true) {            | while (true) {    \
|     P(rc_mutex);              |            \
|     rc++;                     |              \
|     if (rc == 1) P(mutex);  // 第一个读者负责关闭写入口    |     P(mutex);           \
|     V(rc_mutex);              |                   \
|     // 读操作                 |     // 写操作               \
|     P(rc_mutex);              |                   \
|     rc--;                     |                     \
|     if (rc == 0) V(mutex);  // 最后一个读者负责打开写入口  |     V(mutex);                \
|     V(rc_mutex);              |                   \
| }                         | }                 \
| ```                       | ```               |
```


#### 哲学家进餐问题

**哲学家进餐问题 (the dining philosophers problem)** 是多线程的互斥问题。考虑5个哲学家围圆桌而坐，桌子上放着5把叉子，每两个哲学家之间放1把；哲学家的动作包括思考和进餐，进餐时需要同时拿起他左边和右边的两把叉子，思考时则同时将两把叉子放回原处。

可以针对叉子设置互斥信号量数组 `fork[5]`，初值为1，表示叉子可用。哲学家进餐时需要同时申请两把叉子，吃完后释放两把叉子。但这种方法可能导致死锁：如果每个哲学家都先拿起左边的叉子，那么所有哲学家都在等待右边的叉子，形成循环等待。为避免死锁，可以引入限制，例如最多允许 4 个哲学家同时尝试进餐，或要求奇数号哲学家先拿左叉、偶数号先拿右叉。
```c
#define N 5

void take_forks(int i) {
    if (i % 2 == 0) {
        P(fork[i]);             // 先拿左边的叉子
        P(fork[(i + 1) % N]);   // 再拿右边的叉子
    } else {
        P(fork[(i + 1) % N]);   // 先拿右边的叉子
        P(fork[i]);             // 再拿左边的叉子
    }
}
```

另一种思路是为哲学家设置信号量数组 `s[5]`，初值为0，表示没有进餐。哲学家拿叉子前先检查左右邻居是否正在进餐，只有邻居都不在吃时才允许拿起叉子。这种方案天然避免了死锁：哲学家不会持有一把叉子又等待另一把，因为要么一次性获得两把、要么根本就不拿。
```c
#define N 5

void test(int i) {
    if (state[i] == HUNGRY && state[LEFT] != EATING && state[RIGHT] != EATING) {
        state[i] = EATING;
        V(s[i]);      // 唤醒哲学家 i，使其通过 take_forks 中的 P(s[i])
    }
}

void take_forks(int i) {
    P(mutex);
    state[i] = HUNGRY;
    test(i);          // 尝试拿叉子
    V(mutex);
    P(s[i]);          // 若 test 未令其进餐，则在此阻塞
}

void put_forks(int i) {
    P(mutex);
    state[i] = THINKING;
    test(LEFT);       // 检查左邻居是否可以进餐
    test(RIGHT);      // 检查右邻居是否可以进餐
    V(mutex);
}
```

#### 睡眠理发师问题

理发店有一位理发师、一把理发椅和 $N$ 把等候椅。无顾客时理发师在理发椅上睡觉；顾客到来须唤醒理发师，理发师正忙时若有空椅则坐下等，否则离开。

为解决该问题，需要三个信号量：
+ 资源信号量 `customers`，初值为0，表示等待理发的顾客数；
+ 资源信号量 `barber`，初值为0，表示理发师是否空闲；
+ 互斥信号量 `mutex`，初值为1，保护对等候椅的互斥访问。

```tx
| 顾客 | 理发师 |
| :- | :- |
| ```c | ```c \
| P(mutex);                      | while (true) {                \
| if (waiting < N) {             |     P(customers);             \
|     waiting++;                 |     P(mutex);                 \
|     V(customers);              |     waiting--;                \
|     V(mutex);                  |     V(barber);  // 响应需求而非释放资源 \
|     P(barber);                 |     V(mutex);                  \
|     // 得以理发                |     // 理发操作                \
| } else {                       | }                            \
|     V(mutex);                  |                              \
|     // 离开理发店              |                              \
| }                              |                              \
| ``` | ``` |
```

这里 `barber` 更接近「信号」而非「资源」，顾客通过 `P(barber)` 等待理发师，理发师通过 `V(barber)` 响应顾客的到来，然后**同步**进入理发环节。`customers` 则是一个真正的资源信号量，表示等待理发的顾客数。

## 管程

### 引入动机

信号量的同步控制操作分散在各个进程中，导致正确性分析困难，易读性差，不利于修改和维护。因此，1971年Dijkstra提出「秘书」进程思想，1973年Hansen和Hoare推广为**管程**。

> [!definition] 管程
> **管程 (monitor)** 是关于共享资源的数据结构及一组针对该资源的操作过程所构成的软件模块，其基本思想是把信号量及其操作原语封装在一个对象内部，将共享变量及对其能够进行的所有操作集中在一个模块中。

管程的主要特性是：
+ **模块化**，一个管程是一个基本程序单位，可以单独编译；
+ **抽象数据类型**，不仅有数据，还有对数据操作的代码；
+ **信息封装**，共享变量在管程外部不可见，外部只能通过调用管程中的过程来间接访问。

引入管程可提高代码的可读性，便于修改和维护，正确性易于保证。

### 管程的组成

管程内部包含三个部分：**共享变量、条件变量、管程过程**。当一个进程调用管程过程时，若管程中已有其他活跃进程，调用进程被阻塞在**入口等待队列**，直到另一个进程离开管程而将其唤醒。

**条件变量 (condition variables)** 是管程内的一种数据结构，只能通过 `wait` 和 `signal` 两个原语操作：
+ 调用 `wait(c)`，调用进程阻塞并移入与条件变量 `c` 相关的等待队列，释放管程。
+ 调用 `signal(c)`，若 `c` 链为空则相当于空操作，否则唤醒 `c` 链中第一个等待者。当执行唤醒操作时，管程中存在两个同时活动的进程，
    + 在Hansen语义下，唤醒是管程中最后一个可执行的操作，执行完 `signal` 后唤醒者直接离开管程；
    + 在Hoare语义下，被唤醒者继续运行，而唤醒者等待，通常使用**紧急等待队列**管理这种情形。

> [!note] 管程的 `signal` 与信号量的 V
> 管程中的 `signal` 操作与信号量中的 V 操作不同。若无等待进程，管程的 signal 被忽略，不会记录历史；而信号量的 V 操作会使信号量值增加，影响后续的 P 操作。

管程是**编程语言概念**，编译器必须识别管程并对其互斥做出安排。Java、Pascal 等支持管程，但 C、C++ 及多数语言不直接支持管程。

## 高级进程间通信

信号量、互斥锁和条件变量等同步与互斥机制的核心作用是**建立进程（或线程）之间的时序约束**，主要用于协调执行顺序和传递状态信息，不适合复杂或大规模数据传输。在需要结构化或大量数据交换时，需引入更高级的通信机制。

### 共享内存

在物理内存中划出一块**共享存储区**，通信进程将其映射到各自的进程地址空间，通过直接读写共享区数据实现通信。这一方式效率很高，进程直接读写内存，无需数据拷贝。

操作系统一般只提供共享的内存空间，进程间的读写互斥由程序开发人员负责解决，通过进程的互斥与同步来保证数据一致性。

> [!example]- Linux/Windows 共享内存机制概览
> Linux 支持 `mmap()` 系统调用、POSIX 共享内存和 System V 共享内存三种方式。Windows 采用文件映射机制(`CreateFileMapping` / `MapViewOfFile`)，将文件映射为进程虚拟地址空间的一部分。

使用全局变量在同一个进程内部的线程之间实现通信不称为共享内存。

### 消息传递

消息传递通信是进程间以内存缓冲区为基础、以**消息 (message)** 为单位进行数据交换，分为两种方式：
+ **直接通信**：发送和接收进程显式指定对方标识，实现点对点通信。原语形式为 `send(receiver, message)` / `receive(sender, message)`。主要用于早期微内核系统，**通信关系绑定在进程身份上**，无法处理服务重启、迁移和复杂通信模式，现代系统中逐渐被淘汰。
+ **间接通信**：进程通过中介实体（邮箱、消息队列）进行通信，发送者和接收者完全解耦，支持异步通信。原语形式为 `send(mailbox/queue, message)` / `receive(mailbox/queue, message)`。

### 管道

**管道 (pipe)** 是 UNIX 首创的**以文件系统为基础**的通信方式，以字节流方式传送信息。利用管道时，一个进程的输出可成为另外一个进程的输入，典型应用是 shell 中的管道命令 `|`，如：
```bash
ls | grep "txt" | wc -l
```

管道分为两类：
+ **无名管道 (anonymous pipe)** 是临时对象，通过 `pipe()` 系统调用创建，获得读、写两个文件描述符。只适用于父子进程或父进程安排的子进程之间的通信。
+ **命名管道 (named pipe)** 也称为FIFO，是文件系统的真正实体，可在无亲缘关系的进程间通信。

> [!example]- POSIX/Linux 同步机制
> POSIX 提供三种同步 API：**互斥锁**(mutex, `pthread_mutex_lock/unlock`)、**条件变量**(cond, `pthread_cond_wait/signal`)、**读写锁**(rwlock, `pthread_rwlock_rdlock/wrlock`)。POSIX 信号量（SEM 扩展）分为有名信号量和无名信号量，通过 `sem_wait/sem_post` 实现 P/V 操作。

> [!example]- Windows 同步对象
> Windows 内核支持三种同步对象：**互斥对象**（Mutex, 等价互斥信号量）、**信号量对象**（Semaphore, 等价资源信号量）、**事件对象**（Event, 相当于触发器）。此外还有**临界区对象 (critical section)**，只能在同一进程内使用（非内核对象）。等待函数 `WaitForSingleObject` / `WaitForMultipleObjects` 用于阻塞等待对象变为有信号状态。


# _2.4_  处理机调度

## 处理机调度的基本概念

处理机是计算机系统中最重要的资源。在多道程序系统中，常有多个进程竞争 CPU，操作系统必须选择运行哪个进程，完成这一选择工作的代码称为**调度程序 (scheduler)**。

### 调度时机

处理机调度的时机有：
```tx
| 时机类型 | 具体事件 | 调度要求 |
| :- | :- | :- |
| 进程状态变化 | 进程创建 | 可不立即调度 |
| ^^           | 进程终止，不论正常结束或异常退出 | **必须立即调度** | 
| ^^           | 进程阻塞，转入等待状态 | ^^           |
| 中断发生 | 时钟中断 | 可不立即调度 |
| ^^           | I/O 中断 | ^^           |
```

根据调度程序在上述「可不立即调度」时机是否执行调度程序，调度程序可分为：
+ **非抢先式调度 (non-preemptive)**，调度程序一旦分配CPU后便让进程一直运行，直到进程完成或因等待事件而阻塞。实现简单、开销小，适用于批处理系统。
+ **抢先式调度 (preemptive)**，调度程序可根据规定原则剥夺正在运行的进程的CPU，将其移入就绪队列。主要适用于分时系统和实时系统。抢先原则有两种：**优先级原则**（高优先级抢占低优先级）和**时间片原则**（时间片用完被抢占）。

### 调度层次

调度程序可分为三个层次，分别负责不同时间尺度的调度决策：
+ **长程调度 (long-term scheduling)**，也称**作业调度**、高级调度或宏观调度，负责选择作业进入内存，其时间尺度为分钟到小时级别。
+ **中程调度 (medium-term scheduling)**，也称中级调度或**内外存交换**，负责进程在内存与外存之间的换入换出。
+ **短程调度 (short-term scheduling)**，也称**进程调度**、低级调度或微观调度，负责选择就绪进程使之运行，其时间尺度为毫秒级。

## 处理机调度算法

处理机调度算法的优劣，可以从多个方面判断，其中有些目标取决于环境，有些适用于所有情形。
+ 针对**所有系统**，都需要保证：
    + **公平**，即给每个进程公平的CPU份额
    + **效率**，即保持系统的所有部分都忙碌
+ 对于**批处理系统**，还需要使得吞吐量最大、周转时间最短、CPU利用率最高
    + **吞吐量**：单位时间内完成的最大作业数量
    + **周转时间**：作业从提交到终止间的最小时间
+ 对于**交互式系统**，还需要使得响应时间最短
+ 对于**实时系统**，还需要保证截止时间

### 批处理系统中的调度算法

批处理系统主要关注**高吞吐量、低平均周转时间和高CPU利用率**，目标是全局优化，而不是提高单个任务的性能。批处理调度算法以**非抢先**为主，以避免上下文切换开销。

#### 先来先服务 (FCFS)

**先来先服务 (first come first served, FCFS)** 按进程提交或变为就绪的先后顺序进行调度，是一种非抢先式算法。

FCFS 实现简单，**有利于长作业和 CPU 密集型作业**，而不利于短作业和 I/O 密集型作业。

#### 最短作业优先 (SJF)

**最短作业优先 (shortest job first, SJF)** 优先将 CPU 分配给预计执行时间最短的作业，是一种非抢先式算法，其目标是最小化平均周转时间。

SJF 与 FCFS 相比**改善了平均周转时间**，但对长作业不利，可能导致饥饿。此外，作业执行时间在实际中难以准确预计。

#### 最高响应比优先 (HRRN)

**响应比 (response ratio)** 定义为
$$
\text{响应比} = \cfrac{\text{等待时间} + \text{要求服务时间}}{\text{要求服务时间}} = 1 + \cfrac{\text{等待时间}}{\text{要求服务时间}}
$$
**最高响应比优先 (highest response ratio next, HRRN)** 是介于 FCFS 和 SJF 之间的折中算法，既考虑了作业的等待时间又考虑了运行时间，既照顾短作业又不使长作业等待时间过长。

#### 最短剩余时间优先 (SRT)

**最短剩余时间 (shortest remaining time, SRT) 优先**是[[#最短作业优先 (SJF)]] 的抢先式版本，调度程序总是选择剩余时间最短的进程运行。若新进程的剩余时间比当前运行进程更短，则当前进程被抢占。SRT 可使新的短作业获得良好的服务。

### 交互式系统中的调度算法

交互式系统普遍采用**抢先式**调度，以保障响应时间和公平性。

#### 时间片轮转 (RR)

**时间片轮转 (round robin, RR)** 为每个进程分配一个**时间片 (quantum)**，进程用完时间片后便被剥夺 CPU 并排到就绪队列末尾。若进程在时间片内阻塞或完成，CPU 立即切换。

RR实现简单、公平，隐含所有进程同等重要的假设。时间片长度的选择需要在**切换开销**和**响应时间**之间权衡，时间片过短会导致过多的进程切换、降低 CPU 效率；时间片过长则会使响应时间变长。通常 20 ms～50 ms 是合理的折中范围。

#### 优先级调度

**优先级调度 (priority scheduling)** 为每个进程赋予一个**优先级**，最高优先级的可运行进程先运行。优先级可以是**静态**的，即创建时指定、运行中不变；也可以是**动态**的，例如等待时间长的进程可升高优先级，$\cfrac{1}{上次运行计算时间占比}$ 作为优先级可使 I/O 密集型进程获得较好的服务。

优先级调度解决先选哪个进程运行的问题，同优先级内的进程通常采用时间片轮转。

#### 多级反馈队列 (MLFQ)

**多级反馈队列 (multilevel feadback queue, MLFQ)** 是[[#优先级调度]]与[[#时间片轮转 (RR)]] 结合的改进方案。MLFQ 根据优先级设置时间片长度，高优先级队列的时间片较短，适合交互式进程；低优先级队列的时间片较长，适合 CPU 密集型进程。

典型的MLFQ调度规则是：
+ 新进程进入时，获得高优先级；
+ 进程将**时间片用完**，则降低优先级，提供更长的时间片；
+ 进程**频繁阻塞**，则提高优先级，同时匹配较短的时间片。

MLFQ 通过这一简单规则能够自动识别进程是交互式还是 CPU 密集型，兼顾快速响应和高吞吐量。

#### 最短进程优先

**最短进程优先 (shortest process next)** 是[[#最短作业优先 (SJF)]] 在交互式系统中的版本，将每一条命令的执行看成一个独立的「作业」，根据过去的行为通过**老化 (aging)** 技术预测下一命令的执行时间，即
$$
T^{*} = aT_{0} + (1-a)T_{1}
$$
其中 $a$ 通常取 0.5。

### 实时系统中的调度

实时系统是**时间起主导作用**的系统，计算机必须保证在**截止时间**之前响应刺激，迟到的响应即使正确，也和没有响应一样糟糕。实时系统又可分为严格要求满足的**硬实时 (hard real-time, HRT)** 系统和允许偶尔错过截止时间的**软实时 (soft real-time, SRT)** 系统。

设有 $m$ 个**周期性**事件，事件 $i$ 的到来周期为 $P_i$ 且需要 $C_i$ 秒 CPU 时间，则在单 CPU 上的**可调度条件**为
$$
U = \sum_{i=1}^{m} \frac{C_i}{P_i} \leq 1
$$
这一值称为 **CPU 利用率 (utilization)**，表示系统的负载程度。

基于优先级的**立即抢先 (immediate preemption) 调度**策略是实时系统的主要调度策略。

#### 速率单调调度 (RMS)

**速率单调调度 (rate-monotonic scheduling, RMS)** 是一种**静态优先级**算法，为每个进程预先分配固定的优先级，优先级与事件的发生频率成正比——周期越短、频率越高，优先级越高。RMS 在静态优先级算法中是最优的：若一组周期任务能被任何静态优先级算法调度，则必能被 RMS 调度。

对于 $n$ 个周期任务，RMS 的充分可调度条件为
$$
U = \sum_{i=1}^{n} \frac{C_i}{P_i} \leq n(2^{1/n} - 1)
$$
当 $n \to \infty$ 时，右端趋近于 $\ln 2 \approx 0.693$，即当 CPU 利用率低于约 69.3% 时，RMS 一定能调度所有任务。实际中许多任务集的利用率远高于此界，RMS 仍能成功调度。

RMS 的优点是实现简单、运行时开销小；缺点在于**无法保证高利用率任务集的可调度性**。

#### 最早截止时限优先 (EDF)

**最早截止时限优先 (earliest deadline first, EDF)** 是一种**动态优先级**算法，调度程序在每次调度时动态计算各进程的截止时限，总是选择截止时限最早的进程运行。若新到达进程的截止时限更早，当前进程将被抢占。

EDF 在单 CPU 上是最优的：若存在某种调度方案能使一组任务全部满足截止时限，则 EDF 也能做到。EDF 的充分必要可调度条件为 $U \leq 1$，即只要 CPU 利用率不超过 100%，EDF 就能保证所有任务在截止时限前完成。

EDF 的缺点是每次调度都需比较截止时限，实现开销高于 RMS；此外在过载情况下（$U > 1$），EDF 的行为难以预测，可能发生「多米诺效应」。

#### 最小裕度优先 (LLF)

**最小裕度优先 (least laxity first, LLF)** 也是一种**动态优先级**算法，**裕度 (laxity)** 定义为
$$
\text{裕度} = (\text{截止时刻} - \text{当前时刻}) - \text{剩余运行时间}
$$
裕度衡量了一个任务「还能拖多久不执行」，裕度越小表示该任务越紧迫。LLF 总是选择裕度最小的进程运行。

与 EDF 类似，LLF 在单 CPU 上也是最优的。但 LLF 的上下文切换频率可能高于 EDF：当两个任务的裕度相等或接近时，调度程序可能在两者之间频繁切换，造成较高的调度开销。因此在实际系统中，EDF 通常比 LLF 更常用。

> [!example]- Linux 处理机调度
> Linux 6.6+ 采用 **EEVDF (earliest eligible virtual deadline first)** 调度算法，基于红黑树实现。核心理念：每个进程根据权重计算应得的 CPU 时间，只有「合格 (eligible)」的进程才有资格被选中；在所有合格任务中，选择虚拟截止时间最早的任务运行。调度分级从高到低：Stop-Task → Deadline (SCHED_DEADLINE) → Real-Time (FIFO/RR) → Fair (CFS/EEVDF, SCHED_OTHER) → Idle。针对多核，每个 CPU 有私有运行队列，通过推拉机制实现负载均衡。

> [!example]- Windows 处理机调度
> Windows 11 采用基于优先级的抢占式 MLFQ 变体，32 个优先级队列 (0-31)：1-15 为动态优先级（普通线程, I/O 提升、前台提升、饥饿补偿），16-31 为实时优先级（固定优先级）。支持软亲和性（倾向于将线程放回上次运行的核心）和理想处理器策略。针对 Intel Thread Director 的 P-Core/E-Core 混合架构，支持软硬件协同的非对称调度。

## 死锁

### 死锁的产生

**死锁 (deadlock)** 是指系统中多个进程无限期地等待永远不会发生的条件。若无外力协助，死锁的每个进程都不能继续执行下去。

根据资源性质，可以将资源分为：
+ **可抢占资源**，资源占有进程虽然需要使用该资源，但另一个进程强行把资源从占有者进程处抢来不会引起失败性后果，如 CPU、内存、硬盘等；抢占式调度算法的核心就是基于可抢占资源的抢占原则。
+ **不可抢占资源**，只有占用者进程不再需要使用该资源而主动释放资源外，其它进程不得在占有者进程使用资源过程中强行抢占，否则会引起失败性后果，如打印机、刻录机等。

根据使用期限，可以将资源分为：
+ **永久资源**，可重复使用，如硬件、可再入代码等。
+ **临时性资源**，由一个进程产生、被另一进程短暂使用后无用，如在进程同步和通信中出现的消息、信号、数据。

产生死锁的原因是，**共享**的**不可抢占资源**或相互产生的**临时性资源**不足以满足所有进程的需要，或者**进程推进顺序不当**。

> [!tip] 死锁发生的必要条件
> 死锁发生需要满足以下四个必要条件：
> + **互斥**：任一时刻只允许一个进程使用资源。
> + **非抢占**：进程已占用的资源不会被强制剥夺。
> + **请求和保持**：进程在请求其余资源时，不主动释放已占用的资源。
> + **环路等待**：存在进程循环链，链中每个进程已获得的资源同时被链中下一个进程所请求。
> 
> 只要破坏上述四个条件中的任意一个，就可以防止死锁的发生。

### 死锁的处理

> [!note] 鸵鸟算法
> 死锁问题的发生概率很小而处理代价很大，人们**视而不见**，称为**鸵鸟算法 (ostrich algorithm)**。这是目前实际系统采用最多的一种策略，UNIX、Linux 和 Windows 均采用此方法。

#### 死锁预防

**死锁预防 (deadlock prevention)** 通过限制并发进程**对资源的请求**，使四个必要条件中至少有一个不满足：
+ **破坏互斥条件**，引入假脱机 (spooling) 操作，让专有程序（如打印机守护程序）占有资源。并非所有设备都可假脱机。
+ **破坏非抢占条件**通常无效。
+ **破坏请求和保持条件**，即**预先静态分配法**，让进程在开始前一次申请全部所需资源。降低了资源利用率和并发度，且可能无法预知所需资源。
+ **破坏环路等待条件**，即**有序资源使用法**，将资源按顺序排列，保证请求不形成环路。限制了进程对资源的请求顺序，且资源的排序占用系统开销。

#### 死锁避免

**死锁避免 (deadlock avoidance)** 不事先限制产生死锁的条件，而是在**资源动态分配**过程中，用某种方法防止系统进入**不安全状态**。

> [!definition] 安全状态与不安全状态
> 若存在某种资源调度顺序，保证所有进程正常运行完成，则该状态为**安全状态**，否则为**不安全状态**。
> 
> 只要系统处于安全状态，便可避免死锁；虽然并非所有不安全状态都是死锁，但当系统进入不安全状态后便有可能进入死锁。

在死锁避免方面，Dijkstra于1965年提出了**银行家算法 (banker’s algorithm)**，适用于**资源数量固定不变、用户数固定不变**的系统。银行家算法在每次资源请求时，先试分配资源并执行安全性算法，只有当新状态安全时才正式分配，否则恢复原状态。 ^bankers-algorithm

> [!algorithm]+ 银行家算法
> 银行家算法维护如下的数据结构：
> + 可利用资源向量 `Available[]`，其中 `Available[j]` 表示资源 `j` 的可利用数量
> + **最大需求**矩阵 `Max[][]`，其中 `Max[i][j]` 表示进程 `i` 对资源 `j` 的最大需求数量
> + 分配矩阵 `Allocation[][]`，其中 `Allocation[i][j]` 表示资源 `j` 已分配给进程 `i` 的数量
> + 需求矩阵 `Need[][]`，满足 `Need[i][j] = Max[i][j] - Allocation[i][j]`，表示进程 `i` 对资源 `j` 的剩余需求数量
> 
> 资源请求 `Request[]` 到来时，银行家算法执行如下步骤：
> 1. 检查是否满足最大需求，若 `Request[i] ≤ Need[i]`，即请求小于需求，则继续，否则抛出错误；
> 2. 检查库存余量，若 `Request[i] ≤ Available[i]`，即请求小于库存，则继续；否则进程等待；
> 3. 试分配，按照需求尝试更新 `Available[]`、`Allocation[][]`、`Need[][]`，然后执行**安全性算法**：
>     1. 令动态可分配资源数量 `Work[] <- Available[]`，进程能否完成标志 `Finish[] <- false`；
>     2. 寻找**尚未完成**（`Finish[i] == false`）且**能满足**（`Need[i] <= Work`）的进程 `i`，标记为完成（`Finish[i] <- true`）并释放资源（`Work[] += Allocation[i][]`），然后重复此步骤直到找不到满足条件的进程；
>     3. 若所有 `Finish[i] == true`，系统处于安全状态；否则系统处于不安全状态。
> 4. 若判断新状态安全，则正式分配，否则恢复原状态。

银行家算法的优点是，其**允许互斥、请求和保持、非抢占三个条件存在**，限制条件比死锁预防更宽松，资源利用率更高。但银行家算法要求每类资源数量固定不变、用户数固定不变，这在实际系统中很难满足。

#### 死锁检测与解除

**死锁检测**主要检查系统中是否存在**循环等待**条件，最常用的是一种基于资源分配图和死锁定理的检测死锁算法。

**资源分配图 (RAG)** 是一种有向图，资源用方框中的圆圈表示，进程用带进程名的圆圈表示，由资源到进程的边表示**分配**，由进程到资源类方框的边表示**请求**。检测死锁的过程就是**化简资源分配图**，不断找出既不阻塞又非孤立的进程结点，消去其所有请求边和分配边。若最终所有进程结点都成为孤立结点，则图是可化简的，无死锁；否则即存在等待环，也即存在死锁。

> [!theorem] 死锁定理
> 所有化简顺序均可得到相同的化简结果图。死锁状态的充分必要条件是系统的资源分配图是**不可完全化简**的。

发现进程死锁时，**死锁解除**有两种常用方法：
+ **剥夺资源法**：从其他进程剥夺足够资源给死锁进程。
+ **撤销进程法**：撤销全部死锁进程，或按某种顺序逐个撤销直至死锁消除。


# _3_  存储器管理

## 存储系统的主要任务

设计一个足够大、足够快又足够便宜的单一存储器是不可能的。现代计算机依靠**存储器访问的局部性原理**构建层次化存储体系：

> [!theorem] 存储器访问的局部性原理
> 处理器访问存储器时，无论取指令还是取数据，所访问的存储单元都趋向于聚集在一个较小的连续区域中。
> + **时间局部性**：最近将要用到的信息很可能就是现在正在使用的信息，主要由循环造成。
> + **空间局部性**：最近将要用到的信息很可能与现在正在使用的信息在空间上邻近，主要由顺序执行和数据聚集存放造成。

依靠存储器访问的局部性，现代计算机构建了**速度递减、容量递增、成本递减**的存储层次结构：
```tx
| 存储层次       | 典型介质       | 说明                                                 |
| :--------- | :--------- | :------------------------------------------------- |
| 寄存器        | CPU内的D触发器  | 最快                                                 |
| 高速缓存 | SRAM       | 保存目前正在使用的指令或数据，或者将要用到的指令或数据    |\
|                |            | 可细分出Data Cache、Code Cache、TLB |
| 主存储器（内存）   | DRAM、SDRAM | 保存过一段时间才会用到的指令或数据 |\
| | | 存储管理的对象                                         |
| 辅助存储器（外存）  | 磁盘、SSD     | 保存暂时用不到的程序或数据                             |\
| | | [[#设备管理]]的对象               |
```

缓存对于软件不可见，外存属于设备管理范畴，操作系统存储管理的核心功能针对的是**内存**。具体地，存储管理需要完成四项主要任务：
+ **内存分配和回收**，即负责内存空间的调度策略与数据结构；
+ **地址变换与重定位**，实现逻辑地址到物理地址的映射；
+ **内存保护**，防止进程越界访问并支持受控共享；
+ **内存扩充**，通过覆盖、交换与虚拟存储技术在逻辑上突破物理内存的限制。

### 内存分配和回收

内存分配和回收是存储管理的基础功能，设计者需要选择和确定分配与回收策略以及相应的数据结构，具体涉及四个方面的策略考量：
+ **调入策略**决定用户程序在何时调入内存——在进程请求某页时才调入称为**请调 (demand-paging)**，预先将可能用到的页面调入称为**预调 (prepaging)**。
+ **放置策略**确定用户程序调入内存时的存放位置，在连续分配、页式、段式等不同管理方式下各有不同的实现。
+ **置换策略**规定当需要调入新程序而内存空间不足时，选择哪个程序或页面换出内存。
+ **分配结构**指登记内存使用情况的数据结构，例如空闲分区表、空闲分区队列和位图等，为上述策略的决策提供依据。

内存分配和回收发生在以下典型时机：
+ 进程创建和结束时需要分配和回收内存空间；
+ 进程运行过程中，其占用的内存可能因栈的变化等原因而增减；
+ 进程映像在内存与外存之间传递时需要调入调出；
+ 系统为充分利用内存空间而对内存布局进行调整时也会触发回收与再分配。

### 地址变换与重定位

程序从源代码到可执行文件的运行，经历了多次地址空间的转换：
1. **名空间**：程序员在源代码中使用标识符命名变量、函数等程序实体，这些标识符的集合构成名空间，其中尚无地址的概念。
2. **相对地址空间**：源程序经编译后生成目标模块，各目标模块均以 0 为基地址顺序编址。
3. **逻辑地址空间**（也称虚拟地址空间）：**链接器**将各目标模块及所需库函数装配为一个完整的装入模块，在标识符和地址之间建立映射关系，生成**首地址为 0**、其余地址相对首地址编址的逻辑地址空间。
4. **物理地址空间**：程序最终运行时由机器直接寻址的实际内存地址。

从逻辑地址到物理地址的转换过程称为**地址重定位 (address relocation)**。当程序装入内存时，操作系统为程序分配物理内存空间，由于程序的逻辑地址与所分配的物理地址不一致而 CPU 按物理地址访存，因此必须进行地址转换。

程序在装入内存之前，需要先经过**链接**阶段。链接程序将汇编或编译生成的可重定位目标模块及其依赖的库函数装配为一个完整的装入模块（可执行文件），将各模块的相对地址修改为统一的逻辑地址。根据链接发生的时间，程序链接有三种方式：
+ **静态链接**，在程序运行前事先完成所有链接工作，形成完整的可执行文件。
+ **装入时动态链接**，在程序装入内存的过程中边装入边链接，便于模块的修改、更新和共享。
+ **运行时动态链接**，将部分模块的链接推迟到程序执行过程中、真正需要调用时才进行，能够节省内存空间。

**装入 (loading)** 程序负责将可执行文件加载到内存。根据装入时地址转换的方式，程序装入同样有三种方式：
+ **绝对装入**，在可执行文件中直接记录实际内存地址。绝对装入简单但不灵活，不适于多道程序环境。
+ **可重定位装入**，在装入时一次性完成逻辑地址到物理地址的转换。可执行文件中附有**重定位表**，列出各个需要修改的地址单元，装入时根据实际分配的**内存地址加上偏移量**进行修正。此方式要求程序连续存放且装入后不可移动。
+ **动态运行时装入**，将地址转换推迟到程序真正执行时才进行，由 MMU 硬件在每次访存时完成地址映射。这种方式允许程序分散存放在不连续的内存空间中，支持程序运行期间的**移动**和**执行中产生的地址引用**，是虚拟存储的基础。

### 内存保护

内存保护确保内存中的多道程序只能在给定的内存区域内活动，互不干扰。

保护机制通过硬件实现，即在CPU中设置一对**界限寄存器**，可采用上限寄存器加下限寄存器的方案，也可采用基址寄存器加限长寄存器的方案。每当 CPU 访问主存时，硬件自动将被访问地址与界限寄存器的内容进行比较，若未越界则正常访问，否则产生越界中断（内存保护中断）。

保护机制在获得许可的情况下应当支持受控共享，允许多个进程访问内存的同一部分，这对于进程间协作和代码共享至关重要。

### 内存扩充

实际内存的容量不应该限制用户程序的大小，操作系统的存储管理功能应该允许运行任意大小的程序。内存扩充技术旨在突破物理内存容量的限制，使得用户程序的大小不受实际内存容量的约束。

根据控制方式的不同，内存扩充可分为两类：
+ 由应用程序控制的**覆盖技术**，将程序分割为若干功能上相对独立的**覆盖块 (overlay)**，按照程序的逻辑结构让**不会同时执行**的程序段共享同一块内存区域。
    覆盖块存放在磁盘上，当前一段程序执行结束时，由操作系统将后续程序段调入内存并覆盖不再需要的程序段，从而在逻辑上「扩大」了可用内存。覆盖块的划分**必须由程序员负责**，操作系统仅完成实际的换入换出工作。
+ 由操作系统控制的**交换技术**与**虚拟存储器技术**。
    + **交换 (swapping) 技术**是一种进程在内存与外存之间的动态调度机制，对应**中级调度**。
        当内存空间紧张时，系统将内存中阻塞进程**整体换出**到外存（交换区），把外存中具备运行条件的进程换入内存。交换区必须足够大以存放用户进程的内存映像副本，且通常采用连续分配方式以侧重交换速度。
    + **虚拟存储器 (virtual memory)** 将程序当前需要使用的部分放在内存、暂时不用的部分放在外存，在需要时由系统调入并将不需要的部分调出。
        由虚拟存储器构成的存储空间称为**虚地址空间**。虚拟存储器由操作系统结合 MMU 硬件共同实现，MMU 完成虚拟地址到物理地址的转换工作。

## 连续分配存储管理

**连续分配**要求为用户程序分配一个**连续的**物理内存空间。根据分区策略的不同，分为[[#连续分配存储管理#单一连续区分配|单一连续区分配]]、[[#连续分配存储管理#固定分区分配|固定分区分配]]、[[#连续分配存储管理#动态分区分配|动态分区分配]]、[[#连续分配存储管理#伙伴系统|伙伴系统]]四种实现方式，各有利弊。

> [!note] 连续分配的核心问题
> 连续分配面临的核心问题是**内存碎片**——内碎片浪费分区内部空间，外碎片导致空闲空间无法利用。动态分区通过四种顺序搜索算法（首次/下次/最佳/最坏适配）选择空闲分区、通过**内存紧缩**将碎片拼接为大分区来缓解外碎片；伙伴系统以 $2^k$ 为单位进行分配与合并，折中了固定与动态分区的优劣。

## 页式分配存储管理

**页式存储管理**属于**非连续分配**技术，将内存划分为固定大小的**页框**，程序划分为同样大小的**页**，通过**页表**实现逻辑页号到物理页框号的映射。页面在物理内存中可**不连续存放**，从根本上解决了连续分配的外碎片问题。

页式管理分为[[#页式分配存储管理#简单页式存储管理|简单页式存储管理]]和[[#页式分配存储管理#虚拟页式存储管理|虚拟页式存储管理]]两种：
+ **简单页式**基于实存，要求程序全部装入内存，通过**页表**和**快表 (TLB)** 实现地址转换，通过**多级页表**解决大页表存储问题；
+ **虚拟页式**在简单页式基础上增加**调页**和**页面置换**功能，以磁盘为后备，突破物理内存限制。页面置换算法（OPT/FIFO/LRU/Clock/工作集）直接影响系统性能，不当选择可能导致**颠簸 (thrashing)**。

> [!note] 页式存储管理的地位
> 页式存储管理是现代通用操作系统的主流方案（ARM、RISC-V、MIPS 等架构直接支持，x86 的分段机制也基本被绕过）。其核心优势在于无外碎片和灵活的内存扩充能力，主要局限在于难以实现页面级别的真正共享。


> [!example]- Windows 内存管理
> **地址空间布局**：x86-64 支持 48 位虚拟地址，采用规范地址格式（高 16 位为第 47 位的符号扩展）。用户空间占低 128TB（`0x0`～`0x00007FFF'FFFFFFFF`），布局为 Null 区（防空指针）→ Text → Data → Heap → 内存映射文件 → 共享库(DLL) → Stack。内核空间占高 128TB（`0xFFFF8000'00000000` 起），包含内核代码与数据、分页池（Paged Pool）、非分页池（Nonpaged Pool）、系统 PTE 区和设备映射(MMIO)，本质上是稀疏动态的地址区间集合。
> 
> **四级页表与地址转换**：使用 PML4 → PDPT → PD → PT 四级页表，每级索引 9 位，页内偏移 12 位。各级页表项均为 64 位，结构基本相同，包含物理页框号(PFN)以及 Valid、Write、Accessed、Dirty 等标志位。CR3 寄存器指向 PML4 基地址，MMU 硬件自动完成地址转换，操作系统仅在缺页时介入。
> 
> **大页支持**：通过提前截断页表链条实现——在 PDE 层设 L=1 即得 2MB 大页（省去 PT 层，21 位偏移）；在 PDPTE 层设 L=1 即得 1GB 超大页（省去 PD 和 PT 层，30 位偏移）。大页减少 TLB 压力和页表遍历开销，有利于大数组、数据库和虚拟化场景，但代价是内碎片大、页面置换开销高。
> 
> **缺页处理**：CPU 在地址转换时发现 PTE 无效（V=0）即触发缺页中断（0xe），自动将故障地址存入 CR2。处理流程：`ntoskrnl!KiTrap0E` → `MmAccessFault` 分析 PTE 内容，根据无效 PTE 的类型分三种情况处理：①页文件后备（Pagefile-backed）——从 `pagefile.sys` 调入；②内存映射文件后备（File-backed）——通过原型 PTE 定位 Section Object → File Object 后读入；③过渡（Transition）——页面在备用/修改链表中，直接恢复有效标志。页面调入 I/O 是同步的。
> 
> **虚拟内存分配**：采用两阶段方法——先用 `VirtualAlloc` 保留地址空间（Reserve），再提交物理页面（Commit），`VirtualFree` 回收。进程的虚拟地址空间状态由 **VAD（虚拟地址描述符）** 追踪，组织为 AVL 自平衡二叉树，根指针存于 EPROCESS 结构。内核提供三级内存分配接口：①以页为单位的 `VirtualAlloc`（适合大型稀疏数据结构）；②内存映射文件 `CreateFileMapping`/`MapViewOfFile`（适合大数据流和共享）；③内存堆 `HeapAlloc`/`HeapFree`（适合大量小型申请，进程启动时自带 1MB 默认堆）。
> 
> **内存映射文件与写时复制**：EXE/DLL 通过内存映射文件加载——进程创建时先保留 EXE 基地址空间（默认 `0x00000001'40000000`），系统标记后备存储为磁盘上的 EXE 文件而非页文件，随后装载依赖 DLL。同一 EXE 的多个实例通过映射同一文件对象共享代码与数据页，写入时触发**写时复制 (copy-on-write)**——系统为写入方分配新物理页并复制内容，确保其他实例不受影响。
> 
> **物理内存管理**：通过 **PFN 数据库**（`MMPFN` 数组，每物理页 24 字节）跟踪所有物理页框状态。每个页面处于以下状态之一：活动/有效（在进程工作集中）、过渡（I/O 中）、后备（曾在使用中且内容未变）、修改不写入、空闲、零初始化、坏页。各状态页面组织为相应链表（备用链表、修改链表、空闲链表、零初始化链表、坏页链表），由工作集管理器、修改页面写回程序和零初始化线程协同维护。
> 
> **工作集与页面调度**：为每个进程分配一定数量的页框构成**进程工作集**，可分页的系统代码和数据构成**系统工作集**（含系统缓存、分页池、可换出内核代码和驱动、Win32k.sys 映射视图）。取页策略采用请求调页结合簇方式；换页策略在**多处理器**上采用**局部 FIFO**，在**单处理器**上采用 **LRU**。
> 
> **安全机制**：① **ASLR**（地址空间布局随机化）——随机化 DLL、堆、栈的加载位置；② **DEP**（数据执行保护）——页表项中设 NX 位，数据页不可执行代码；③ **KASLR**（内核地址空间布局随机化）——随机化内核各部分的加载地址，打破固定结构。

> [!example]- Linux 内存管理
> **地址空间布局**：x86-64 平台上同样使用四级页表和规范地址规则，但与 Windows 不同，Linux 采用**不对称切分**——用户空间占据 0 到 `0x00007FFF'FFFFFFFF` 的极大范围，内核空间压在最顶部。内核空间采用**线性映射 (linear mapping)**：`virt = phys + offset`，虽然仍通过 MMU 四级页表映射，但虚拟地址被设计为物理地址的线性偏移，简化了内核态的内存访问。
> 
> **进程用户空间管理**：每个进程的虚拟用户空间由两个核心数据结构描述。① **`mm_struct`**：记录进程地址空间的全局信息（代码段 `start_code`/`end_code`、数据段、堆边界 `start_brk`/`brk`、栈边界 `start_stack`）、页目录指针 `pgd`、由 `vm_area_struct` 构成的内存映射链表 `mmap` 和红黑树 `mm_rb`。② **`vm_area_struct`**：每次在用户空间分配一块内存时生成一个，描述该虚拟区域的起始地址 `vm_start`、结束地址 `vm_end`、访问权限和文件后备（匿名映射时 `vm_file` 为 NULL）。查找页故障对应的虚拟区域时，利用 **AVL 树**（2.4.10 版前）或**红黑树**（2.4.10 版起）快速定位。
> 
> **内存映射**：Linux 将可执行映像映射到进程虚拟地址空间而非直接装入物理内存，引用时才调入。映射分两种：①**文件映射**——可执行文件的 text/data 段、共享库以及普通数据文件（通过 `open`/`read`/`write` 操作），映射区域关联一个文件对象；②**匿名映射**——堆、栈、bss 等不属于任何文件的运行时内存，`vm_file` 为 NULL。`mmap()` 系统调用创建映射，通过 `flags` 参数指定 `MAP_SHARED`（共享修改）或 `MAP_PRIVATE`（写时复制），以及 `MAP_ANONYMOUS`（匿名映射），`munmap()` 撤销映射。
> 
> **缺页处理**：处理器访问未装入物理内存的虚拟地址时报告页故障。Linux 首先查找故障地址对应的 `vm_area_struct`——若找不到则地址无效，向进程发送 `SIGSEGV` 信号；若找到且地址有效，则进一步通过页表项区分：PTE 非空但无效 → 页面在交换文件中，需从交换文件调入；PTE 为空 → 分配新物理页，对映像映射则从可执行映像中读入并更新页表项。这种仅在必要时调入的方式即为**请求分页 (demand-paging)**，处理过程中因涉及磁盘 I/O，系统会切换到其他进程。
> 
> **伙伴系统 (Buddy System)**：物理页框分配采用伙伴算法。内核定义 `free_area` 数组，第 i 项管理大小为 $2^i$ 个页的空闲块，每项包含一个双向链表（串联空闲块起始 PFN）和一张分配位图 `map`。分配时从大于等于请求尺寸的最小可用链表搜索，若只找到更大的块则不断对分并将剩余部分加入对应链表；释放时检查相邻等大小的伙伴是否空闲，若空闲则合并，递归向上形成更大块。这保证了分配以 2 的幂次为单位，兼顾分配速度与碎片控制。
> 
> **Slab 分配器**：仅靠页级分配无法满足内核中大量小对象的需求，因此 Linux 引入 slab 分配器。其核心思想是：为每种常用对象类型（inode、PCB 等）建立独立**缓存 (cache)**，每个缓存由若干 **slab** 构成，每个 slab 占用一个或多个连续物理页框，内部包含预分配好的同类型对象。主要操作接口：`kmem_cache_create`/`kmem_cache_destroy`（创建/销毁缓存）、`kmem_cache_alloc`/`kmem_cache_free`（单类型缓存分配/释放）、`kmalloc`/`kfree`（通用缓存分配/释放）、`kmem_cache_grow`/`kmem_cache_reap`（增长/缩减缓存）。
> 
> **内存交换**：当物理内存紧张时，Linux 采用三级释放策略：①**缩减缓存**——用 **Clock 算法**遍历 `mem_map` 向量，丢弃页高速缓存和缓冲区高速缓存中不再需要的页；②**换出 System V 共享内存页**——同样用 Clock 算法选择换出页；③**换出或丢弃进程页**——`kswapd` 选择可交换进程，基于 **LRU**（页面寿命/最近最少使用）挑选要换出的页，可执行映像的大部分内容可直接丢弃（因为可从磁盘映像重新获取），被锁定的页永不换出。


# _3.1_  连续分配存储管理

物理内存分配可分为连续分配和非连续分配两种方式。连续分配要求为一个用户程序分配一个**连续的**物理内存空间，进一步分为**单一连续区分配**和**分区分配**两类，其中分区分配时操作系统也占用其中一个分区。

一般地，分区存储管理会产生内存碎片问题：
+ **内碎片**是指分配给用户程序的占用分区之内未被利用的空间，通常是因为分配单位过大导致的。
+ **外碎片**是指占用分区之间难以利用的空闲分区，会导致内存中即使存在足够的总空闲空间也可能无法满足连续分配要求。

## 单一连续区分配

单一连续区分配将内存分为操作系统区和用户区，应用程序装入用户区后可使用全部用户空间。这种分配方式适用于单用户、单任务的操作系统（如 MS-DOS、单片机系统）。

为保证用户程序不干扰系统区，需通过硬件保护机构实现内存保护——使用**界限寄存器**，作业运行时检查每次内存访问的地址是否在限定的用户区范围内，若越界则产生越界中断。

单一连续区分配方法简单、对硬件要求低、易于实现，但程序必须全部装入内存，即使很少使用的程序部分也占据内存空间，对小程序而言内存浪费严重。

## 固定分区分配

固定分区分配把内存划分为若干个固定大小的连续分区，每个分区的边界固定不变。

根据分区大小是否一致，可区分为：
+ **分区大小相等的固定分区**，无需放置策略，进程可以放入任何一个空闲分区。若没有空闲分区，操作系统可通过交换技术将某个非运行状态的进程换出到外存；若程序超过了分区大小，可采用覆盖技术解决。这种方案的最大缺点是**内碎片**严重，即使程序很小也要占用整个分区。
+ **分区大小不等的固定分区**，设置多个小分区、适量的中等分区和少量的大分区，放置策略为将作业**放入能够容纳它的最小分区**中。系统为便于管理需要建立**分区表**，记录各分区的大小、起始地址和状态（已分配/未分配）等信息。

固定分区分配没有外碎片，但存在很大的内碎片。分区总数固定，限制了系统中并发执行的程序数目。但整体实现简单、管理开销小。

## 动态分区分配

动态分区分配不预先划分分区，当程序需要装入时，系统从空闲内存区中分配大小恰好等于程序所需的内存空间。分区的大小、数量和位置随着内存中进程的大小和数量而动态变化。进程执行过程中还可通过系统调用改变分区大小，并可为可能增长的进程预留空间。

动态分区没有内碎片，但可能产生**外碎片**。

为控制分区状态，操作系统可以采用两种方法跟踪内存的使用情况：
+ **位图**，将内存划分为若干字或KB的小的分配单位，每个分配单位对应位图中的一个位，1 表示占用，0 表示空闲。当需要分配 $k$ 个分配单位时，存储管理器须**搜索位图找出 $k$ 个连续的 0**。
+ **空闲链表**，维护一个记录已分配分区和空闲分区的链表，每个表项包含指示标志（空闲区 H / 进程 P）、起始地址、长度和指向下一表项的指针。进程终止或被换出时需要更新链表，通常将相邻空闲区合并。

### 分区分配算法

当有多个空闲分区满足需求时，需要分配算法选择合适的空闲分区。四种顺序搜索算法均需将大于需求的分区分割为两部分，一部分满足需求并标记为「占用」，另一部分为剩余部分标记为「空闲」。

| 算法                   | 策略                           | 特点                                                  |
| :------------------- | :--------------------------- | :-------------------------------------------------- |
| **首次适配 (first-fit)** | 按分区先后次序**从头查找**，选择第一个满足要求的分区 | 时间性能好，较大空闲分区可保留在内存高端，_但_低端积累小碎片后查找开销增大，且低端的外碎片得不到利用 |
| **下次适配 (next-fit)**  | 从上次分配的位置开始查找，到末尾后回到开头        | 分配和释放的时间性能较好，空闲分区分布更均匀，_但_较大的空闲分区不易保留               |
| **最佳适配 (best-fit)**  | 找到大小与需求相差最小的空闲分区             | 可保留较大的空闲分区，_但_随运行时间增长会积累较多小碎片                       |
| **最坏适配 (worst-fit)** | 找到最大的空闲分区                    | 不易产生小碎片，_但_较大的空闲分区不被保留，对后续大作业不利                     |

这四种算法统称为**顺序搜索法**，各有利弊，哪种最好需针对具体作业序列分析。对于某一作业序列，**能将所有作业安置完毕的算法就是合适的**。

### 内存紧缩

动态分区分配中，可能出现空闲分区总容量大于程序需求，但因碎片化而无法容纳的情形。此时，可通过移动已装入的程序，将分散的小分区拼接成大分区，这一操作称为**内存紧缩 (memory compaction)**。因为程序已经装入并运行，内存紧缩需要动态重定位支持，且操作本身十分耗时。

## 伙伴系统

固定分区限制了活跃进程数目且内存利用率低，动态分区使管理复杂化且需付出内存紧缩的额外开销。**伙伴系统 (buddy system)** 综合了两者的优点，通过**不断对分**大的空闲分区来获得所需大小的分区，对分得到的两个部分互称为**伙伴 (buddy)**。空闲分区的分配和合并都使用 $2^k$ 大小。

系统按 $2^0, 2^1, \dots, 2^m$ 的不同大小维护 $m+1$ 个**空闲分区链表**。对于大小为 $n$ 的分配请求，
1. 计算 $k = \lceil \log_2 n \rceil$，从子链表 $k$ 开始查找空闲分区；
2. 若子链表 $k$ 非空，则直接分配；
3. 若子链表 $k$ 为空，则向上查找更大的空闲分区，将其再不断对分直至 $2^{k}$ 分配，剩余部分分别插入对应子链表。

一个分区释放时，检查其伙伴是否空闲，若空闲则合并。对于大小为 $2^k$、地址为 $x$ 的内存块，其伙伴地址为
$$
\text{buddy}_k(x) = \begin{cases}
x + 2^k, & \text{if } x \bmod 2^{k+1} = 0 \\
x - 2^k, & \text{if } x \bmod 2^{k+1} = 2^k
\end{cases}
$$

伙伴系统的优点在于多个空闲链表可立即找到空闲分区，算法简单，速度快。缺点包括：
+ 总是按 $2^k$ 分配导致内碎片，如申请 129 字节分得 256 字节，申请 1025 字节分得 2048 字节；
+ 只归并伙伴而容易产生外碎片；
+ 申请、释放时可能有连锁切块或合并，影响系统效率。


# _3.2_  页式分配存储管理

连续分配要求进程在内存中连续存放，使进程受到可用连续内存容量的限制。为突破这一限制，诞生了**非连续分配（离散分配）技术**，运行时将一个进程的程序和数据离散存储在多个独立的分配单位中。

非连续分配技术包括**页式**、**段式**和**段页式**三种：
+ **页式存储管理**将内存划分为固定大小的页 (page) 框，程序划分为同样大小的页，按页分配内存。每个进程有一个页表，记录逻辑页号与物理页框号的映射关系。
+ **段式存储管理**将程序和数据划分为若干个大小不等的段 (segment)，每个段代表一个功能单元，如代码段、数据段、堆栈段等，内存也划分为同样大小的段，按段分配内存，不同的段在物理内存中不需要连续。
+ **段页式 (segmentation with paging) 存储管理**结合了页式和段式的优点，先将程序划分为若干段独立管理，每段再划分为若干页，物理内存按页分配。

**页式存储管理是现代通用操作系统的主流方案**。现代处理器中，采用段页式存储管理的处理器架构主要是x86，而其他主流架构（如ARM、RISC-V、MIPS）则采用页式存储管理。在x86中，分段也基本被绕过，往往设置基地址为 0、段界限为最大值，等同于页式管理。

> [!note] 分页式存储管理的特点
> 分页式存储管理以**页框**为基本分配单位，用户作业的逻辑地址空间按照页框大小被划分为若干页，这一划分由系统内部自动完成、**对用户完全透明**。逻辑地址空间中的页可以装入内存中的**任何一个空闲页框**，打破了作业必须占据连续存储空间的限制，即使作业存放在不连续的物理存储区中也能正确运行，从根本上解决了连续分配方式下的外碎片问题。
>
> 分页式存储管理的主要局限在于**难以实现真正意义的共享**。被共享的程序文本或数据不一定恰好占据一个或几个完整的页面，同一个页面中可能既有允许共享的内容又有不能共享的私有数据，因此在分页环境下实现页面级别的共享较为困难。

## 简单页式存储管理

简单页式管理是基于**实存 (real memory)** 的页式存储管理。支持页式管理的处理器将物理内存划分为固定大小的**页框 (page frame)**，操作系统据此将进程的逻辑地址空间划分为同样大小的**页 (page)**，以页为单位实现内存的分配和回收，通过**页表**实现逻辑地址到物理地址的转换。页的划分由系统自动完成，对用户透明。逻辑上相邻的页在物理上不一定相邻。

指令给出的逻辑地址分为两部分：**逻辑页号 $P$** 和**页内偏移地址 $d$**。对于逻辑地址 $A$，若页面大小为 $L$ 且 $L$ 为 2 的幂次，则 $P = \left\lfloor \frac{A}{L} \right\rfloor$，$d = A \bmod L$。物理地址同样分为物理页框号和页内偏移地址两部分。

简单页式管理维护三种系统级数据结构：
+ 每个进程有一个**页表**，在逻辑页号与本进程的物理页框号之间建立映射关系。
+ 整个系统有一个**物理页框表**，描述物理内存空间的分配使用状况，可通过位图或空闲页面链表实现。
+ 系统还维护一个**请求表**，记录各进程页表的位置和大小，用于地址转换，也可以结合到各进程的 PCB 里。

### 地址变换

处理器的**页表寄存器 (page-table register, PTR)** 中存放页表在内存的首地址和页表长度，地址变换机构自动将逻辑地址分解为页号和页内地址，然后**以页号为索引**查询页表项
$$
\text{页表项地址} = \text{页表首地址} + \text{页号} \times \text{页表项长度}
$$
找到对应的**物理页框号**，最终计算得到**物理地址**
$$
\text{物理地址} = \text{物理页框号} \times \text{页面大小} + \text{页内地址}
$$
若页号不小于页表长度，则产生越界中断。

#### 快表 (TLB)

页式存储管理中每次内存访问需要两次访存——先访问页表获取页框号，再访问实际数据。为解决这一性能问题，引入一种具有并行查询能力的特殊 Cache，称为**快表 (translation lookaside buffer, TLB)**，存放当前最常访问的页表项。

地址转换时，根据逻辑页号同时查找快表中是否存在对应表项。若命中，直接取出**页框号**加上页内偏移得到物理地址；若缺失，则查找内存中的页表获得页框号，同时将该页表项**更新**到快表中，并计算物理地址。

#### 多级页表

对于大型程序，页表本身可能很大且需要连续存放，可能出现**难以找到一块连续大内存空间存放页表**的问题，需要**将页表本身也分页**，并对页表所占空间进行索引形成外层页表，即**页目录表**、**二级页表**，更进一步可形成多级页表。

二级页表结构将逻辑地址分为三部分：**页目录索引**、页表索引和页内偏移地址。访存时，先由页目录索引访问页目录表找到对应的页表地址，再由页表索引访问二级页表找到对应的物理页框号，最后加上页内偏移地址得到物理地址。

### 页面与页框大小

页框的大小由计算机硬件逻辑定义，通常在 512 B—4 KB 之间，典型大小为 4 KB。将页框大小设置为 2 的幂次可以简化地址变换硬件逻辑的设计与实现。

页面大小影响内碎片、地址转换速度和页面交换效率：
+ 较小的页面可减少内碎片，但会导致页表过大、降低快表命中率；
+ 较大的页面有利于提高内外存之间的数据交换效率，但会增加内碎片。

> [!note] 简单页式存储管理的特点
> 简单页式存储管理的优点主要在于：
> + **没有外碎片**，每个内碎片不超过页大小；
> + 程序**无需连续存放**；
> + **便于改变程序占用空间**，随着程序运行而动态生成的数据增多，地址空间可相应增长。
> 
> 简单页式存储管理的缺点主要在于：
> + 存在内碎片；
> + 作业每次**仍要求一次全部装入**内存，不能运行超过物理内存的大作业；
> + 难于实现真正意义的**共享**，一个页面中可能既有允许共享的内容又有不能共享的私有数据。

## 虚拟页式存储管理

简单页式管理要求程序一次性全部装入内存后才能运行，这导致两个问题：程序特别大时无法一次装入，以及系统中有太多进程时内存容纳不下。解决方法有两种——从物理上扩大内存，或从逻辑上扩充内存。**虚拟存储器**正是后一种方案，其正确性由存储器访问的局部性原理保障。

> [!definition] 虚拟存储器
> 在具有层次结构存储体系的计算机系统中，采用自动实现**部分装入**和**部分交换**功能，为用户提供一个独立于物理内存容量的、可寻址的逻辑存储器。

虚拟存储器以大容量辅存为后备，进程保存在后备存储器上，装入时**只将当前需要的部分**装入内存，其余部分暂存外存，进程执行中需要访问不在内存的信息时再将其调入。虚拟存储器可在较小的物理内存中执行**较大的用户程序**，提供给用户的虚拟空间可大于物理内存，可在内存中容纳更多程序**并发**执行。

虚拟页式存储管理在简单页式基础上增加**调页**和**页面置换**功能。
+ 程序装入时，只需将当前需要执行的部分页读入内存即可开始执行；
+ 执行过程中，若需访问的页面不在内存，由处理器通知操作系统执行缺页中断处理程序将相应页调入；
+ 操作系统也可将内存中暂时不使用的页调出到外存以腾出空间。

为支持这些功能，进程页表需要扩充以下字段：
+ 标志位需添加指示页面是否在内存中的**存在位**（P 位）、指示页面是否被修改过**修改位**（M 位）
+ 访问统计信息
+ 外存地址。

MMU 因而承担更复杂的职责，除管理页表寄存器、分解虚拟地址、访问页表和快表外，还需**设置和检查页表中各特征位**，并在缺页时**发出缺页中断**将控制权交给内核存储管理程序。**缺页中断 (page fault)** 是一种特殊的中断，需要在指令执行**期间**产生和处理中断信号。一条指令执行期间可能产生多次缺页中断，因为指令本身和操作数可能都不在内存中。

### 页面调入策略

页面**调入策略**决定何时将页面装入内存：
+ **请调 (demand paging)** 仅在发生缺页时调入所需页面，实现简单、节省物理内存，但每次仅调一页导致磁盘 I/O 次数较多。
+ **预调 (prepaging)** 在发生缺页前根据预测预先调入页面，可减少缺页中断次数，但若调入的页面以后很少被访问则效率低下，常用于程序装入时的初始调页。

### 页面分配策略

页面**分配策略**决定系统为每个进程分配多少物理页框。需要考虑的权衡因素包括：
+ 分给进程的页框越少，内存中可容纳的进程越多，至少有一个进程处于就绪态的可能性越大；
+ 若进程只有小部分在内存中，即使局部性很好，缺页中断率仍会较高；
+ 因局部性原理，分配的内存超过一定限度后，再增加也不会明显降低缺页率。

页面分配有两种方式：
+ **固定分配**，进程保持页框数不变——进程创建时根据进程类型和程序员要求决定页框数，每当缺页即在本进程页框内置换一页
+ **可变分配**，根据进程的缺页率动态调整页框数——缺页率高说明当前局部性较差，系统可多分页框以降低缺页率，反之可减少分配的页框数

### 页面置换策略

当需要调入新页而内存中无空闲页框时，必须选择一个页面换出。页面置换算法的目标是将未来不再使用或短期内不使用的页面调出，通常只能在局部性原理指导下依据过去的统计数据进行预测。若选用不合适的算法，可能出现刚被调出的页立即又要调入、调入不久又被调出的现象，称为**颠簸**或**抖动 (thrashing)**，使大部分 CPU 时间花费在来回调度上。

#### 固定分配页面置换算法

固定分配页面置换算法在每个进程的固定页框内进行置换，选择一个页面换出以腾出空间调入新页。常用的固定分配页面置换算法如下：

| 算法                                                | 策略                                                                     | 评价                                                      |
| ------------------------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------- |
| **最优 (optimal, OPT) 页面置换**                        | 置换未来最远才会被使用的页面                                                         | 理论最优，但需预知未来访问序列，**无法实现**，仅用作性能比较基准                      |
| **最近未使用 (not recently used, NRU) 页面置换**           | 以引用位 R 和修改位 M 将页面分为未引用未修改、未引用已修改、已引用未修改、已引用已修改，在**按此次序的第一个非空类**中随机淘汰一页 | 实现非常简单，但策略粗糙、性能不稳定                                      |
| **先进先出 (first-in first-out, FIFO) 页面置换**          | 维护页面进入顺序，选择在内存中驻留时间最长的页面置换                                             | 实现简单，但较早调入的页往往是经常被访问的页；存在 **Belady 异常**，即分配页框数增加反而缺页率升高 |
| **第二次机会页面置换**，是对 FIFO 的改进                         | 检查最老页面的 R 位，若 R=0 则淘汰该页，若 R=1 则清除 R 位并将其移至链表末尾                         | 对 FIFO 的重大改进，避免频繁访问页面被反复置换                              |
| **时钟 (clock) 页面置换**                               | 将所有页面组织为循环链表，指针依次扫描各页并检查 R 位，R=0 时淘汰，R=1 时清除 R 位并继续扫描                  | 第二次机会的逻辑等价实现，实现高效实用，是实际系统中最常用的算法之一                      |
| **最近最少使用 (least recently used, LRU) 页面置换**        | 选择最近最久未被访问的页面置换，是局部性原理的合理近似                                            | 性能接近最优，但需硬件记录页面访问时序，硬件开销较高                              |
| **最不常用 (not frequently used, NFU) 页面置换**，是对LRU的模拟 | 每个页面关联一个计数器，每次时钟中断时将 R 位累加到计数器，选择计数最小的页面置换                             | LRU 的粗略近似，但计数器不衰减导致无法区分近期与远期访问                          |
| **老化 (aging) 算法**，是对NFU 的改进                       | 每次累加前先将计数器右移一位，再将 R 位加到最左端，赋予近期访问更高的权重                                 | LRU 的有效近似，兼顾了历史信息与近期访问趋势                                |

> [!danger] Belady 异常
> Belady 异常是指 FIFO 算法中，**分配的页框数增多但缺页率反而升高**的异常现象。LRU、Clock 等基于访问历史的算法不会出现此异常。例如，页面访问序列 0 1 2 3 0 1 4 0 1 2 3 4：分配 3 个页框时缺页 9 次，分配 4 个页框时缺页反而升至 10 次。

#### 工作集模型

进程在一段时间内总是集中访问一些页面，称为**活跃页面**。如果分配给进程的物理页框数太少，导致**活跃页面不能全部装入内存**，进程将频繁发生缺页中断。若能提供与活跃页面数相等的物理页框数，则可有效减少缺页中断次数。

**工作集 (working set)** 是一个进程当前正在使用的页面的集合。
+ 在访存次数意义下，工作集表示为 $W(k, t)$，定义为进程在最近 $k$ 次内存访问中所访问的页面集合。
+ 近似地，在CPU时间意义下，工作集表示为 $W(t, \tau)$，定义为进程在其CPU时间段 $[t - \tau, t]$ 内所访问的页面集合。

进程开始执行后，随访问新页面逐步建立较稳定的工作集，工作集大小随时间变化而变化，反映了程序的局部性特征：
+ **局部性区域稳定**时，**工作集大小大致稳定**；
+ 局部性区域改变时，工作集会快速扩张和收缩过渡到下一个稳定值。

采用工作集策略的困难在于操作系统需要持续监视活动进程的行为和缺页中断率，会增加系统开销。

### 页面清除策略

**页面清除策略**决定何时将修改过的页面**写回**辅助存储器：
+ **请式清除 (demand cleaning)** 仅当页面被选中替换且之前被修改过时才写回，但写回与调入成双操作，导致进程等待两次 I/O
+ **预清除 (precleaning)** 将页面提前写出但仍保留在内存中直到被替换，可成批写出减少 I/O，但若写出后页面又被修改则前功尽弃
+ **页缓冲 (page buffering)** 使清除和替换不必成双进行——替换的页面进入修改队列和非修改队列，修改队列的页面周期性地成批写出并移入非修改队列，非修改队列中的页面被再次引用时回收或分配给其他页面

> [!note] 简单页式 vs 虚拟页式
> ```tx
> | 特性 | 简单页式 | 虚拟页式 |
> |:-----|:---------|:---------|
> | 磁盘使用 | 不使用磁盘。进程所有页面在加载时全部装入物理内存，执行期间始终驻留内存，无需后备存储。 | 使用磁盘作为后备存储。程序保存在磁盘上，仅当前需要访问的页面装入内存，暂时不用的页面可换出到磁盘。 |
> | 缺页中断 | 无。进程的全部页面已在内存中，任何逻辑地址访问均能直接映射到物理页框，不存在「访问的页面不在内存」的情况。 | 有。当 CPU 访问的页面不在物理内存中时，MMU 产生缺页中断，操作系统随之将所需页面从磁盘调入内存，然后进程继续执行。 |
> | 页面置换 | 无。若进程所需内存超过空闲页框数，则进程无法装入，必须等待足够空闲页框出现。 | 有。当发生缺页中断且内存无空闲页框时，操作系统根据页面置换算法（如 LRU、FIFO、Clock 等）选择一个页面换出到磁盘，腾出空间后调入所需页面。 |
> | 内存限制 | 进程的整个逻辑地址空间必须全部装入物理内存，每个进程的大小受物理内存容量的严格限制。 | 进程的虚拟地址空间可以远大于物理内存容量，虚拟地址空间的大小由处理器地址结构决定（如 32 位系统为 4GB），与物理内存大小无直接关系。 |
> | 适用场景 | 嵌入式系统、实时系统。这些系统对响应时间有严格要求，不能容忍缺页中断和磁盘 I/O 的不可预测延迟；且系统功能相对单一，内存需求可预先确定。 | 现代通用操作系统（如 Windows、Linux、macOS）。需同时运行大量复杂应用程序，必须支持大地址空间和灵活的内存管理。 |
> ```


# _4_  设备管理

## 设备管理概述

控制计算机的所有 I/O 设备是操作系统的主要功能之一。**I/O 子系统**是操作系统中负责管理输入输出设备的部分，它负责控制所有输入输出设备以完成期望的数据传送，实现操作系统的设备管理功能。I/O 子系统的核心设计目标是在设备和系统的其他部分之间提供简单且易于使用的接口，并且该接口对于所有设备应该是相同的，这就是[[#^device-independence|设备无关性（设备独立性）]]。

### 设备管理的特点与任务

设备管理面临一系列独特的挑战：
+ CPU 与 I/O 设备的**速度差别极大**，I/O 设备在速度上覆盖了巨大的范围，从键盘的每秒几个字节到高速网络的每秒数 GB，跨越数个数量级。要使软件在跨越这么多数量级的数据率下保证性能优良，给操作系统的 I/O 管理造成了相当大的压力，I/O 的性能经常成为**系统性能的瓶颈**。
+ 操作系统庞大复杂的重要原因之一——**资源多、杂，并发**，均来自 I/O。

理解 I/O 的工作过程与结构是理解操作系统工作过程与结构的关键。此外，I/O 管理与其他功能联系密切，特别是文件系统。

设备管理的具体任务包括：
+ 按照用户的请求控制设备的各种操作，完成 I/O 设备与内存之间的**数据交换**，包括设备分配与回收、设备驱动程序、设备中断处理、缓冲区管理等，最终完成用户的 I/O 请求。
+ 向用户提供使用外部设备的方便**接口**，使用户摆脱繁琐的编程负担。
+ 充分利用各种技术（通道、中断、缓冲等）提高 CPU 与设备、设备与设备之间的**并行工作**能力，充分利用资源，提高资源利用率。
+ 保证在多道程序环境下，当多个进程竞争使用设备时，按一定策略**分配和管理**各种设备，使系统能有条不紊地工作。
+ 保护设备传送或管理的**数据安全**，使之不被破坏、不被泄露。
+ 实现[[#^device-independence|设备无关性]]。用户在编制程序时使用逻辑设备名，由系统实现从逻辑设备到物理设备（实际设备）的转换，使**用户能独立于具体物理设备**而方便地使用设备。

> [!note] 设备独立性 ^device-independence
> 
> **设备独立性 (device independence)** 使用户或程序员可以访问任何 I/O 设备而无需事先指定设备。例如，命令 `sort <input >output` 无论输入来自软盘、硬盘或者键盘，输出送往任意类型的磁盘或屏幕，都可以正常工作。尽管这些设备的差别很大，其处理由操作系统负责。
> 
> 实现设备独立性的关键技术之一是**统一命名**，对输入输出设备采用**与文件统一**的命名方式。所有文件和设备都使用相同的方式：通过**路径名**进行寻址。例如在 UNIX 系统中，像 `/dev/tty00` 这样的文件名就代表一个设备。

## I/O 硬件原理

### I/O 设备的分类

I/O 设备可以从多个角度进行分类。
+ **按数据组织分类**，可分为
    + **块设备**，以数据块为单位存储、传输信息。
    + **字符设备**，以字符为单位存储、传输信息。
    + 有些设备既不是块设备也不是字符设备，例如时钟。
+ **按功能特性分类**，设备可分为存储型设备、输入输出型设备（交互型设备）和数据通信设备。
+ **从资源分配角度分类**，可分为
    + **独占设备**，在一段时间内只能有一个进程使用，一般为低速 I/O 设备，如打印机、磁带等。
    + **共享设备**，在一段时间内可有多个进程共同使用，多个进程以交叉的方式来使用设备，其资源利用率高，如硬盘。

### 设备控制器

I/O 设备一般由**机械部分**和**电子部分**两部分组成。机械部分是设备本身（物理装置），电子部分叫做**设备控制器**或适配器。将这两部分分开处理，以提供更加模块化、更加通用的设计。

设备控制器实现设备与主机间的连接和通信。设备控制器与主机之间通过标准总线相连接，而设备控制器与设备之间的接口一般是私有的。控制器通过其中的若干接口寄存器或接口缓冲区与 CPU 通信。

设备控制器的**功能**可以概括为：
+ 设备端口地址译码；
+ 按照主机与设备之间约定的格式和过程，接受计算机发来的数据和控制信号，并向主机发送数据和状态信号；
+ 将计算机的数字信号转换成机械部分能识别的模拟信号，或反之；
+ 实现设备内部硬件缓冲、数据加工等提高性能或增强功能。

#### 设备接口

每个设备控制器有几个寄存器用来与 CPU 进行通信，**控制寄存器**接收命令，**状态寄存器**报告状态，**数据缓冲区**暂存数据。操作系统将命令写入控制寄存器（或数据缓冲区）中，以实现输入/输出，并从状态寄存器（或数据缓冲区）读取状态信息或结果信息。

CPU使用设备控制器的过程如下：
+ 当控制器接受一条命令后，可独立于 CPU 完成指定操作，CPU 可以转去执行其他运算。
+ 命令完成时，控制器产生一个中断，CPU 响应中断，控制转给操作系统。
+ 操作系统通过读设备控制器的状态寄存器中的信息，获得操作结果和设备状态。

控制器与设备之间的接口常常是一个**低级接口**，控制器的核心任务就是将这一低级接口的细节屏蔽在硬件层，向 CPU 呈现一个干净的**寄存器**界面。控制器的任务是把串行的位流转换为字节块，并进行必要的错误修正：
+ 按位进行组装，存入控制器内部的缓冲区中，形成以字节为单位的块；
+ 在对块验证检查和并证明无错误时，再将它复制到主存中。

#### 端口编址

设备控制器中有多个寄存器（称为 I/O 端口），每个寄存器有唯一的地址，该地址称为 I/O 端口地址。I/O 端口地址主要有两种形式，与 I/O 指令的形式相关联：
+ **I/O 独立编址**（I/O 专用指令）中，每个**寄存器**被分配一个 I/O 端口号。分配给系统中所有端口的地址空间是完全独立的，与内存的地址空间没有关系。主机使用专门的 I/O 指令对端口进行操作。例如，8086/8088 分配给 I/O 端口的地址空间为 0000H–0FFFFH（64K），只能用 IN 和 OUT 指令对其进行读写操作。
+ **内存映射编址**（内存映射 I/O 模式）中，分配给系统中所有端口的地址空间与内存的地址空间统一编址。主机把 I/O 端口看作一个存储单元，**对 I/O 的读写操作等同于对存储器的操作**。

大部分处理器采用**内存映射 I/O**，优点在于：
+ C/C++ 不存在执行 IN/OUT 指令的方法，所以独立编址 I/O 访问设备寄存器只能采用汇编代码；而对于内存映射 I/O，设备寄存器只是内存中的变量，可以和其他变量一样寻址。
+ 不需要特殊的保护机制来阻止用户进程执行 I/O，操作系统要做的只是避免将包含设备寄存器的那部分地址映射到任何用户的虚拟地址空间之中。
+ 可以访问内存的每条指令也可以访问设备寄存器。例如，如果存在 TEST 指令可以测试一个内存字是否为 0，则也可用来测试某个设备状态寄存器是否为 0。

但内存映射 I/O 也有两个主要缺点：
+ **高速缓存问题**：缓存一个设备寄存器可能是灾难性的。如果设备状态寄存器被缓存，CPU 将反复读到缓存中的旧值而非设备的实际状态。为避免这一问题，硬件必须针对每个页面具备选择性禁止高速缓存的能力。
+ **总线检查开销**：只存在单一内存空间，所以所有内存模块和设备都必须检查所有内存访问。采用单总线结构是自然的，但是影响系统性能。为提高性能，现代计算机系统多采用多总线结构，必须采取特殊的措施使内存映射 I/O 工作在多总线系统上。

PC 机实际采用**混合方案**，具有内存映射 I/O 的数据缓冲区，而控制/状态寄存器则具有单独的 I/O 端口。

### I/O 控制方式

#### 程序控制 I/O

**程序控制 I/O (programmed I/O, PIO)**，也称为直接 I/O 或轮询，由 CPU 负责在机器内存与设备控制器数据寄存器之间进行数据传送。其缺点是直到全部 I/O 完成之前要占用 CPU 的全部时间，CPU 循环查询设备状态，形成忙等待。

#### 中断驱动 I/O

**中断驱动 I/O** 旨在减少程序控制 I/O 中 CPU 的等待时间，提高系统的并行工作程度。为了使 CPU 能够处理中断，中断控制器向 CPU 发出中断信号，同时在**地址总线**上放置一个数字表明哪个设备需要关注。中断信号导致 CPU 暂停当前工作，地址总线上的数字被用作指向**中断向量表**的索引，中断向量表中存放**中断服务程序**的入口地址。

在中断服务程序开始前，硬件必须保存一定的信息。最低限度要保存被中断进程的**程序计数器**的值，另一个极端是保存**所有寄存器**的值。
+ 信息可以保存在 CPU 内部的寄存器，此时中断无法嵌套；
+ 也可以保存在当前用户进程栈或内核栈中。

中断驱动 I/O 的缺点是中断发生在每个字符上，浪费一定数量的 CPU 时间。

> [!note] 流水线和超标量对中断处理的影响
> 在流水线满的时候，当中断出现时，程序计数器的值可能无法正确反映已经执行过的指令和尚未执行的指令之间的边界。在超标量计算机上，因为指令可能乱序执行，所以在已经执行过的指令和尚未执行的指令之间可能并不存在明确的边界。

#### 直接存储器存取 (DMA)

**直接存储器存取 (direct memory access, DMA)** 实现数据在内存与 I/O 设备间的直接成块传送。CPU 在开始时向设备发「传送一块」命令，结束时进行相应处理，实际操作由 DMA 控制器硬件直接完成。DMA 的功能可以以独立的 DMA 部件在系统 I/O 总线上完成，也可整合到 I/O 部件中完成。

DMA 控制器能够通过系统总线**代替 CPU 管理**数据的存入或取出。当 CPU 不需要系统总线时可以使用总线；DMA 也可以强迫 CPU 暂时延迟其他操作，获取一个总线周期，这一机制称为**周期窃取 (cycle stealing)**。

没有 DMA 时，磁盘读取需要 CPU 全程参与：
1. 控制器从磁盘驱动器串行地一位一位地读一个块，直到将整块信息放入控制器的内部缓冲区中；
2. 做和校验计算，以核实没有读错误发生；
3. 控制器产生一个中断，CPU 响应中断，控制转给操作系统；
4. 当操作系统开始运行时，它重复地从控制器缓冲区中一次一个字节或一个字地读这个磁盘块的信息，并将其送入内存中。

采用 DMA 方式时，除向控制器提供要读块的磁盘地址外，还要向控制器提供两个信息：要读块送往内存的**起始地址**和要传送的**字节数**。DMA 控制磁盘与存储器之间数据传送的过程如下：
1. 每当磁盘把一个数据读入控制器的数据缓冲区，
    1. DMA 控制器取代 CPU，接管地址总线的控制权；
    2. DMA 控制器按照其内部的**存储器地址寄存器**内容，把数据送入相应的**内存**单元；
    3. DMA 硬件自动将**传送字节计数器**减 1，将**存储器地址寄存器**加 1，然后恢复 CPU 对内存的控制权；
2. 对每一个传送的数据重复上述过程，直到传送字节计数器为 0；
3. 传送字节计数器归零时，向 CPU 产生一个**中断**信号。

当操作系统接管 CPU 控制权时，无需再做块复制的工作。

> [!note] DMA 方式与中断方式的关键区别
> 中断方式是在数据缓冲寄存器满后发中断请求，CPU 进行中断处理，**数据传送由 CPU 控制完成**。DMA 方式则是在所要求传送的数据块全部传送结束时才要求 CPU 进行中断处理，**数据传送在 DMA 控制器的控制下**，不经过 CPU 控制完成。DMA 方式大大减少了 CPU 进行中断处理的次数。

#### 通道

通道又称为输入输出处理器、I/O 处理机，能完成主存和外围设备间的信息传送，与 CPU 并行地执行操作。通道的工作特点在于：
+ 实现外围设备和 CPU 能并行操作
+ 通道和通道之间能并行操作
+ 各通道上的外围设备并行操作

通道技术解决了 **I/O 操作的独立性**和**各部件工作的并行性**，把 CPU 从琐碎的 I/O 操作中解放出来，是计算机技术的一个重要进步。

采用通道后的 I/O 操作过程分为：
1. 启动 I/O 操作时，CPU 在执行主程序时遇到 I/O 请求，它启动指定通道上选址的外围设备，
2. 一旦启动成功，通道开始控制外围设备进行操作，之后 CPU 就可执行其他任务并与通道并行工作。
3. 结束 I/O 操作时，I/O 通道完成操作后发出操作结束中断，CPU 才停止当前工作，转向处理 I/O 操作结束事件。

按照信息交换方式和连接设备的不同，通道可以分为三种类型：
+ **字节多路通道**：同时为多个低速 I/O 设备（软盘、纸带、磁带机等）服务。
+ **选择通道**：分时地为各个高速 I/O 设备（磁盘、磁带机等）服务。
+ **数组多路通道**：同时地为多个像磁盘机（移臂时间长）一样的 I/O 设备服务。

## I/O 软件技术

### 错误处理

错误应该在**尽可能靠近硬件**的层面得到处理。大多数错误是与设备密切相关的，因此只有驱动程序知道应如何处理。在很多情况下，错误处理可以在低层透明地得到解决，高层软件甚至不知道存在这一错误。

但还有一些典型的错误不是输入输出设备的错误造成的，如由于磁盘块受损而不能再读，驱动程序将尝试重读一定次数，若仍有错误，则放弃重读并通知与上层与设备无关的软件。

### I/O传输

I/O 传输根据是否需要等待 I/O 操作完成可以分为**同步传输**和**异步传输**两种方式：
+ **同步传输**要求发出 I/O 请求的进程必须等待 I/O 操作完成后才能继续执行，I/O 操作期间进程处于**阻塞**状态。
+ **异步传输**允许发出 I/O 请求的进程在 I/O 操作完成之前继续执行，是**非阻塞**的。I/O 操作完成后，系统会通过中断或其他机制通知进程。

大多数物理I/O是异步的，但是阻塞式I/O使用户程序更加容易编写，因此操作系统使实际上是中断驱动的I/O变为在用户看来是阻塞式的操作。

### 缓冲技术

CPU 和 I/O 设备的速度差异是明显的，CPU 的速度以纳秒计，外部设备的处理速度则一般以毫秒甚至秒计。虽然操作系统为各部分并行工作提供了可能，但是由于 CPU 和外部设备速度不匹配，它们的并行程度并不能得到充分发挥。因此必须引入缓冲技术。

以**行式打印机**为例说明缓冲的作用。假设在一段时间内，系统中只有一个用户进程正在使用行式打印机。其工作过程是首先计算并产生一行需打印的信息，存放在字符型数组 `line[LINELENGTH]` 中，然后要求系统将 `line` 数组中的信息送行式打印机打印。
+ **无缓冲**时，为了使用行式打印机，处理机调度操作比较频繁，增加了系统开销，同时打印机的**忙闲程度也很不均匀**，不能充分实施并行操作。
+ **单缓冲**时，在系统中为行式打印机设置一个缓冲存储区 `buffer[LINELENGTH]`。进程使用行式打印机的过程变成，系统先将 `line` 数组中的信息传送到 `buffer` 中，然后命令行式打印机将 `buffer` 中的内容打印出来。引入单缓冲后，进程可以在打印机打印前一行数据的同时计算下一行数据，实现了计算与打印的部分重叠。
+ 为了进一步提高行式打印机的利用程度并减少进程调度次数，可以继续增加缓存数，形成**多缓冲**（双缓冲、循环缓冲、缓冲池）。多缓冲允许计算与打印的重叠程度更高，进一步提高了并行性。

引入缓冲的主要原因可以归结为：
+ 匹配 CPU 与 I/O 设备的速度差异；
+ 减少对设备的中断频率；
+ 提高 CPU 与 I/O 设备间的并行性。

### 虚拟设备

系统中的独占设备是有限的，往往不能满足诸多进程的要求，因而会引起大量进程由于等待某些独占设备而阻塞，成为系统中的「瓶颈」，且隐含着**死锁**的必要条件。另一方面，申请到独占设备的进程在其整个运行期间虽然占有设备，利用率却常常很低，设备还是经常处于空闲状态。

为了解决这种矛盾，最常用的方法就是**用共享设备来模拟独占设备的操作**，从而提高系统效率和设备利用率。实现这一技术的软、硬件系统被称为同时外围联机操作 (simultaneous peripheral operation on line) 系统，简称 **SPOOLing 系统**，又称**假脱机**。

> [!note] SPOOLing 系统的组成
> + **输入井/输出井**：磁盘上开辟的缓冲区，用于暂存输入/输出数据
> + **输入进程/输出进程**：负责在井和设备之间传递数据
> + **井管理程序**：负责管理井空间和队列

以打印机这一典型的独占设备为例，引入 SPOOLing 技术后，用户的打印请求传递给 SPOOLing 系统，而并不是真正把打印机分配给用户。SPOOLing 系统处理打印请求的过程如下：
1. SPOOLing 系统的输出进程在磁盘上申请一个空闲区，把需要打印的数据传送到里面；
2. 把用户的打印请求挂到打印队列上；
3. 如果打印机空闲，就会从打印队列中取出一个请求，再从磁盘上的指定区域取出数据，执行打印操作。

由于磁盘是共享的，SPOOLing 系统可以随时响应打印请求并把数据缓存起来，这样就把独占设备改造成了共享设备，从而提高了设备的利用率和系统效率。


## I/O 软件层次

I/O 软件的基本思想是按分层的思想构成。较低层软件要使较高层软件独立于硬件的特性，较高层软件则要向用户提供一个友好的、清晰的、简单的、功能更强的接口。

I/O 软件由低到高通常分为以下各层：
+ **硬件层**，实现物理 I/O 的操作。
+ **中断处理程序**，负责在 I/O 完成时唤醒设备驱动程序进程，进行中断处理。
+ **设备驱动程序**，设置设备寄存器、检查设备的执行状态。
+ **与设备无关的 I/O 软件**，实现设备的命名、设备的保护、成块处理、缓冲技术和设备分配。
+ **用户进程层**，执行输入输出系统调用，对 I/O 数据进行格式化，为假脱机输入/输出做准备。

### 中断处理程序

中断应该深深隐藏在操作系统内部。隐藏的办法是每个进程在启动一个 I/O 操作后阻塞，直到 I/O 操作完成并产生一个中断由操作系统接管 CPU 后唤醒该进程为止。

驱动程序阻塞自己的手段可以是在**信号量**上执行 P 操作、在**条件变量**上执行 wait 操作、或在**消息**上执行 receive 操作。中断发生时，中断处理程序将对中断进行处理，然后将被阻塞的驱动程序解除阻塞，对应的手段是在信号量上执行 V 操作、在条件变量上执行 signal 操作、或在消息上执行 send 操作。

### 设备驱动程序

与设备密切相关的代码放在设备驱动程序中，每个设备驱动程序处理一种设备类型。每一个控制器都设有一个或多个设备寄存器，用来存放向设备发送的命令和参数。设备驱动程序负责释放这些命令，并监督它们正确执行。一般而言，设备驱动程序的任务是接收来自与设备无关的上层软件的抽象请求，并执行这个请求。

在设备驱动程序的进程发出一条或多条命令后，系统有两种处理方式：
+ 多数情况下，执行设备驱动程序的进程必须等待命令完成，即在命令开始执行后它**阻塞自己**，直到中断处理时将它解除阻塞为止。
+ 另一些情况下，命令执行不必延迟就很快完成。

### 与设备无关的 I/O 软件

虽然 I/O 软件中一部分是设备特定的，但大部分软件是与设备无关的。设备驱动程序和与设备无关的软件之间的确切界限依赖于具体系统。

与设备无关的 I/O 软件承担以下功能：
+ **设备驱动程序的统一接口**：为上层提供一个一致的调用界面，屏蔽不同设备驱动程序的差异。
+ **设备命名**：对输入输出设备采用与文件统一的命名方式。区分这些命名同文件一样的输入输出设备的是与设备无关的软件，它负责把设备的符号名映射到相应的设备驱动程序上。例如，在 UNIX 系统中，像 `/dev/disk0` 这样的文件名唯一确定了一个特殊文件的 i 节点，这个 i 节点包含了**主设备号**和**次设备号**。主设备号用于寻找对应的设备驱动程序，次设备号提供了设备驱动程序的有关参数，用来确定要读写的具体设备。
+ **缓冲**：为块设备和字符设备提供缓冲管理。
+ **错误报告**：许多错误是设备特定的，必须由驱动程序来处理，但是错误处理的框架应该是与设备无关的。编程错误应返回错误代码给调用者；实际 I/O 错误由驱动程序解决，如果驱动程序解决不了问题，则将问题向上传递给与设备无关的 I/O 软件。
+ **分配与释放专用设备**：分配时根据申请的设备类型查询系统设备表找到对应入口，执行 P 操作获取互斥，再查对应的设备表找一空闲设备并分配；释放时根据设备类型查询系统设备表找到对应入口，查对应的设备表找到要释放的设备去配，再执行 V 操作释放互斥。
+ **提供与设备无关的块大小**：向上层屏蔽不同设备物理块大小的差异。

### 用户空间的 I/O 软件

尽管大部分 I/O 软件都包含在操作系统中，但仍有一小部分是由与用户程序连接在一起的库过程，甚至完全由运行于内核外的程序构成。系统调用，包括 I/O 系统调用，通常由库过程实现。这些过程所做的工作只是将系统调用时所用的参数放在合适的位置，由其他的 I/O 过程实际实现真正的操作。

## 典型I/O设备

### 时钟设备

**时钟 (clock)** 又称为**定时器 (timer)**，负责维护时间、防止一个进程长期占用 CPU 等功能。时钟既不是块设备也不是字符设备。

可编程时钟在硬件上由以下部分组成：
+ **晶体振荡器**产生精确的周期信号；
+ **计数器**接收时钟信号并递减计数，当计数器变为 0 时，产生一个 CPU 中断信号；
+ **存储寄存器**存放计数的初始值，是「可编程」的关键所在，CPU 可以通过写入存储寄存器来设置时钟的频率。

时钟有两种操作模式：
+ **一次完成模式 (one-shot mode)**：当时钟启动时，它把存储寄存器的值拷贝到计数器中，然后晶体的每一个脉冲使计数器减 1。当计数器为 0 时产生一个中断并停止工作，直到软件再一次显式启动它。
+ **方波模式 (square-wave mode)**：当计数器为 0 并产生中断时，存储寄存器的值自动拷贝到计数器，这个过程不断地重复下去。周期性的中断称为**时钟滴答 (clock tick)**。

可编程时钟的优点是其中断频率可由软件控制。

除开每隔一定的时间间隔产生一个中断外，涉及时间的其他所有工作都必须由软件（时钟驱动程序）完成。时钟软件的功能包括：
+ 维护时间；
+ 防止进程超时运行；
+ 对 CPU 的使用情况记帐；
+ 处理用户进程提出的 `alarm` 系统调用；
+ 为系统本身各部分提供监视定时器 (watchdog timer)；
+ 完成监视和统计信息收集。

### 人机交互设备

人机交互设备是人和计算机之间建立联系、交流信息的输入输出设备，分为输入设备和输出设备。输入设备用于把原始数据和处理这些数据的应用程序输入到计算机中，典型的输入设备有键盘、鼠标、扫描仪等。输出设备用于将各种计算结果以数字、字符、图形、图像、声音等形式表示出来，典型的输出设备有显示器、打印机等。

#### 键盘

**键盘**的历史非常悠久。1868 年，美国人 Christopher Latham Sholes 设计了现代打字机的规范键盘，即 QWERTY 键盘。键盘根据按键的接触方式可分为机械式键盘、薄膜式键盘、导电橡胶式键盘和无接点静电电容式键盘。

键盘的按键按 M 行和 N 列排列，可排 $M \times N$ 个按键，按键识别只需 $M + N$ 条线。键盘内部的控制器采用扫描法来识别按键的位置，如果有闭合键就根据其位置（行值和列值）获得对应的扫描码：闭合时产生闭合扫描码，松开时产生断开扫描码。键盘控制器收到扫描码后，向 CPU 发出键盘中断请求，由 CPU 响应键盘中断而调入扫描码，并将其转换为按键编码（ASCII 字符码），保存到键盘缓冲区中，以便程序使用。

#### 鼠标

**鼠标**诞生于 1968 年的美国斯坦福大学，发明者是 Douglas Engelbart，设计初衷是为了快速地屏幕定位，使计算机的操作更加简便。

光电式鼠标通常由发光二极管、光学透镜、光学感应及图像分析芯片、接口控制芯片、轻触式按键、滚轮、外壳等部分组成。其工作原理为：光电鼠标通过发光二极管发出光线照亮鼠标底部表面，然后将鼠标底部表面反射回的一部分光线经过一组光学透镜传输到光学传感器件内成像。当鼠标移动时，其移动轨迹会被记录为一组高速拍摄的连续图像。然后利用专用图像分析芯片对其进行处理，通过对这些图像上特征点位置的变化进行分析，给出鼠标的移动方向和移动距离，最后将这些信息传输给接口控制芯片，接口控制芯片通过接口向主机传送鼠标移动的信息，主机再通过处理使屏幕上的光标与鼠标同步移动。

#### 显示器

**显卡与显示器**协同工作。显卡又称为显示适配器，工作在主机与显示器之间，负责把 CPU 送来的图像数据处理成显示器接收的格式，再送到显示器形成图像。

显卡的工作过程为：首先由 CPU 将需要处理的数据通过 PCIe 总线传送到显卡的显存中，GPU 提取存储在显存中的数据进行处理。GPU 内部由**管线 (pipeline)** 构成：
+ **顶点管线**，主要负责 3D 建模；
+ **像素管线**，主要负责 3D 渲染。

当 GPU 处理完毕后，数字图像数据会被送入 RAMDAC 转换成显示器需要的模拟数据（数字信号接口不需此转换），由显示输出接口传输给显示器，成为人们所看到的图像。主流显卡输出接口包括 VGA、DVI、HDMI 和 DisplayPort 接口。

LCD 显示器利用液晶的光电效应：液晶分子的排列在电场作用下发生变化，影响其液晶单元的透光率或反射率，从而影响它的光学性质，产生具有不同灰度层次及颜色的图像。

### 辅助存储设备

#### 硬盘 (HDD)

硬盘是磁盘的一种具体形式，是由覆盖有铁磁性材料的金属或玻璃盘片、磁头及密封外壳组成的固定存储设备，具有大容量和较高的读写速度。尽管**硬盘驱动器 (hard disk drive，HDD)** 和**硬盘 (hard disk)** 是两个概念，但由于两者通常被封装在一起，所以无论是硬盘还是硬盘驱动器通常都指二者封装在一起所形成的设备。

硬盘的物理结构为：
+ 硬盘由许多盘片组成，每个盘片有两个盘面
+ 每个盘面上有一个读写磁头，盘面号即磁头号，所有磁头在磁头臂的作用下同时移动，磁头从 0 开始编号
+ 每个盘面被划分成许多同心圆，这些同心圆称为**磁道**，磁道从外向内从 0 开始顺序编号
+ 磁道被分成许多**扇区**，每个扇区包含相同的字节数，典型值是 512 字节，**扇区从 1 开始编号**
+ 所有盘面上的同一磁道构成一个圆柱，称作**柱面**

硬盘操作**按柱面优先**，而不是按盘面优先。写入数据时，首先确定柱面，即所有磁头一起定位到指定柱面。当前柱面的当前磁道写满后，再在当前柱面的下一个磁道写入，只有当前柱面全部写满后，才能将磁头移动到下一个柱面。对硬盘进行分区时，各个分区也是以柱面为单位划分，**不存在一个柱面属于多个分区**。

磁盘有两种寻址方式：
+ **CHS (cylinder-head-sector)** 使用柱面 $C$、磁头 $H$、扇区 $S$ 三个参数构成的三元组 $(C,H,S)$ 来定位一个扇区。柱面参数确定磁头臂移动到某个磁道，磁头参数确定是哪个盘面上的磁道，扇区参数确定该磁道上某个扇区的具体位置
+ **逻辑块寻址 (Logical Block Addressing, LBA** 将 CHS 三元组转换为**一维**的线性地址，0 柱面 0 磁头 1 扇区编址为逻辑 0 扇区，0 柱面 0 磁头 2 扇区编址为逻辑 1 扇区，以此类推

CHS 与 LBA 的转换按照如下公式进行：
$$
\text{LBA} = (C \times \text{HPC} + H) \times \text{SPT} + S - 1, \qquad
\begin{cases}
C = \text{LBA} \,/\, (\text{HPC} \times \text{SPT}) \\
H = (\text{LBA} \,/\, \text{SPT}) \mathbin{\%} \text{HPC} \\
S = \text{LBA} \mathbin{\%} \text{SPT} + 1
\end{cases}
$$
其中 HPC 为每柱面磁头数，SPT 为每磁道扇区数，$/$ 为整数除法，$\%$ 为取余。

磁盘 I/O 访问时间 $T_{\text{access}}$ 由三部分组成：
$$
T_{\text{access}} = T_s + T_r + T_t
$$
其中，
+ $T_s$ 为**寻道时间**，即磁头移动到指定柱面的机械运动时间，通常每个柱面移动需要 10ms 左右，这是访问时间中的主要部分
+ $T_r$ 为**旋转延迟时间**，即等待指定扇区旋转到磁头下的机械运动时间，与磁盘转速相关，例如某硬盘转速为 7200 rpm，则平均 $T_r = 4.17\,\text{ms}$
+ $T_t$ 为**实际数据传输时间**，即数据读写的时间，与读写的字节数和磁盘转速有关

> [!note] 磁盘臂调度算法
> 
> 来自不同进程的磁盘 I/O 请求构成一个随机分布的请求队列。由于寻道时间在访问时间中占主要部分，磁盘臂调度的主要目标就是减少请求队列对应的平均寻道时间。
> 
> + **先来先服务**（FCFS，First Come First Served）按磁盘 I/O 请求的先后顺序执行。该算法的特点是公平，但是性能比较差。
> + **最短寻道优先**（SSTF，Shortest Seek Time First）考虑磁盘 I/O 请求队列中各请求的磁头定位位置，选择从当前磁头位置出发移动最少的磁盘 I/O 请求。该算法的目标是使每次磁头移动时间最少，但对中间的磁道有利，可能会有进程处于饥饿状态。
> + **电梯算法**（SCAN）选择在磁头前进方向上从当前位置移动最少的磁盘 I/O 请求执行，没有前进方向上的请求时才改变方向。该算法是对 SSTF 的改进，磁盘 I/O 性能较好，且没有进程会饿死，但对中间磁道有利。
> + **单向扫描算法**（C-SCAN）可改进电梯算法对中间磁道的偏好。它在一个方向上使用电梯算法，当到达边沿时直接移动到另一沿的第一个位置。实验表明，该算法在中负载或重负载时磁盘 I/O 性能比电梯算法好。
> 

合理组织磁盘数据上的存储位置可显著提高磁盘 I/O 性能。例如，某磁盘平均柱面寻道时间为 10ms，磁盘转速为 10000 rpm，每个磁道有 32 个扇区，每个扇区 512 字节。现读取包含 256 个扇区的文件：
+ 若文件采用顺序存储：$10 + (3 + 6) \times 8 = 82\,\text{ms}$
+ 若文件由 256 个随机分布的扇区构成：$(10 + 3 + 6/32) \times 256 = 3376\,\text{ms}$

随机存储的访问时间约为顺序存储的 **41 倍**。

#### 固态硬盘（SSD）

**SSD** 是固态驱动器（Solid State Drive）的缩写，通常称为固态硬盘。SSD 的三个核心组件为：
+ **NAND 闪存**：负责数据的存储
+ **控制器及固件**：协同合作完成数据的读写，维护 SSD 的性能和延长使用寿命
+ **缓存芯片**：SSD 一般会带有

SSD 的存储核心是 NAND Flash，其基本单元是**浮栅晶体管**。浮栅晶体管由控制栅极、氧化层、浮栅极、源极和漏极构成，其中浮栅极可以存储电子，上下被氧化层包围，避免电子流出。浮栅极中如果没有电子，表示状态 1；存储电子后表示状态 0。

浮栅晶体管的**写操作只能将 1 改写成 0**，不能将 0 改为 1。将 0 改为 1 需要采取另外一个称为**擦写**的操作，即将浮栅极中的电子拉出。擦写操作会损伤隧道氧化层，所以 SSD 有擦写次数的限制。

NAND Flash 组成成多个 **Block**，每个 Block 包含多个 **Page**，通常每个 Page 的大小为 4k 或 8k。NAND 的存取有两个关键约束：
+ 存取必须以 **Page** 为单位，即每次读写至少是一个 Page
+ 不能覆盖写某个 Page：覆盖写入时必须先擦写再写入，而擦写由于电压较高必须以 **Block** 为单位

因此，没有空闲的 Page 时，必须要找到没有有效内容的 Block，先擦写，然后再选择空闲的 Page 写入。

在 SSD 中，一般会维护一个**映射表** (LBA → PBA)，维护逻辑地址到物理地址的映射。每次读写时，可以通过逻辑地址直接查表计算出物理地址，与传统的机械磁盘相比，省去了寻道时间和旋转时间。

> [!note] SSD 与 HDD 的比较
> SSD 的优势在于定位数据快（通过映射表直接计算，无需寻道和旋转）和读写速度快（SSD 可达 500MB/s 以上，而 HDD 通常低于 200MB/s）。SSD 的劣势在于成本高（价格约是 HDD 的 2–3 倍）、有寿命限制（闪存有擦写次数限制）和数据恢复困难（HDD 损坏后可通过物理手段恢复数据，成功率较高；SSD 损坏后难以修复）。


#### 廉价磁盘冗余阵列 (RAID)

**廉价磁盘冗余阵列 (redundant array of independent disks, RAID)** 是利用一台磁盘阵列控制器，统一管理和控制一组磁盘驱动器，组成一个速度快、可靠性高、性能价格比好的大容量磁盘系统。RAID 填补了 CPU 速度快与磁盘设备速度慢之间的间隙。

RAID 的基本策略是：用一组较小容量的、独立的、可并行工作的磁盘代替一个大容量的磁盘，应用冗余技术，数据能多方式组织和分布存储，提高了 I/O 性能和系统可靠性。RAID 的核心技术包括：
+ **条带化 (striping)**：将连续的数据分成条带 (stripe) 分布式存储到不同的物理磁盘驱动器上，每个条带由 $k$ 个扇区组成。数据的读写操作在多个硬盘上并行执行，因此成倍地提高读写速度
+ **冗余磁盘**：保存奇偶校验信息，当磁盘出现失误时确保数据的恢复

RAID 共有 7 级（RAID 0–RAID 6），其中 RAID 0 无校验，RAID 1–RAID 6 的区别仅在于实现冗余校验的方法不同。
+ **RAID 0** 至少需要两块或以上大小相同的硬盘，将连续的条带以轮转方式写到全部驱动器上。读写速度快，但不具有容错性。
+ **RAID 1** 由两块硬盘互为镜像。一个驱动器出现故障时，数据可以从镜像驱动器获得，具有容错性。实际容量等于一块硬盘的容量，写操作速度与单块硬盘相同，读操作时可以使用其中任意副本，因此速度可以是单块硬盘的 2 倍。
+ **RAID 2** 工作在字或字节的基础上。例如，每个字节分割成 4 位的半字节，对每个半字节加入一个汉明码形成 7 位的字，存放在 7 个驱动器上。驱动器的磁盘臂同步工作，每个磁盘的磁头都在相同位置。
+ **RAID 3** 是 RAID 2 的简化版本，为每个数据字计算一个奇偶检验位并将其写入一个奇偶驱动器中。如果一个驱动器崩溃，控制器只需假设该驱动器的位为 0，如果有奇偶错误，则原来的位为 1。
+ **RAID 4** 与 RAID 0 相似，但是对各个条带计算出一个奇偶条带，并将其写入一个额外的驱动器中。如果一个驱动器崩溃，损失的字节可以从奇偶驱动器计算出来。缺点是奇偶驱动器成为瓶颈。
+ **RAID 5** 以循环方式在所有驱动器上均匀地分布奇偶检验位。组建 RAID 5 要求至少 3 块硬盘，可以允许坏掉 1 块硬盘。缺点是一个驱动器崩溃时，重构故障驱动器的内容非常复杂。
+ **RAID 6** 比 RAID 5 增加了一个额外的校验块。组建 RAID 6 要求至少 4 块硬盘，可以允许坏掉 2 块硬盘。

此外还有组合级别：
+ **RAID 0+1**（RAID 01）：采用 2 组 RAID 0 的磁盘阵列互为镜像，它们之间又成为一个 RAID 1 的阵列
+ **RAID 1+0**（RAID 10）：RAID 1 与 RAID 0 的结合，容错性显著高于 RAID 0+1



# _5_  文件管理

## 文件系统概述

信息是计算机系统中的重要资源，所有的计算机应用系统都要存储信息和检索信息，操作系统的一个重要功能就是负责信息的组织、存储和访问。存储信息的三个基本要求是能够存储大量的信息、长期保存信息、以及可以共享信息。解决方法就是把信息以一种单元，即**文件**的形式存储在磁盘或其他存储介质上。

文件是通过操作系统来管理的。文件的结构、命名、存取、使用、保护和实现方法都是操作系统设计的主要内容。操作系统中处理文件的部分称为**文件系统 (file system)**，它管理文件的存储、检索、更新，提供安全可靠的共享和保护手段，并且方便用户使用。

文件系统的功能主要包括：
+ 以符号名称作为文件标识，便于用户使用
+ 在多道程序系统中支持对文件的并发访问和控制
+ 在不同设备上提供同样的接口，方便用户操作和编程（设备无关性）
+ 在多用户系统中为不同用户提供不同的文件访问权限
+ 优化性能，包括存储效率、检索性能和读写性能
+ 差错恢复：能够验证文件的正确性，并具有一定的差错恢复能力

对大多数用户而言，文件系统是操作系统中最为可见的部分。操作系统和所有用户的程序与数据的存储都是由文件系统负责的。文件系统在操作系统接口中占的比例最大，用户使用操作系统的感觉在很大程度上取决于对文件系统的使用效果。

文件系统可以从两个角度来理解。从**用户观点**看，文件系统关注文件如何呈现在用户面前：一个文件由什么组成、如何命名、如何保护文件、可以进行何种操作等。核心目标是实现**按名存取**。从**操作系统观点**看，文件系统关注文件和目录怎样实现、怎样管理存储空间、文件存储位置、磁盘实际运作方式（与设备管理的接口）、文件系统性能等。

## 文件系统的硬件支持

### 磁盘分区

物理磁盘按照**扇区 (sector)** 来寻址，扇区的大小由硬件决定，通常是 512 字节。**分区 (partition)** 是磁盘上一组连续扇区的集合。

磁盘的 0 号扇区称为**主引导记录 (MBR，Master Boot Record)**，用来引导计算机。MBR 大小固定，其中包含了可执行的指令（称为引导代码）和分区表，分区表中给出了每个分区的起始和结束地址，以及指示该分区是否可引导的标记。每个硬盘最多可以包括 4 个主分区，每个主分区又可以包含扩展分区。

PC 机的系统引导过程为：PC 机引导时首先执行 BIOS，由 BIOS 选择一个引导设备，将该设备的 MBR 读入内存，然后将控制权交给 MBR 的引导代码。引导代码扫描主分区表，直到找到引导分区 (boot partition)，再将引导分区中的引导扇区 (boot sector) 读入内存，并将控制权交给引导扇区代码。

MBR 分区支持的存储容量上限为 2TB，为支持更大容量的磁盘引入了 **GPT 分区**（GUID Partition Table）。GPT 分区的存储容量理论上无限制，且分区数目没有上限。为了兼容，GPT 分区也有 MBR 引导，称为保护引导。

### 卷与文件系统格式

文件系统在**卷 (volume)** 上创建。**文件系统格式 (file system format)** 定义了文件数据在卷上的存储方式，它影响了一个文件系统的各种特性。

> [!note] FAT 文件系统格式
> FAT 文件系统的卷结构为：引导区 → 文件分配表 1 → 文件分配表 2（镜像备份） → 根目录 → 其他目录和文件。FAT 表中每个簇对应一个表项，文件分配采用链式分配方法。

> [!note] Unix 类文件系统格式
> Unix 类操作系统的文件系统格式包含：**超级块**（包含文件系统的所有关键参数）、空闲块信息、以及 **i 节点**（一个数据结构数组，每个元素对应一个文件）。

**簇 (cluster)** 是文件系统格式使用的可寻址数据块，簇的大小总是扇区大小的整数倍。文件系统使用簇来管理磁盘空间。簇越大，其潜在缺点是可能会产生内部碎片，浪费更多的磁盘空间。

**元数据 (metadata)** 是为支持文件系统格式管理而存储在卷上的数据。例如，元数据包含了卷上的文件和目录的位置信息等。对于应用程序，元数据通常是不可访问的。

## 文件

### 文件的属性

文件是一种抽象机制，它提供了一种在磁盘上保存信息并方便以后读取的方法。这种方法可以使用户不用了解存储信息的方法、位置和实际磁盘工作方式等有关细节。

> [!definition] 文件
> **文件**是一组具有标识符号的在逻辑上具有完整意义的信息项的集合，这个标识符号称为**文件名**。

文件包括两部分：**文件体**（文件本身的信息）和**文件属性**（文件的存储和管理信息）。文件属性包括文件名、文件内部标识、文件存储地址、访问权限、访问时间等。

文件的命名规则在不同的操作系统中是不同的，涉及长度限制、数字和特殊字符的使用、大小写是否区分等方面。文件扩展名通常用于提示文件内容、类型等相关信息。在 Linux/UNIX 中，文件扩展名只是一种约定，操作系统并不强迫采用它；Windows 则对扩展名赋予含义，可以在操作系统中注册扩展名并规定哪个程序打开该文件。

文件可以从多个角度进行分类：
+ **按文件性质和用途**，分为
    + 系统文件：与操作系统有关的文件
    + 用户文件
    + 库文件：标准子程序及常用应用程序组成文件，允许用户使用但不能修改
+ **按信息保存期限**，分为临时文件、永久文件、档案文件；
+ **按文件的保护方式**，分为只读文件、读写文件、可执行文件；
+ **按文件的逻辑结构**，分为流式文件和记录式文件；
+ **按文件的物理结构**，分为顺序文件、链接文件、索引文件。

Linux/UNIX 系统将文件分为三类：
+ **普通文件**，包含用户信息，一般为 ASCII 或二进制文件
+ **目录文件**，管理文件系统结构的系统文件
+ **特殊文件**，即设备文件，把外部设备也看作文件，包括字符特殊文件和块特殊文件

### 文件的逻辑结构

文件的逻辑结构从用户角度看文件的组织形式，它可以独立于在外存上的物理存储。文件逻辑结构的设计要求包括：访问性能（便于检索、便于修改）和存储性能（向物理存储转换方便，节省空间）。文件的不同组织层次为：数据项 → 记录 → 文件。

#### 无结构文件

**无结构文件**（流式文件）的文件体为字节流，不划分记录，构成文件的基本单位是字节。提供最大的灵活性。

#### 累积文件

**累积文件 (pile)** 的文件体为无结构记录序列，通过特定分隔符来划分记录，各记录大小和组成可变，新记录总是添加到文件末尾。例如日志 log、电子邮件的邮箱文件 (mailbox)。

#### 顺序文件

**顺序文件 (sequential file)** 的文件体为大小相同的排序记录序列，按某个关键字域 (key field) 排序，存放在主文件 (master file) 中。新记录暂时保存在日志或事务文件 (log file or transaction file) 中，定期归并入主文件。

#### 索引顺序文件

**索引顺序文件 (indexed-sequential file)** 在顺序文件的基础上，另外建立索引 (index) 和溢出文件 (overflow file)，以加快检索速度。在索引文件中，可将关键字域中的取值划分为若干区间，每个区间对应一个索引项，后者指向该区间的开头记录。新记录暂时保存在溢出文件中，定期归并入主文件。

#### 索引文件

**索引文件 (indexed file)** 的记录大小不必相同、不必排序，存放在主文件 (primary file) 中。与索引顺序文件的区别在于主文件不排序，而是另外建立索引，每个索引项指向一个记录，索引项按照记录中的某个关键字域排序。对同一主文件，可以针对不同的关键字域相应建立多个索引。索引文件的记录项通常较小，查找速度快，便于随机访问 (random access)。

#### 哈希文件

**哈希文件 / 直接文件 (hashed file / direct file)** 的记录大小相同，由主文件和溢出文件组成。记录位置由哈希函数确定。检索时给出记录编号，通过哈希函数计算出该记录在文件中的相对位置。访问速度快，但在主文件中有空闲空间。

### 文件操作

以 Linux/UNIX 的系统调用为例，基本的文件操作有：
+ **creat**：创建一个空文件或将已有文件截空。`fd = creat(name, pmode);`，其中 `fd` 为返回的文件描述符，`name` 为文件名，`pmode` 为保护模式
+ **open**：打开一个已存在的文件。`fd = open(name, rwmode[, pmode]);`，读写模式包括 `O_RDONLY`（只读）、`O_WRONLY`（只写）和 `O_RDWR`（读写）
+ **read / write**：对文件进行读写。`n = read(fd, buffer, size);`，`n` 返回实际读写的字节数
+ **close**：关闭一个已打开的文件。`close(fd);`
+ **lseek**：改变文件指针所指的位置。`newpos = lseek(fd, offset, origin);`，基准点 `origin` 可取 `SEEK_SET`（文件首）、`SEEK_CUR`（当前位置）或 `SEEK_END`（文件尾）

文件存取方式分为：
+ **顺序存取**：只能从头开始按顺序存取文件的字节或记录，不能跳过某些内容
+ **随机存取**：可以不按顺序存取文件中的字节或记录，或者按照关键字而不是位置来存取记录

## 目录

### 文件控制块

**文件控制块 (FCB，File Control Block)** 是操作系统为管理文件而设置的数据结构，它存放了为管理文件所需的所有有关信息（文件属性）。文件控制块是文件存在的标志。FCB 的内容包括：文件名、文件号、用户名、文件地址、文件长度、文件类型、文件属性、共享计数、文件的建立日期、保存期限、最后修改日期、最后访问日期、口令、文件逻辑结构、文件物理结构等。

把所有的 FCB 组织在一起就构成了**文件目录**，即文件控制块的有序集合。构成文件目录的项目称为**目录项**（目录项可以就是 FCB）。为了实现对文件目录的管理，通常将文件目录以文件的形式保存在外存，该文件称为**目录文件**。

### 目录结构

+ **一级目录系统**：为所有文件建立一个目录文件，组成一线性表。结构简单、易实现，但文件多时目录检索时间长，且限制了用户对文件的命名（不允许重名）
+ **两级目录系统**：目录分为两级。一级称为主文件目录，给出用户名和用户子目录所在的物理位置；二级称为用户文件目录（用户子目录），给出该用户所有文件的 FCB。解决了文件的重名问题，查找时间降低，但增加了系统开销
+ **层次目录系统**（树状目录）：在文件数目较多时，便于系统和用户将文件分散管理，适用于较大的文件系统管理

### 路径名

用目录树组织文件系统时，需要有某种方法指明文件名。
+ **绝对路径名**：由从根目录到文件的路径组成
+ **相对路径名**：用户可以指定一个目录为当前工作目录，这时所有不从根目录开始的路径名都是相对于当前工作目录的
+ `.` 指当前目录，`..` 指当前目录的父目录

### 目录操作

以 Linux/UNIX 的系统调用为例：

+ **mkdir**：创建一个目录。`int mkdir(const char *path, mode_t mode);`
+ **rmdir**：删除一个目录。`int rmdir(const char *path);`
+ **opendir**：打开一个目录。`DIR *opendir(const char *dirname);`，返回指向 DIR 类型的指针
+ **readdir**：读取一个目录项。`struct dirent *readdir(DIR *dirp);`
+ **closedir**：关闭一个目录。`int closedir(DIR *dirp);`
+ **link**：创建到一个文件的连接。`int link(const char *existing, const char *new);`
+ **unlink**：删除目录项。`int unlink(const char *path);`

## 文件系统的实现

### 文件的物理结构

从系统的角度看文件，从文件在物理介质上的存放方式来研究文件，就是文件的物理结构。

#### 连续分配

文件存放在若干连续的簇（磁盘块）中。目录项中只需存放首簇的地址和簇数。

连续分配的优点在于简单易实现，支持顺序存取和随机存取，顺序存取速度快，所需的磁盘寻道次数和寻道时间最少。缺点在于文件不能动态增长，且会产生外部碎片问题。连续分配在 CD-ROM 文件系统中广泛使用，因为文件大小事先已知且在使用中不再改变。

#### 链接分配

一个文件的信息存放在若干不连续的簇中，各簇之间通过指针连接，前一个簇指向下一个簇。每个簇中包含一个指针字段，其余部分存放数据。在目录项中只需存放首簇的地址，文件的其他簇可以通过簇中的指针得到。

链接分配的优点在于充分利用磁盘空间，没有外碎片，且顺序访问速度快。缺点在于随机访问性能差，寻道次数和寻道时间较多，且由于指针占去一些字节，数据部分不再是 2 的整数幂次。

#### FAT 分配

FAT 分配是链接分配的一种实现方式，目的是克服数据部分不是 2 的整数幂次的缺点。具体方法是取出每个簇中的指针字段，存放在一个单独的表中，称为**文件分配表 (FAT，File Allocation Table)**。在目录中只需记录文件的起始簇号，其余的簇可以从 FAT 表中获得。

FAT 分配的优点在于充分利用磁盘空间、没有外碎片、顺序访问速度快，且消除了指针占用数据空间的问题。缺点在于随机访问性能差，寻道次数和寻道时间较多，且 FAT 本身占用一定的磁盘空间。

#### 索引分配

一个文件的信息存放在若干不连续的磁盘块（簇）中，系统为每个文件建立一个专用数据结构，称为 **i 节点**（索引节点），并将这些块的块号存放在一个 i 节点中。一个 i 节点就是磁盘块地址数组，其中第 $n$ 个条目指向文件的第 $n$ 块。当一个文件包含的磁盘块数超出 i 节点所能容纳的数目时，最后一个项不是指向数据块，而是指向包含附加磁盘地址的磁盘块（间接索引）。

索引分配的优点在于既能顺序存取又能随机存取，满足了文件动态增长、插入删除的要求，能充分利用外存空间。缺点在于较多的寻道次数和寻道时间，且索引表本身带来了系统开销。

### 目录的实现

目录是目录项的有序集合。目录项中包含文件名和查找文件磁盘块所需要的信息。是否包含文件属性则随不同的文件系统而不同。

在**简单目录**中，文件属性直接存放在目录项中。在采用 **i 节点**的系统中，文件属性存放在 i 节点中而不是目录项中，目录项只引用 i 节点。

变长文件名的存储有两种方式：

+ **在行中**：每个目录项包含固定部分（目录项长度、文件属性）和可变部分（实际文件名，填充成整数个字）。缺点是删除文件后留下可变长度的空隙，且一个目录项可能分布在页面的边界上
+ **在堆中**：每个目录项都有固定的长度，文件名放置在目录后面的堆中。优点是当一个目录项被删除后，另一个目录项总是可以适合这个空隙；缺点是需要对堆进行管理

## 磁盘空间管理

### 簇的大小

簇的大小是一个重要的设计选择。两个极端是：大到能容纳整个文件，或者小到只有一个外存扇区。

簇较大时，可以提高 I/O 访问性能、减小管理开销，但簇内碎片浪费问题较严重。簇较小时，簇内的碎片浪费较小（特别是有大量小文件时有利），但可能存在簇编号空间不够的问题（如 FAT12、FAT16、FAT32 的编号位数限制）。

文件卷容量与簇大小的关系也存在权衡。若簇的总数保持不变即簇编号所需位数保持不变，文件卷容量越大则簇越大，簇内碎片浪费越多。若簇大小不变，文件卷容量越大则簇总数越多，相应簇编号所需位数越多，可能导致簇编号空间不够。

### 空闲空间管理

**空闲块位图**：每个块用 1 位表示（0 为空闲，1 为占用）。实现简单，但位图本身需要占用空间。

**空闲块链表**：将所有空闲块链接起来，分配和回收操作简单。

## 文件共享

多个用户协同工作时常常需要共享文件。共享文件同时出现在属于不同用户的目录下。实现共享文件有两种方式。

### 硬链接

硬链接的目录项指向目标文件的 i 节点，本质是为同一个文件起了一个新别名。i 节点维护**链接计数**。创建硬链接：`ln /源路径/源文件 /目标路径/硬链接名`。

硬链接的优点在于不占用额外空间（只是在目录中增加了一条记录指向现有 inode，不分配新的 inode），且访问速度快、没有性能损耗。缺点在于不能跨文件系统（因为 inode 在不同的文件系统之间是独立的），不允许指向目录（否则可能在文件系统中形成死循环），且硬链接和源文件共享同一个 inode，它们的权限、所有者、修改时间等完全相同，无法独立设置。

在多用户系统中硬链接还有一个问题：假设文件由 C 创建，B 在自己的目录下创建了该文件的硬链接。当 C 删除文件时，因为链接计数不为零，所以只删除目录项，不删除 i 节点和文件本身。但文件的所有者仍为 C，C 继续为文件付帐。

### 符号链接（软链接）

符号链接创建一个类型为 LINK 的独立文件，该文件只包含它所连接的文件的路径名。创建符号链接：`ln -s /源路径/源文件 /目标路径/符号链接名`。

符号链接的优点在于使用灵活（可以跨文件系统甚至跨网络共享文件），拥有独立的 inode 可以有独立的权限和属性设置，且允许指向目录（潜在的死循环可以检测）。缺点在于容易断链（如果源文件被删除或者重命名，软链接就会失效），访问时需二次解析（先读取链接文件获取路径，再根据路径去寻找目标文件，存在轻微的性能损耗），且因为是一个独立的文件，会占用一个 inode 以及存储路径字符串的少量磁盘空间。

## 文件系统可靠性

比起计算机的损坏，文件系统破坏的损失要大得多。文件系统的**可靠性**指系统抵抗和预防各种物理性破坏和人为性破坏的能力。

### 文件备份

文件备份主要解决两个问题：从意外的灾难中恢复，以及从错误的操作中恢复。文件备份通常是将磁盘文件转储到磁带上保存，有两种方案。

**物理转储**从磁盘的第 0 块开始，将全部的磁盘块按顺序输出到磁带上。优点是简单；缺点是不能跳过选定的目录、不能实现增量转储、不能恢复单个的文件。**增量转储**指周期性地做全面备份，而每次只对最近更改的数据做备份。

**逻辑转储**从一个或几个指定的目录开始，递归地转储自给定基准日期后有所更改的全部文件和目录。优点是可以很容易满足恢复特定文件或目录的请求；缺点是实现复杂。在逻辑转储中，需要转储通向修改过的文件或目录上的所有目录（即使其本身未修改过），以方便将这些转储的文件和目录恢复到另一台计算机上。

### 文件系统一致性检查

文件的修改操作一般是读取磁盘块、修改、再写回磁盘。如果修改过的磁盘块全部写回之前系统崩溃，则文件系统可能处于不一致状态。

**块的一致性检查**：构造两张表，每张表中为每个块设立一个计数器。第一个表中的计数器跟踪该块在文件中出现的次数，第二个表中的计数器跟踪该块在空闲表中出现的次数。两张表每个块对应的计数初值为 0。检验程序首先读取全部 i 节点，得到相应文件使用的块的块号，每读到一个块时第一个表中的计数值增 1；然后检查空闲表，找到全部未使用的块，使第二个表中对应的计数值增 1。正常状态下每个块在两表中的计数值之和应为 1。

不一致的情况及处理方法：
+ **块丢失**（两表计数均为 0）：不会造成大的损害但是浪费磁盘空间，解决方法是将该块加到空闲表中
+ **空闲表中有重复块**（空闲表计数大于 1）：解决方法为重建空闲表
+ **重复数据块**（文件表计数大于 1）：分配一个空闲块，将原块内容复制到其中，并将该块插入到一个文件中。这样虽然可以保证文件系统的一致性，但是错误并未消除，因此应该报告这一错误

**文件的一致性检查**：构造一张计数器表，为每个文件设立一个计数器，初值为 0。检验程序从根目录开始检验，对每个目录中的每个文件，将文件计数器增 1。检查完成后，得到一张由 i 节点索引的表，说明每个文件被包含在多少个目录中。然后将计数值与 i 节点中的连接数进行比较：
+ 两值相等：文件系统一致
+ i 节点的连接数大于目录项数：可能存在不属于任何目录的文件从而浪费磁盘空间，应修改 i 节点中的连接计数
+ i 节点的连接数小于目录项数：存在潜在的灾难（目录项指向不存在的文件），应修改 i 节点中的连接计数

## 文件系统实例

### FAT 文件系统

FAT（File Allocation Table，文件分配表）文件系统属遗留文件系统。为了向后兼容，也为了方便用户升级，Windows 仍然提供对 FAT 的支持。

每一种 FAT 文件系统都用一个数字来标识磁盘上簇号的位数。例如，FAT12 的簇标识为 12 位，这限制了它的单个分区最多只能存储 $2^{12} = 4096$ 个簇，而 FAT12 在 Windows 中的簇大小在 512B 与 8KB 之间，这意味着 FAT12 卷的大小至多只有 32MB。FAT 各版本的区别如下：

+ **FAT12**：每个 FAT 表项 1.5 字节，适用于簇数小于 4087 的文件卷
+ **FAT16**：每个 FAT 表项 2 字节，适用于簇数界于 4087 和 65526 之间的文件卷
+ **FAT32**：每个 FAT 表项 4 字节，适用于簇数界于 65526 和 268,435,456 之间的文件卷

FAT 文件系统的卷结构为：引导区 → FAT 表 1 → FAT 表 2（镜像备份） → 根目录 → 其他目录和文件。每个 FAT 表项的值以 FAT12 为例：
+ 0：表示该簇空闲
+ FF7h：物理坏扇区
+ FF8h–FFFh：表示该簇是文件的最后一个簇
+ 其他值：表示该簇被文件占用，且表项中的值是文件下一个簇的编号

FAT 的目录是目录项的顺序文件，不对目录项排序，若目录中包含的文件数目较多则搜索效率低。每个目录项大小为 32 字节，内容包括文件名（8+3 个字符）、属性（文件/子目录/文件卷标识）、最后一次修改时间和日期、文件长度、第一个簇的编号。在目录项中，若第一个字节为 E5h 则表示空目录项。文件名不区分大小写。

FAT32 的根目录区不再是固定区域、固定大小，而是可以看作数据区的一部分，采用与子目录文件相同的管理方式，根目录下的文件数目不再受最多 256 的限制。FAT32 支持长文件名格式和 Unicode，但无法支持高级容错特性，不具有内部安全特性。

### NTFS 文件系统

NTFS 是 Windows 的现代文件系统，为改进的多级目录结构，支持文件别名。NTFS 文件由多个文件属性构成，每个属性由属性名和属性流（简单字节队列）组成，用户可自定义属性。NTFS 支持按用户和用户组分配权限，包含读、写、运行、删除和修改五种权限。NTFS 文件支持数据压缩功能，卷结构支持容错功能。

卷建立在磁盘分区上，一个磁盘可以有多个卷，一个卷也可以由多个磁盘组成。卷上的数据分为元数据和用户数据。NTFS 使用两种簇号进行定位：
+ **LCN (Logical Cluster Number，逻辑簇号)**：对整个卷中所有的簇从头到尾所进行的简单编号
+ **VCN (Virtual Cluster Number，虚拟簇号)**：对属于特定文件的簇从头到尾进行编号，以便于引用文件中的数据

**MFT (Master File Table，主控文件表)** 是 NTFS 卷结构的核心，包含了卷中所有文件的信息。MFT 是以文件记录数组来实现的，每个文件记录的大小都固定为 1KB。卷上的每个文件（包括 MFT 本身）都有一个 MFT 记录。MFT 中开始的 16 个记录是保留的，留给系统的 16 个元数据文件使用，每个元数据文件都有一个以 `$` 开头的文件名称。

每个文件都有一个 64 位的、称为**文件引用号 (File Reference Number)** 的唯一标识，由两部分组成：文件号（48 位，对应于该文件在 MFT 中的位置）和文件顺序号（随每次文件记录的重用而增加，用于 NTFS 进行内部一致性检查）。

NTFS 的簇作为磁盘空间分配和回收的基本单位，簇的大小是物理扇区的整数倍，由用户在格式化时确定。当属性值能直接存放在 MFT 中时，该属性称为**常驻属性 (resident attribute)**。小文件或小目录的所有属性均可以在 MFT 中常驻，NTFS 对它的访问时间将大大缩短。如果一个属性太大而不能存放在只有 1KB 的 MFT 文件记录中，那么 NTFS 将从 MFT 之外分配区域，这些区域称为一个**延展 (run)** 或延伸 (extent)，用于存储属性值。值存储在延展中的属性称为**非常驻属性 (nonresident attribute)**。

NTFS 的索引根属性将相应目录中的文件名和子目录名进行排序。对于一个大目录，文件名实际存储在组织文件名的固定 4KB 大小的索引缓冲区中，索引缓冲区通过 B+ 树数据结构实现。索引分配属性包含了索引缓冲区的 VCN 到 LCN 映射，位图属性跟踪在索引缓冲区中哪些 VCN 在使用而哪些是空闲的。

NTFS 支持数据压缩。对于稀疏文件（相对于文件大小而言只有少量非零数据的文件），NTFS 只给那些包含非零数据的延展分配磁盘空间。当程序从压缩文件中读取数据时，NTFS 通过检测该位置是否有 VCN 到 LCN 的映射来决定该数据是不是零数据。对于非稀疏文件的压缩，NTFS 以 16 个簇为压缩单元来进行，确保每个延展都以一个虚拟 16 簇边界开始。

### EXT 文件系统

Linux 最初使用 MINIX 文件系统，只能支持最高 64MB 硬盘和 14 字符文件名。EXT（EXTended File System，扩展文件系统）的发展经历了多个版本：

+ **EXT**：1992 年推出，专为 Linux 设计的文件系统。文件容量上限为 2GB，允许最长 255 字符的文件名。采用 inode 记录文件的权限、UID、GID 以及指向数据块的指针。采用链表追踪空闲空间，容易导致碎片
+ **EXT2**：1993 年推出。引入了块组的概念，将数据和 inode 尽可能聚集存放，减少了磁头寻道时间。抛弃了链表，改用 inode 位图和块位图跟踪文件和空闲空间，缓解了碎片化问题。支持最大 4TB 的文件系统，单个文件最大可达 2GB
+ **EXT3**：2001 年引入。与 EXT2 在硬盘布局上相同，差别在于 EXT3 多出了一个特殊的 inode 用来记录文件系统的日志。支持最大 16TB 的文件系统，单个文件最大支持 2TB
+ **EXT4**：2008 年推出。修改了 EXT3 中部分重要的数据结构，引入了区段 (Extents) 概念，可以提供更好的性能。支持最大 1EB 的文件系统，单个文件最大支持 16TB

EXT 文件系统中的文件保存在数据块中。对同一个 EXT 文件系统来说，其上所有的数据块大小都是一样的（在建立文件系统时指定）。单个文件所占用的空间以数据块为单位分配，平均来说每个文件会浪费半个数据块的空间。

EXT 文件系统利用 **inode** 来描述文件系统的拓扑结构。在单个文件系统中，每个文件对应一个 inode，而每个 inode 有一个唯一的整数标识符。文件系统中所有文件的 inode 保存在 inode 表中。EXT 文件系统中的目录是一种特殊文件，也有对应的 inode，其 inode 指向的数据块中包含该目录中所有的目录项（文件、目录、符号链接等），每个目录项对应自己的 inode。

EXT 文件系统将逻辑分区划分为**块组 (Block Group)**。每个块组重复保存着对文件系统的完整性非常关键的信息，同时也用来保存实际的文件和目录数据。文件系统关键信息的重复存储有助于在发生故障时还原。每个块组的结构为：超级块 → 组描述符 → 块位图 → 索引节点位图 → 索引节点表 → 数据块。

**超级块 (Superblock)** 包含了文件系统的基本大小和形式，其中包含的数据由文件系统管理程序用来进行文件系统的维护。每个块组中包含有相同的超级块信息，但通常只需读取块组 0 的超级块。**块组描述符**保存在每个块组超级块的后面，每个块组中包含所有块组的描述符。

inode 的结构中，前 12 个数据块指针直接指向包含文件数据的数据块（直接指针），而其后的三个数据块则是间接指针：单级间接指针、二级间接指针和三级间接指针。在 EXT 目录中，每个目录的前两个目录项始终是标准的 `.` 和 `..`，分别代表"本目录"和"父目录"。每个目录项包含索引节点号、目录项名称长度和目录项名称。
