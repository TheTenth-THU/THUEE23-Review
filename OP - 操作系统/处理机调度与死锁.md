## 处理机调度

### 基本概念

处理机是计算机系统中最重要的资源。在多道程序系统中，常有多个进程竞争 CPU，操作系统必须选择运行哪个进程——完成这一选择工作的代码称为**调度程序**（scheduler）。

处理机调度的时机：
1. 创建新进程后
2. 进程运行完毕或因错误终止（**必须调度**）
3. 进程转入等待状态（**必须调度**）
4. I/O 中断发生时
5. 时钟中断发生时

### 调度方式

**非抢先式调度**（non-preemptive）：调度程序一旦分配 CPU 后便让进程一直运行，直到进程完成或因等待事件而阻塞。实现简单、开销小，适用于批处理系统。

**抢先式调度**（preemptive）：调度程序可根据规定原则剥夺正在运行的进程的 CPU，将其移入就绪队列。适用于分时系统和实时系统。抢先原则有两种：**优先级原则**（高优先级抢占低优先级）和**时间片原则**（时间片用完被抢占）。

### 调度层次

| 层次 | 别名 | 功能 | 时间尺度 |
|------|------|------|---------|
| 长程调度 | 作业调度/高级调度/宏观调度 | 选择作业进入内存 | 分钟~小时 |
| 中程调度 | 中级调度/内外存交换 | 进程换入换出内存 | — |
| 短程调度 | 进程调度/低级调度/微观调度 | 选择就绪进程运行 | 毫秒级 |

### 调度算法的目标

- **所有系统**：公平（给每个进程公平的 CPU 份额）、效率（保持系统所有部分忙碌）
- **批处理系统**：吞吐量最大、周转时间最短、CPU 利用率最高
- **交互式系统**：响应时间最短
- **实时系统**：保证截止时间

## 批处理系统中的调度算法

### 先来先服务（FCFS）

按进程提交或变为就绪的先后顺序进行调度。非抢先式。

- **优点**：实现简单
- **缺点**：有利于长作业和 CPU 密集型作业，不利于短作业和 I/O 密集型作业

### 最短作业优先（SJF）

预计执行时间短的作业优先分配 CPU，非抢先式。目标是最小化平均周转时间。

- **优点**：与 FCFS 相比改善了平均周转时间
- **缺点**：对长作业不利（可能导致饥饿）；难以预计作业执行时间

### 最高响应比优先（HRRN）

响应比定义为：

$$
\text{响应比} = \frac{\text{等待时间} + \text{要求服务时间}}{\text{要求服务时间}} = 1 + \frac{\text{等待时间}}{\text{要求服务时间}}
$$

介于 FCFS 和 SJF 之间的折中算法，既考虑等待时间又考虑运行时间，既照顾短作业又不使长作业等待时间过长。

### 最短剩余时间优先（SRT）

SJF 的抢先式版本——调度程序总是选择剩余时间最短的进程运行。若新进程的剩余时间更短，当前进程被抢占。

## 交互式系统中的调度算法

交互式系统普遍采用**抢先式**调度，以保障响应时间和公平性。

### 时间片轮转（RR）

每个进程被分配一个**时间片**（quantum），用完则剥夺 CPU 并排到就绪队列末尾。若进程在时间片内阻塞或完成，CPU 立即切换。

- 时间片过短 → 过多的进程切换，降低 CPU 效率
- 时间片过长 → 响应时间变长
- 通常 20ms～50ms 是合理折中

### 优先级调度

每个进程被赋予一个优先级，最高优先级的可运行进程先运行。优先级可以是**静态**的（创建时指定，运行中不变）或**动态**的（等待时间长可升高优先级，使 I/O 密集型进程获得较好服务）。同优先级内通常采用时间片轮转。

### 多级反馈队列（MLFQ）

优先级调度 + 时间片轮转的改进：设立多个优先级类，高优先级队列时间片短（适合交互式进程），低优先级队列时间片长（适合 CPU 密集型进程）。典型规则：新进程进入高优先级 → 时间片用完降级 → 频繁阻塞升级。MLFQ 能自动识别进程是交互式还是 CPU 密集型。

### 最短进程优先

SJF 的交互式版本。通过**老化**（aging）技术预测下一命令的执行时间：$T_{\text{新}} = aT_{0} + (1-a)T_{1}$，其中 $a$ 通常取 0.5。

## 实时系统中的调度

实时系统是**时间起主导作用**的系统。分为**硬实时**（HRT，必须满足绝对截止时间）和**软实时**（SRT，偶尔超时可容忍）。

可调度性条件：设 $m$ 个周期性事件，事件 $i$ 的周期为 $P_i$，需要 $C_i$ 秒 CPU 时间，则在单 CPU 上的可调度条件为

$$
\sum_{i=1}^{m} \frac{C_i}{P_i} \leq 1
$$

常用实时调度算法：

- **速率单调调度**（RMS）：静态优先级，优先级与事件频率成正比
- **最早截止时限优先**（EDF）：动态优先级，总是选择截止时限最早的进程
- **最小裕度优先**（LLF）：动态优先级，裕度 = (截止时刻 - 当前时刻) - 剩余运行时间，总是选择裕度最小的进程

> [!note] EDF vs LLF
> EDF 的紧急程度取决于 deadline 的早晚（谁最近先做谁），LLF 的紧急程度取决于还能拖多久（谁最禁不起拖先做谁）。

> [!example]- Linux 处理机调度
> Linux 6.6+ 采用 **EEVDF**（Earliest Eligible Virtual Deadline First）调度算法，基于红黑树实现。核心理念：每个进程根据权重计算应得的 CPU 时间，只有"合格"（eligible）的进程才有资格被选中；在所有合格任务中，选择虚拟截止时间最早的任务运行。调度分级从高到低：Stop-Task → Deadline (SCHED_DEADLINE) → Real-Time (FIFO/RR) → Fair (CFS/EEVDF, SCHED_OTHER) → Idle。针对多核，每个 CPU 有私有运行队列，通过推拉机制实现负载均衡。

> [!example]- Windows 处理机调度
> Windows 11 采用基于优先级的抢占式 MLFQ 变体，32 个优先级队列（0-31）：1-15 为动态优先级（普通线程，I/O 提升、前台提升、饥饿补偿），16-31 为实时优先级（固定优先级）。支持软亲和性（倾向于将线程放回上次运行的核心）和理想处理器策略。针对 Intel Thread Director 的 P-Core/E-Core 混合架构，支持软硬件协同的非对称调度。

## 死锁

### 死锁的基本概念

**死锁**（Deadlock）是指系统中多个进程无限期地等待永远不会发生的条件——若无外力协助，每个进程都不能继续执行下去。

产生死锁的原因：
- **竞争资源**：共享资源不足以同时满足所有进程需要
- **进程推进顺序不当**：请求和释放资源的顺序不当

资源的分类：
- 按性质：**可抢占资源**（CPU、内存等）vs **不可抢占资源**（打印机、刻录机等）
- 按使用方式：共享资源 vs 独占资源
- 按使用期限：**永久资源**（可重复使用，如硬件和可再入代码）vs **临时性资源**（由一个进程产生、被另一进程短暂使用后无用，如消息、信号）

> [!theorem] 死锁发生的四个必要条件
> 1. **互斥**：任一时刻只允许一个进程使用资源
> 2. **非抢占**：进程已占用的资源不会被强制剥夺
> 3. **请求和保持**：进程在请求其余资源时，不主动释放已占用的资源
> 4. **环路等待**：存在进程循环链，链中每个进程已获得的资源同时被链中下一个进程所请求

### 死锁处理策略

> [!note] 鸵鸟算法
> 当人们对某问题没有很好的解决方法时，选择忽略它——这是目前实际系统采用最多的一种策略。因为死锁发生概率很小而处理代价很大。UNIX、Linux 和 Windows 均采用此方法。

#### 死锁预防

**死锁预防**（deadlock prevention）通过限制并发进程对资源的请求，使四个必要条件中至少有一个不满足：

- **破坏互斥条件**：假脱机（spooling）操作（如打印机守护程序），但并非所有设备都可假脱机
- **破坏非抢占条件**：通常无效——强行剥夺正在打印的打印机会造成混乱
- **破坏请求和保持条件**：**预先静态分配法**——进程开始前一次申请全部所需资源；但降低资源利用率和并发度，且可能无法预知所需资源
- **破坏环路等待条件**：**有序资源使用法**——资源按顺序排列，保证请求不形成环路；但限制请求顺序且需额外开销

#### 死锁避免

**死锁避免**（deadlock avoidance）不事先限制产生死锁的条件，而是在资源动态分配过程中，用某种方法防止系统进入**不安全状态**。

> [!definition] 安全状态与不安全状态
> 若存在某种资源调度顺序，保证所有进程正常运行完成，则该状态为**安全状态**。否则为**不安全状态**。只要系统处于安全状态，便可避免死锁；虽然并非所有不安全状态都是死锁，但当系统进入不安全状态后便有可能进入死锁。

##### 银行家算法（Dijkstra, 1965）

数据结构：
- `Available[m]`：可利用资源向量
- `Max[n][m]`：最大需求矩阵
- `Allocation[n][m]`：分配矩阵
- `Need[n][m]`：需求矩阵，满足 `Need[i][j] = Max[i][j] - Allocation[i][j]`

资源请求算法：
1. 若 `Request[i] ≤ Need[i]`（请求小于需求），转 2；否则错误
2. 若 `Request[i] ≤ Available`（请求小于库存），转 3；否则进程等待
3. 试分配：更新 `Available`、`Allocation`、`Need`
4. 执行**安全性算法**：若新状态安全则正式分配，否则恢复原状态

**安全性算法**：
1. `Work = Available; Finish[i] = false;`
2. 寻找满足 `Finish[i] == false` 且 `Need[i] ≤ Work` 的进程 i
3. `Work += Allocation[i]; Finish[i] = true;` 转 2
4. 若所有 `Finish[i] == true`，系统处于安全状态

> [!tip] 银行家算法的优缺点
> **优点**：允许互斥、请求和保持、非抢占三个条件存在，限制条件比死锁预防更宽松，资源利用率更高。
>
> **缺点**：要求每类资源数量固定不变（难做到——资源需维护，可能损坏）；要求用户数固定不变（多道程序系统中用户数常变）。

#### 死锁检测与解除

**死锁检测**通过化简**资源分配图**（RAG）实现：不断找出既不阻塞又非孤立的进程结点，消去其所有请求边和分配边。若最终所有进程结点都成为孤立结点，则图是可化简的（无死锁）；否则存在环路（死锁）。

> [!theorem] 死锁定理
> 死锁状态的充分必要条件是系统的资源分配图是**不可完全化简**的。所有化简顺序均可得到相同的化简结果图。

**死锁解除**的两种常用方法：
- **剥夺资源法**：从其他进程剥夺足够资源给死锁进程
- **撤销进程法**：撤销全部死锁进程，或按某种顺序逐个撤销直至死锁消除
